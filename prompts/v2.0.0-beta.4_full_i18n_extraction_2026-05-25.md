# Cycle: v2.0.0-beta.4 — Full i18n string extraction across the app

Per [[v2_clean_break]] + [[roadmap_state]], v2.0 ships multilingual UI. The previous cycles set up the plumbing:
- v2.0.0-beta.3: scaffolding + 2 representative pages (UserMenu + /files)
- v2.0.0-beta.3.1: synchronous static-resource init + localStorage shim (so useTranslation never suspends and tests pass)
- v1.13.32: cleanup of dead UserMenu binary-switch tests so the test surface is solid

**THIS cycle (beta.4):** extract EVERY remaining hard-coded user-facing UI string in the React frontend into the i18n JSON files (en + es) and replace with `t("namespace.key")` calls.

## Read first

- `frontend/src/shared/i18n/index.ts` — current config with SUPPORTED_LANGUAGES=["en","es"], static-resource init, useSuspense:false
- `frontend/src/shared/i18n/locales/en/common.json` — common UI primitives (already populated)
- `frontend/src/shared/i18n/locales/en/pages.json` — page-specific strings (currently has filesHome only)
- `frontend/src/shared/i18n/locales/es/*.json` — Spanish translations (must mirror en exactly)
- `docs/development/i18n.md` — naming conventions

## Scope

**IN SCOPE:** Every user-facing English text string in:
- `frontend/src/routes/**/*.tsx` — every page in the app
- `frontend/src/shared/ui/**/*.tsx` — shared UI components (excluding UserMenu which is done)
- `frontend/src/components/**/*.tsx` — shadcn/ui wrappers + custom components
- Toast messages, error banners, empty states, form labels, button labels, page titles, subtitles, table column headers, modal titles + bodies, confirmation dialogs

**OUT OF SCOPE — do NOT extract:**
- Backend Go error messages (operator-facing English-only per existing convention)
- Test files (`__tests__/**` + `*.test.tsx`)
- Story/example fixtures
- Code comments
- Identifiers and code names (variable names, route paths, role kinds like "ui-admin", capability strings like "bucket:delete")
- Product name "basement" stays untranslated everywhere
- Brand/marketing copy in landing pages (README, etc.) — these are markdown docs, not React
- Version banners showing `v1.13.32` etc. — version numbers are not strings
- Skin palette names ("default", "high-contrast", "minimal", "95", "terminal") — these are skin IDs
- Date/time formatting strings — those go through Intl, not i18n

## Approach

### 1. Inventory the strings

Run `grep -rn '"[A-Z][^"]\{3,\}"' frontend/src/routes frontend/src/shared/ui frontend/src/components | grep -v __tests__ | grep -v ".test." | head -200` to find candidate strings. Most matches will be user-facing copy. Filter out:
- Identifiers
- CSS classes / data-testids
- Already-extracted t() calls
- Backend error message constants

Expect 200-400 distinct strings to extract.

### 2. Organize into namespaces

- `common.json` — strings used across multiple pages: button labels, common error/empty messages, role labels, navigation items, confirmation copy, validation messages
- `pages.json` — page-specific strings, organized by page key:
  - `filesHome.*` (already populated)
  - `admin.clusters.*` (list page, create wizard, detail page, etc.)
  - `admin.buckets.*`
  - `admin.keys.*`
  - `admin.policies.*` (matrix editor, role-assignment UI)
  - `admin.serviceAccounts.*`
  - `admin.audit.*`
  - `admin.system.*` (org capabilities, skin upload)
  - `admin.users.*` (user list, invites)
  - `admin.gateways.*`
  - `auth.login.*`
  - `auth.firstRun.*` (onboarding wizard)
  - `auth.signup.*`
  - `setup.*` (5-minute install wizard)
  - `errors.*` (full-page error states)

Use nested keys for substructure: `pages.admin.clusters.list.pageTitle`, `pages.admin.clusters.create.passwordHint`, etc.

### 3. Add useTranslation hook + extract

For each component file with user-facing strings:

```tsx
import { useTranslation } from "react-i18next";

export function MyPage() {
  const { t } = useTranslation();   // default namespace = "common"
  // For page-specific copy:
  const { t: tp } = useTranslation("pages");
  
  return (
    <div>
      <h1>{tp("admin.clusters.list.pageTitle")}</h1>
      <button>{t("buttons.refresh")}</button>
    </div>
  );
}
```

### 4. Spanish translations

For every key added to `en/common.json` and `en/pages.json`, add the Spanish equivalent to `es/common.json` and `es/pages.json`. Both files must have the SAME key structure — fallbackLng:"en" handles a missing key but the goal is full coverage.

Use natural Spanish, formal-second-person ("usted") for buttons. Don't translate "basement" (product name), AccessKey IDs (BMNT...), role kind strings ("user", "cluster-admin", "ui-admin"), or capability IDs ("bucket:delete").

### 5. Tests

- Don't write new render tests for every extracted page — that's 50+ files.
- DO write 3-4 high-value smoke tests: pick the largest pages (admin/clusters/index, admin/system, admin/policies) and verify they render in Spanish after `i18n.changeLanguage("es")`.
- Add to `frontend/src/shared/i18n/__tests__/i18n.test.ts`: a parity test that fails if any key exists in en but not in es (or vice versa). Pseudocode:
  ```ts
  it("en and es common.json have the same key set", () => {
    expect(keys(enCommon).sort()).toEqual(keys(esCommon).sort());
  });
  ```
