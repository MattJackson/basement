## Tasks (in order)

### 1. internal/auth/policy/enforcer.go (~line 477)

Just a stale comment ("only code marks roles deprecated, never the UI / API"). The Deprecated field on Role still exists in types.go and is fine (operators can mark roles deprecated). Just clean up the comment phrasing — no field change.

### 2. internal/api/admin_audit.go (~line 29)

Drop the `Truncated` field from `auditResponse`. v1.4.0a was years ago; FE consumers all derive truncated state from total/offset/limit now. Search the FE for any `truncated` consumer and confirm none exist (`grep -rn "truncated" frontend/src/`). If the FE still reads it, escalate.

### 3. internal/clustersecret/clustersecret.go (~line 496)

The `MigrateFromJWT` function itself is the v1.12.0a → v1.12.0b migration path. Per [[v2_clean_break]] migration shims go. Drop:
- The function itself
- Its tests in clustersecret_*_test.go
- Any caller in internal/store/connections.go (load path that reads legacy ConfigEnc and migrates to ConfigEncCSK)

But — be careful. If there are still ConfigEnc-only entries on disk in basement.pq.io, removing the bridge will leave those clusters with NO credentials. Two options:
- (a) Add a one-shot boot-time migration that drops un-migrated entries with WARN log (per the bucket_user pattern in v2.0.0a)
- (b) Add an explicit pre-flight check: refuse to start if any cluster has ConfigEnc but no ConfigEncCSK, with a clear error message

Pick (a) for consistency with bucket_user. Log line: `WARN: dropped {N} clusters with legacy JWT-encrypted credentials; re-add them via /admin/clusters`.

### 4. internal/store/users.go (~line 24)

Drop the `OIDCSubject` field entirely. The `Subject` field above it is the canonical replacement. Migration: any User loaded from disk with OIDCSubject set + Subject empty → copy OIDCSubject into Subject, drop OIDCSubject. Log count of migrations at WARN level.

### 5. internal/store/org_capabilities.go (~line 57-61)

Drop the `SkinPolicy` field entirely. Migration on load: any OrgCapabilities with SkinPolicy set should translate per the existing loader logic into `UserOverridableSkin` + `AllowedUserSkins` THEN drop SkinPolicy. Log: `WARN: migrated SkinPolicy={value} -> UserOverridableSkin={bool}+AllowedUserSkins={list}`.

### 6. internal/store/org_capabilities.go (~line 109)

Drop the legacy WebDAV hand-typed field from `GatewaySettings`. Migration on load: if `WebDAV.Enabled` is set and `Protocols["webdav"]` is missing, copy WebDAV into Protocols["webdav"]. Then drop the WebDAV field. Log: `WARN: migrated legacy GatewaySettings.WebDAV to Protocols[webdav]`.

### 7. internal/store/org_capabilities.go (~line 160)

`IsEnabled` falls back to the legacy WebDAV field. With #6 the field is gone — IsEnabled simplifies to just `Protocols[name]` lookup + default-by-name. Drop the legacy branch.
