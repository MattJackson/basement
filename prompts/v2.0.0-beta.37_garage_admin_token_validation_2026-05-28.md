# v2.0.0-beta.37 — garage admin_token validation + clearer surface

## What you're producing

ONE commit, ONE tag. Single durable fix lifted out of beta.36 (item 7
was skipped that cycle — this prompt finishes it).

When a `garage` / `garage-v1` cluster is persisted without an
`admin_token`, the upstream call to Garage returns a raw 403
"Forbidden: Authorization token must be provided" which leaks through
the UI as an unreadable error blob. The cluster's persisted config can
miss `admin_token` because the `POST /admin/clusters` and
`PATCH /admin/clusters/{cid}` API handlers don't validate it (unlike
the env-var path which `internal/config/config.go:227-228` already
requires).

## Repro (from the operator's live cluster)

```bash
curl -s -X POST -H 'Content-Type: application/json' \
  -d '{"username":"matthew","password":"password"}' \
  https://basement.pq.io/api/v1/auth/login -c /tmp/c.txt -o /dev/null
curl -s -X POST -b /tmp/c.txt -c /tmp/c.txt -H 'Content-Type: application/json' \
  -d '{"target_mode":"admin","password":"password"}' \
  https://basement.pq.io/api/v1/auth/elevate -o /dev/null
curl -s -X PUT -b /tmp/c.txt -c /tmp/c.txt -H 'Content-Type: application/json' \
  -d '{"kind":"ui-admin"}' https://basement.pq.io/api/v1/auth/active-role -o /dev/null
curl -s -b /tmp/c.txt https://basement.pq.io/api/v1/admin/clusters | python3 -m json.tool
```

The "Classe" cluster has `"config": {"admin_url": "..."}` — no
`admin_token` field at all. Health probes against it return:

```
✗ Health check failed: driver(garage-v1).GET: { "code": "AccessDenied",
"message": "Forbidden: Authorization token must be provided",
"region": "garage", "path": "/v1/health" }: driver: permission denied
```

## Three-layer fix

### Layer 1 — driver-level pre-check

In the garage driver. First find it:

```bash
grep -rn "admin_token\|adminToken" /Users/mjackson/Developer/basement-ui/internal/driver/ | head -20
```

Wherever the driver reads `cfg["admin_token"]` and uses it as a bearer
token, add a guard BEFORE any HTTP call:

```go
adminToken := cfg["admin_token"]
if adminToken == "" {
    return nil, &DriverError{
        Code:    "MISSING_ADMIN_TOKEN",
        Message: "Garage admin token is not configured for this cluster. Edit the cluster to provide it.",
    }
}
```

Use the existing error pattern — read `internal/driver/driver.go`
(or wherever `DriverError` / error sentinels live) and match the
convention. If a typed error already exists for "missing config"
(e.g. `ErrMissingConfig`), use it; otherwise add a typed sentinel.

The goal: callers (test/health/list/etc.) get a structured error they
can map to HTTP 422 instead of the driver issuing a bound-to-fail HTTP
request.

### Layer 2 — API handler validation

The `POST /admin/clusters` and `PATCH /admin/clusters/{cid}` handlers
need symmetric validation. Find them:

```bash
grep -rn "func.*createCluster\|func.*updateCluster\|func.*patchCluster" /Users/mjackson/Developer/basement-ui/internal/api/ | head
```

Add validation that mirrors `internal/config/config.go:223-229`: if
the driver name is `garage` or `garage-v1` AND `config.admin_token`
is empty, return HTTP 422 with body
`{"error":{"code":"MISSING_ADMIN_TOKEN","message":"…"}}`. Match the
existing error-response shape — look at how other 4xx responses are
emitted in the same file.

Same validation MUST apply on PATCH: don't let an operator
accidentally clear the token by submitting a partial config without
it. The handler should preserve existing token if not present, OR
explicitly require it on every PATCH (whichever matches the
existing patch semantics in the file — confirm by reading the
PATCH handler).

### Layer 3 — frontend surface

Two places.

**3a — Cluster detail page test/health banner.**
`frontend/src/routes/admin/clusters/$cid/index.tsx` renders the test
result. When the response carries `code === "MISSING_ADMIN_TOKEN"`,
render an amber-banner CTA instead of the raw blob. Add an i18n key:

```
adminClustersDetail.missingAdminTokenBanner = "Admin token isn't set
for this cluster — add it on the Edit page so the health check can
authenticate."
```

Add to both `en/pages.json` and `es/pages.json` under
`adminClustersDetail`. The CTA links to
`/admin/clusters/{cid}/edit`.

Find the existing test-result rendering in the file:

```bash
grep -n "testResult\|test_result\|testQuery\|Health check" /Users/mjackson/Developer/basement-ui/frontend/src/routes/admin/clusters/\$cid/index.tsx | head -10
```

Wrap the banner branch around a check for the new error code.

**3b — Cluster edit page form-level validation.**
`frontend/src/routes/admin/clusters/$cid/edit.tsx` and
`frontend/src/routes/admin/clusters/new.tsx`: when the user submits a
garage / garage-v1 cluster without an admin token, the backend now
returns 422 MISSING_ADMIN_TOKEN. Catch that error in the form's
error handler and surface as a field-level message on the admin_token
input (or as a top-of-form error if no admin_token input exists yet).

Read the existing error-handling in the submit handlers and add a
specific branch for `MISSING_ADMIN_TOKEN` that renders the message
near the relevant input rather than as a toast.
