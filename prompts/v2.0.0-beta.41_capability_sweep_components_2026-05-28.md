# v2.0.0-beta.41 — ADR-0009 Phase E: component sweep

## What you're producing

ONE commit, ONE tag. Mechanical sweep — replace every
`{activeRole?.kind === "ui-admin" && ...}` /
`{activeRole?.kind === "cluster-admin" && ...}` /
`isUIAdminActive ? ... : ...` site in the frontend components with
`useCan(...)` reading the capability table.

This is the final phase of ADR-0009. After this cycle, the codebase
has no more "role kind" pattern checks in components — every gate
goes through `useCan`.

## Prerequisites already shipped

- beta.38: backend capability table + `/auth/me` carries `capabilities[]`
- beta.39: backend middleware sweep
- beta.40: FE useCan hook + ProtectedRoute rewrite

## Mapping table

Each site needs to convert the role-kind test into a capability test.
Reference table (extend as needed):

| Site / context | Old gate | New gate |
|---|---|---|
| AppShell top nav: Buckets link | `isUIAdminActive` | `useCan(Cap.ClusterBucketsAggr)` |
| AppShell top nav: Usage link | `isUIAdminActive` | `useCan(Cap.ClusterUsageAggr)` |
| AppShell top nav: Clusters link | `isUIAdminActive` | `useCan(Cap.ClusterWiringList)` |
| AppShell top nav: Policies / Service accounts / Audit / System / Users links | `isUIAdminActive` | corresponding `useCan(Cap.Platform*)` |
| Cluster edit page "Per-user S3 credentials see /admin/users" line | `activeRole?.kind === "ui-admin"` | `useCan(Cap.PlatformUsersList)` |
| Cluster detail "Back to Clusters" link | `activeRole?.kind !== "cluster-admin"` | `useCan(Cap.ClusterWiringList)` (only those who can see the list should see the back link) |
| Cluster detail "Delete cluster" button | `activeRole?.kind === "ui-admin"` | `useCan(Cap.ClusterWiringDelete)` |
| Cluster detail "Edit cluster" button | (currently shown to ui-admin) | `useCan(Cap.ClusterWiringUpdate)` |
| Bucket detail "Lifecycle" / "Delete bucket" buttons | check current gates | `useCan(Cap.ClusterBucketsLifecycleWrite)` / `useCan(Cap.ClusterBucketsDelete)` |
| Encryption admin "Add admin" / "Remove" buttons | likely no gate today | `useCan(Cap.ClusterEncryptionAdminsAdd)` / `useCan(Cap.ClusterEncryptionAdminsRm)` |

For sites NOT in this table, look at what the gate is protecting and
pick the matching capability. When unsure, ask the operator —
DO NOT invent a capability not in `capabilities.ts`.

## Special cases — KEEP role-kind checks

These are legitimate uses of `activeRole.kind` that should NOT be
swept (they're about WHICH SURFACE / SHELL is active, not about
permission):

- `frontend/src/components/layout/PersonaPill.tsx` — the pill
  literally renders the role badge and switches between role
  options. Role kind is the right primitive here.
- `frontend/src/shared/layout/AppShell.tsx` — the SHELL CHOICE
  (which top-nav config to render) is driven by role kind, because
  it's about visual identity not permissions. Individual NAV LINKS
  inside each shell are gated by useCan; the shell skeleton itself
  uses role kind.
- Landing redirects in `ProtectedRoute` and elsewhere — landing on
  /admin/system vs /admin/clusters/{cid} is about role identity, not
  capability.

Document these in code comments referencing ADR-0009 so future
maintainers know not to sweep them.

## Process

1. `git -C /Users/mjackson/Developer/basement-ui pull --ff-only origin main`
2. Build the checklist from the grep.
3. For each site, decide: capability check or legitimate role check?
4. Convert capability-check sites to `useCan(...)`.
5. Where the operator's intent for a site is unclear, leave the
   site UNTOUCHED and mark with `// TODO ADR-0009: capability TBD`.
   Better to flag than to invent.
6. `pnpm tsc --noEmit && pnpm build && pnpm test:run` — all green.
7. Stage explicit files. ONE commit:
   `v2.0.0-beta.41: ADR-0009 Phase E — component sweep to useCan`
8. Push HEAD:main, tag, push tag, ls-remote.
9. Verify live: as UI Admin, all platform/wiring affordances are
   visible; cluster contents controls are hidden. As cluster-admin,
   cluster contents are visible; wiring controls are hidden.

## Anti-fabrication gates (NON-NEGOTIABLE)

- `git status --short` before commit
- `git -C <absolute-path>` always
- Stage by file name. NEVER `git add .` or `-A`
- NO `--no-verify` / `Co-authored-by` / `--force`
- ONE commit + ONE tag this cycle
- DO NOT exit before pushing — escalate if blocked
- **Run `pnpm build` and `pnpm test:run` — all green required.**
- DO NOT invent capabilities not in capabilities.ts
- DO NOT delete the legitimate-role-check sites
