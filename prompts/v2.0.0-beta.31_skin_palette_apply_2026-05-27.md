# v2.0.0-beta.31 — skins actually re-color the UI

## What you're producing

ONE commit, ONE tag. Single bug fix:

**S1** — Switching skin (Terminal, Windows 95, etc) only changes the
font, not the colors. Operator expects Terminal to be black+green and
Windows 95 to be gray. Both still render with default basement
colors after a skin switch.

## Root cause (TWO bugs compound)

`frontend/src/shared/hooks/useSkin.ts:78-104` has a `useEffect` that
writes palette tokens to `documentElement.style` as
`--basement-${key}`. Two things are broken:

### Bug 1 — key name mismatch

The hook iterates these keys:
```ts
const tokens = ["primary", "background", "foreground", "muted", "accent",
                "destructive", "warning", "success", "info"];
```

But the backend palette response uses SHORT names (verified via curl
against beta.28+):
```json
"palette": {
  "light": { "primary": "...", "bg": "...", "fg": "...", "muted": "...", ... }
}
```

So `variant["background"]` is `undefined` — the hook skips writing
background AND foreground. Only primary/muted/accent/destructive/warning/success/info
get written. The bg/fg fields silently no-op.

### Bug 2 — wrong CSS variable prefix

The hook writes `root.style.setProperty(\`--basement-${key}\`, value)`.
But the app's Tailwind components reference `--color-${name}`
(see `frontend/src/index.css:19-45` — `--color-background`,
`--color-foreground`, `--color-primary`, etc).

`--basement-*` is set, but nothing READS from `--basement-*`. So
even the keys that DO match end up writing to dead variables.

## The fix

Edit `frontend/src/shared/hooks/useSkin.ts:78-104` so the `useEffect`:
1. Maps palette field names to CSS variable names (bg→background, fg→foreground).
2. Writes to `--color-${cssName}` not `--basement-${key}`.
3. Includes `*-foreground` variants where possible (the palette has
   `primary` but the CSS theme also wants `--color-primary-foreground`).
   For now, leave foreground-pair tokens alone (use the theme defaults);
   only override the BASE color values from the palette.

Concretely, replace the existing useEffect body with:

```ts
useEffect(() => {
  if (!skin?.palette) return;
  const variant = isDark ? skin.palette.dark : skin.palette.light;
  if (!variant) return;
  const root = document.documentElement;
  // Map backend palette field names → frontend CSS variable suffix
  // (Palette type in internal/skin/skin.go uses bg/fg short forms;
  // frontend index.css uses --color-background / --color-foreground.)
  const mapping: Array<[keyof typeof variant, string]> = [
    ["primary", "primary"],
    ["bg" as keyof typeof variant, "background"],
    ["fg" as keyof typeof variant, "foreground"],
    ["muted", "muted"],
    ["accent", "accent"],
    ["destructive", "destructive"],
    ["warning", "warning"],
    ["success", "success"],
    ["info", "info"],
  ];
  for (const [paletteKey, cssName] of mapping) {
    const value = (variant as Record<string, string | undefined>)[paletteKey as string];
    if (typeof value === "string" && value) {
      root.style.setProperty(`--color-${cssName}`, `hsl(${value})`);
    }
  }
  // Common derived tokens follow the base palette so cards / popovers /
  // borders pick up the new colors without each skin needing to specify
  // them explicitly. These are the same defaults pattern index.css uses.
  const bg = (variant as Record<string, string | undefined>).bg;
  const fg = (variant as Record<string, string | undefined>).fg;
  if (bg) {
    root.style.setProperty("--color-card", `hsl(${bg})`);
    root.style.setProperty("--color-popover", `hsl(${bg})`);
  }
  if (fg) {
    root.style.setProperty("--color-card-foreground", `hsl(${fg})`);
    root.style.setProperty("--color-popover-foreground", `hsl(${fg})`);
  }
}, [skin?.palette, isDark]);
```

Notes:
- The palette values are HSL "space-separated H S L" strings (e.g.
  "142 100% 50%" for Terminal green). The CSS variables in index.css
  use `hsl(...)` form (e.g. `--color-primary: hsl(240 5.9% 10%);`),
  so wrap in `hsl(...)`.
- Don't touch `--color-primary-foreground`, `--color-muted-foreground`,
  etc — those are paired/derived. Leave theme defaults so contrast
  stays correct.
- The frontend `Palette` TypeScript type also needs the `bg`/`fg` field
  names. Check `frontend/src/shared/api/queries.ts` for the
  `interface Palette` declaration and align if needed.
