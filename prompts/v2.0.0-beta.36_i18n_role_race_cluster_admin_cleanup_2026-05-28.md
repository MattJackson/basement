## 2. "Switch to an admin role to access this page." race

### Repro

Operator on `/admin/system` (or any UI-admin page) selects "Cluster
Admin: default" from the PersonaPill. The role switches, but the toast
"Switch to an admin role to access this page." fires and they bounce
to `/files`.

### Root cause

`frontend/src/shared/api/mutations.ts:384-405` —
`useSwitchActiveRole`'s `onSuccess` only invalidates the `auth/me`
query:

```ts
onSuccess: () => {
  queryClient.invalidateQueries({ queryKey: ["auth", "me"] });
},
```

`handleRoleChange` in `PersonaPill.tsx:122-134` awaits the mutation
then immediately calls `navigate({ to: "/admin/clusters/{cid}" })`.
The invalidate fires a refetch but doesn't wait for it. The next
render's `useUser()` still returns the stale `activeRole = "user"`.
`ProtectedRoute` (`shared/auth/ProtectedRoute.tsx:76-79`) checks
`isClusterScopedRoute && activeRole.kind === "user"` and bounces with
"Switch to an admin role to access this page."

### Fix

The mutation's response body IS the new `UserResponse` (line 398
returns `(await res.json()) as UserResponse`). Use `setQueryData` to
update the cache synchronously before the navigate happens — no race.

```ts
onSuccess: (user) => {
  queryClient.setQueryData(["auth", "me"], user);
  queryClient.invalidateQueries({ queryKey: ["auth", "me"] });
},
```

`setQueryData` updates synchronously. The follow-up `invalidate` is
defensive (refresh stale fields like `mode` / `expiresAt` if backend
mutated them too).

## 5. Lifecycle "Couldn't load lifecycle policy."

`shared/ui/LifecycleSection.tsx:43-52` renders the red banner when
the `useBucketLifecycle` query errors. Operator reports seeing this
on at least one bucket detail page.

Senior probe showed the live endpoint
`/api/v1/admin/clusters/<CID>/buckets/<BID>/lifecycle` returns 200
for the first bucket of the default cluster:
`{"rules":[],"capabilities":{"supported":false,...}}`. With
`supported: false`, the component SHOULD render the gray
"Lifecycle policies not supported on this driver" pill (lines 56-69),
NOT the red error banner.

So either:
- (a) the operator hit a bucket where the request actually errors
  (different bucket, transient 5xx)
- (b) the query is erroring intermittently due to auth race during
  role-switch (related to item 2)
- (c) there's a logic bug in error-handling

### Action

Read `LifecycleSection.tsx` carefully + the `useBucketLifecycle`
hook in `shared/api/queries.ts:1540-1556`. The query has
`retry: 1`. If the first call hits an auth race (cookie not yet
persisted from a recent re-elevation) the second retry should
succeed. But if the FE renders the banner before the retry resolves,
that's a perceived bug.

Make the error banner more diagnostic: include the underlying error
message instead of the generic copy. That way next time it happens
we'll know which error code we're staring at.

```tsx
<p className="text-sm text-destructive">
  Couldn't load lifecycle policy{error?.message ? `: ${error.message}` : "."}.
</p>
```

Also i18n-wrap this string (key
`bucketLifecycle.couldntLoadLifecyclePolicy` or similar — check the
existing namespace for lifecycle copy in pages.json).

If the operator's actual bucket reproduces the issue after deploy,
they'll have a usable error code to share next cycle.

contribute to cell sizing. One clean option: render the hint as a

    <Input id="s3Url" … />

    <Label htmlFor="s3Region">S3 Region</Label>

## 7. Garage cluster shows raw "Forbidden: Authorization token must be provided"

### Repro

Operator viewing `/admin/clusters/{cid}` (cluster "Classe", driver
`garage-v1`, admin_url `http://10.1.7.10:3903`) sees in the test/health
panel:

```
✗ Health check failed: driver(garage-v1).GET: { "code": "AccessDenied",
"message": "Forbidden: Authorization token must be provided",
"region": "garage", "path": "/v1/health" }: driver: permission denied
```

### Root cause

Cluster's persisted config has `admin_url` but **no** `admin_token`:

```json
{"id":"02c3e5ff-…","label":"Classe","driver":"garage-v1",
 "config":{"admin_url":"http://10.1.7.10:3903"}}
```

`BASEMENT_DRIVER_GARAGE_ADMIN_TOKEN` is required by
`internal/config/config.go:227-228` for env-seeded clusters, but
clusters created/edited via the UI (`POST /admin/clusters` /
`PATCH /admin/clusters/{cid}`) can be persisted without an admin token
because there's no equivalent validation in the API handler. When the
driver later calls Garage's admin API, the request goes out without an
`Authorization: Bearer …` header and Garage returns 403.

### Fix (two layers)

**Layer 1 — driver-level pre-check.** In the garage driver (file
location TBD; grep for the function that handles the health/test
path). When `admin_token` is empty in the connection config, return a
structured error instead of issuing the upstream HTTP call:

```go
if cfg["admin_token"] == "" {
    return fmt.Errorf("driver: admin_token missing — edit the cluster to provide BASEMENT_DRIVER_GARAGE_ADMIN_TOKEN equivalent")
}
```

Or, better, a typed error the API layer can translate (e.g. a sentinel
`ErrMissingAdminToken` so handlers can return an HTTP 422 with a
machine-readable code like `MISSING_ADMIN_TOKEN`). Check the existing
error model in `internal/driver/driver.go` for the convention.

**Layer 2 — API handler validation.** The `POST /admin/clusters` and
`PATCH /admin/clusters/{cid}` handlers should refuse to persist a
garage/garage-v1 cluster without `admin_token` set. Return 400 with
`MISSING_ADMIN_TOKEN` so the frontend can render a field-specific
validation error on the form, not just a generic 400.

Find the handler in `internal/api/clusters.go` or similar (grep
`func.*createCluster\|func.*updateCluster`). Add validation:

```go
if (driver == "garage" || driver == "garage-v1") && config["admin_token"] == "" {
    return errors.New("admin_token required for garage drivers")
    // translate to MISSING_ADMIN_TOKEN/422 via the existing error model
}
```

### Frontend surface

On `/admin/clusters/{cid}` index, when the test/health call returns
the new `MISSING_ADMIN_TOKEN` code, render an amber-banner CTA pointing
at `/admin/clusters/{cid}/edit` instead of the raw error blob. Same
key in en + es JSON. Add a translation key like
`adminClustersDetail.missingAdminTokenBanner`: "Admin token isn't set
for this cluster — add it on the Edit page so the health check can
authenticate."

### Out of scope

DO NOT auto-PATCH the running operator cluster. Recovery is a separate
manual step the operator will run. This item ONLY adds durable
prevention + better surface for the failure mode.

### Verify

After deploy:
1. Read the current Classe cluster — it will still surface
   `MISSING_ADMIN_TOKEN` until operator PATCHes the token.
2. Verify the new banner text appears (not the raw "Forbidden:
   Authorization token must be provided" string).
3. Attempt to create a NEW cluster via `POST /admin/clusters` with
   `driver=garage-v1` and no `admin_token` — expect HTTP 422 with
   code `MISSING_ADMIN_TOKEN`, not a successful create.
