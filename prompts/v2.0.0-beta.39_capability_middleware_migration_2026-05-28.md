# v2.0.0-beta.39 — ADR-0009 Phase C: backend middleware migration

## What you're producing

ONE commit, ONE tag. Migrate backend route gating from scattered
`auth.RequireRole(...)` / `auth.ActiveRoleXxxMiddleware()` to a new
`auth.RequireCapability(cap)` middleware that reads the capability
table landed in beta.38 (`internal/auth/capabilities.go`).

This is the third phase of the ADR-0009 capability-based RBAC
migration. See `docs/adr/0009-capability-based-rbac.md` for the full
plan.

## Why

The "UI Admin is super-admin for cluster routes" branch at
`internal/auth/active_role.go:68-72`:

```go
// UI Admin is super-admin — passes any cluster route
if ar.Kind == "ui-admin" {
    next.ServeHTTP(w, r)
    return
}
```

violates the locked wiring-vs-contents split. UI Admin should only
hold `cluster.wiring.*` capabilities — NOT `cluster.contents.*`.

Today's middleware applies role-kind matching directly. Tomorrow's
middleware reads the capability table.

## The migration

### 1. Add `RequireCapability` middleware

In `internal/auth/middleware.go` (or `active_role.go` — match the
file's convention), add:

```go
// RequireCapability returns middleware that allows the request only
// when the caller's active role grants `capability`. Cluster-scoped
// capabilities ALSO require the chi URLParam "cid" to match
// activeRole.Cluster.
//
// Capability strings are defined in capabilities.go.
func RequireCapability(capability string) func(http.Handler) http.Handler {
    return func(next http.Handler) http.Handler {
        return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            claims, ok := FromContext(r.Context())
            if !ok || claims == nil || claims.ActiveRole == nil {
                http.Error(w, `{"error":{"code":"UNAUTHORIZED","message":"No active role in session"}}`, http.StatusUnauthorized)
                return
            }
            clusterID := ""
            if IsClusterScopedCapability(capability) {
                clusterID = chi.URLParam(r, "cid")
            }
            if !Can(claims, capability, clusterID) {
                http.Error(w, `{"error":{"code":"FORBIDDEN","message":"Missing required capability: `+capability+`"}}`, http.StatusForbidden)
                return
            }
            next.ServeHTTP(w, r)
        })
    }
}
```

Add a unit test in `middleware_test.go` covering:
- ui-admin + cluster.wiring.update → 200 (no cid needed)
- ui-admin + cluster.buckets.create → 403 (NOT a super-admin)
- cluster-admin + cluster.buckets.create + matching cid → 200
- cluster-admin + cluster.buckets.create + mismatched cid → 403
- user + any cluster cap → 403
- nil claims → 401

### 2. Replace mount sites in server.go

For each mount site identified in the investigation, choose the
correct capability. Reference table:

| Current middleware | Route surface | New capability |
|---|---|---|
| `RequireRole("admin")` on `/admin/*` (the catch-all auth gate) | KEEP — this is the "you are authenticated as some admin" baseline. Capability checks are FINER inside. | (unchanged) |
| `ActiveRoleUIAdminMiddleware()` on platform routes (`/admin/users`, `/admin/policies`, `/admin/audit`, `/admin/system`, `/admin/skins`, `/admin/oidc`, `/admin/onboarding`, `/admin/service-accounts`) | per-route | one of `CapPlatformUsersList` / `CapPlatformPoliciesList` / `CapPlatformAuditRead` / `CapPlatformSystemWrite` / etc. — match the route's operation |
| `ActiveRoleClusterMiddlewareFromPath()` on cluster routes | various | DEPENDS on what the route does: wiring (UI Admin) → `CapClusterWiring*`; contents (Cluster Admin) → `CapClusterContents*` / `CapClusterBuckets*` / etc. |
| `ActiveRoleAnyAdminMiddleware()` on /admin/* baseline | KEEP for the chi group baseline; per-route caps refine | (unchanged) |

**Critical scoping example:**

`PATCH /admin/clusters/{cid}` is cluster WIRING, so:
```go
clusters.Patch("/{cid}", auth.RequireCapability(auth.CapClusterWiringUpdate), s.updateClusterHandler)
```
NOT `CapClusterContentsRead`. The operator can edit any cluster's
wiring as UI Admin — wiring is NOT cluster-scoped.

`GET /admin/clusters/{cid}` (the detail page reads bucket/key/admin
contents):
```go
clusters.Get("/{cid}", auth.RequireCapability(auth.CapClusterContentsRead), s.getClusterDetailHandler)
```
This IS cluster-scoped — cluster-admin on cluster-a can't read
cluster-b's contents. Middleware enforces via chi URLParam("cid")
matching activeRole.Cluster.

`POST /admin/clusters/{cid}/buckets`:
```go
auth.RequireCapability(auth.CapClusterBucketsCreate)
```

`GET /admin/buckets` (cross-cluster aggregate):
```go
auth.RequireCapability(auth.CapClusterBucketsAggr)
```

Work through every cluster route carefully. When in doubt about
wiring vs contents, re-read the ADR.

### 3. Remove the UI-Admin super-admin branch

`internal/auth/active_role.go:68-72` (the "UI Admin is super-admin"
branch in `ActiveRoleClusterMiddlewareFromPath`):

DELETE IT. The capability table now correctly does NOT grant UI Admin
cluster-contents capabilities — the middleware swap above takes care
of routing UI Admin to the right capabilities (wiring) instead.

If the old middleware is still referenced by any route NOT yet
migrated to RequireCapability, keep the old middleware AROUND for
those routes — but the super-admin branch goes regardless. After this
cycle every cluster route must be on RequireCapability OR the old
middleware (without the super-admin branch).

### 4. Update existing tests

Every test that elevates as UI Admin and expects to access cluster
contents will now correctly get 403. Update those tests to either:
- Switch to a cluster-admin role for cluster contents tests
- Switch to UI Admin for wiring tests
- Document the expected 403 if testing the deny path

This will surface every place a test relied on the "super-admin"
behavior. That's the point of the migration — those were the bug
class.

### 5. Smoke

The Playwright smoke harness exercises live /admin/* endpoints. Run
it locally if convenient (operator's local has Playwright); otherwise
verify the live behavior post-deploy.
