# Cycle: v1.13.18 — Active-role selector (single active role at a time)

Replace the binary "Switch to admin view / Switch to user view" with a Role selector — dynamic, one active role at a time, drives every persona-dependent UI surface.

## Model (operator-locked)

Three possible roles. User can be eligible for multiple, but ONE active at a time:

```
Active Role: [User ▾]
              User                       (always available)
              Cluster Admin: <cluster>   (per cluster they admin; can have many)
              UI Admin                   (only if user is uiAdmin)
```

**Eligibility**:
- User: every authenticated user
- Cluster Admin: gated per cluster — user has the `cluster:admin` capability on that specific cluster. Multiple cluster grants = multiple options in the role menu.
- UI Admin: gated on `user.uiAdmin === true`

**Switching**:
- → User: free (drop privileges)
- → Cluster Admin (any cluster): elevation prompt (sudo re-auth)
- → UI Admin: elevation prompt (sudo re-auth)
- Already elevated + same role: no-op
- Cross-role switch (e.g. Cluster Admin classe → UI Admin): re-elevation prompt because the scope changes

## Persisted state

`/auth/me` response includes:
```
{
  username, role, uiAdmin, ...,
  activeRole: { kind: "user" | "cluster-admin" | "ui-admin", cluster?: string },
  availableRoles: [
    { kind: "user", label: "User" },
    { kind: "cluster-admin", cluster: "classe", label: "Cluster Admin: classe" },
    { kind: "cluster-admin", cluster: "lsi", label: "Cluster Admin: lsi" },
    { kind: "ui-admin", label: "UI Admin" },
  ]
}
```

New endpoint `PUT /api/v1/auth/active-role`:
```json
{ "kind": "ui-admin" }            // or { "kind": "cluster-admin", "cluster": "classe" }
```
- Validates user is eligible for the requested role (cluster grant or uiAdmin)
- If kind requires elevation AND user isn't elevated → return 423 LOCKED with `{requires_elevation: true, prompt: "..."}` (FE pops elevation modal, retries)
- On success: persist activeRole in session + return updated /auth/me

## Frontend changes

### UserMenu

REPLACE the current "Switch to admin view / Switch to user view" group with:

```tsx
<DropdownMenuSub>
  <DropdownMenuSubTrigger>
    Role
  </DropdownMenuSubTrigger>
  <DropdownMenuSubContent>
    <DropdownMenuRadioGroup value={activeRoleKey} onValueChange={handleRoleChange}>
      {availableRoles.map(r => (
        <DropdownMenuRadioItem value={roleKey(r)} key={roleKey(r)}>
          {r.label}
        </DropdownMenuRadioItem>
      ))}
    </DropdownMenuRadioGroup>
  </DropdownMenuSubContent>
</DropdownMenuSub>
```

`handleRoleChange` calls PUT `/auth/active-role`; on 423 LOCKED, pop elevation modal + retry; on success, invalidate `["auth","me"]` query + navigate to that role's landing route (User → /files, Cluster Admin → /admin/clusters/{cid}, UI Admin → /admin/system).

### PersonaPill

Dynamic per active role:
- User: shows username + "User" tag
- Cluster Admin: shows username + "Cluster Admin: classe" tag (or whichever cluster)
- UI Admin: shows username + "UI Admin" tag + admin-session-TTL countdown if elevated

### Admin nav (in UserMenu dropdown + any sidebar)

Show ONLY links appropriate for activeRole:
- User: nothing (just /files navigation in main shell)
- Cluster Admin (cluster X): /admin/clusters/{X}/{buckets,keys,layout,users} only
- UI Admin: /admin/{system,skins,policies,oidc-mappings,audit} only

### Route gating

Each `/admin/*` route in `frontend/src/routes/admin/` checks activeRole:
- UI Admin routes: redirect to /files if activeRole !== "ui-admin"
- Cluster Admin routes: redirect to /files if activeRole !== "cluster-admin" OR if the URL's {cid} doesn't match activeRole.cluster
- Cross-cluster admin routes: redirect to per-cluster page in active role's cluster

## Backend changes

`internal/auth/`:
- Add `ActiveRole struct { Kind string; Cluster string }` to the session payload
- Default activeRole on first login = "user"
- New handler `PUT /api/v1/auth/active-role` that validates + persists
- Cookie includes activeRole (encrypted, same pattern as existing elevation marker)

`internal/api/auth.go` /me response:
- Include `activeRole` + `availableRoles` (computed from user's grants)

Per-route middleware:
- For /admin/system/*, /admin/skins/*, /admin/policies/*, /admin/oidc-*, /admin/audit: gate on activeRole == "ui-admin"
- For /admin/clusters/{cid}/...: gate on activeRole == "cluster-admin" && activeRole.cluster == cid

Backwards-compat: existing "uiAdmin + elevated" still gates server-side as a safety net; activeRole is the FE-driven layer.
