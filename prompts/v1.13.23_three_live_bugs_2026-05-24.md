## Bug A — Role selector silently drops Cluster Admin

### Symptom

Operator (UI Admin on basement.pq.io) sees only "User" and "UI Admin" in the role selector dropdown, NOT any "Cluster Admin: <cluster>" entries. They can SEE clusters in the UI (because UI Admin sees everything) but can't EDIT them from User mode. The Cluster Admin option is missing entirely from the role picker.

### Root cause (already located)

`internal/api/auth.go` `computeAvailableRoles` (~line 150):

```go
if a.Scope == "cluster:*" {
    // For wildcard cluster admin, we need to list actual clusters — for now, skip
    continue
}
```

Wildcard `cluster:*` grants are silently skipped with a TODO. Operator likely has a wildcard grant (UI Admins usually do).

Also: line 165 — `strings.HasPrefix(a.RoleID, "cluster_admin") || a.RoleID == "cluster_admin"` — the OR is redundant (HasPrefix covers exact match). Clean up while you're there.

### Fix

1. When `a.Scope == "cluster:*"`, enumerate all known clusters via `s.drv.Clusters()` or whatever the registry call is (read `internal/api/server.go` and `internal/api/clusters.go` to find the right call — there's already a list-clusters handler, follow its pattern). Append one `AvailableRole{Kind: "cluster-admin", Cluster: <cid>, Label: "Cluster Admin: <name>"}` per cluster.
2. **Also**: if `claims.UIAdmin == true`, treat that as implicit cluster admin on EVERY cluster (per the operator's clear expectation that UI Admin >= Cluster Admin). De-dupe with explicit grants — don't list a cluster twice.
3. Use cluster *name* (not just cid) in the Label when available — match how the cluster list UI labels them. Fall back to cid if name is empty.
4. Drop the redundant `|| a.RoleID == "cluster_admin"`.

## Bug B — https://basement.pq.io/webdav returns "Not Found"

### Symptom

Operator hits `https://basement.pq.io/webdav` in browser → "Not Found".

### Investigation needed

`internal/api/server.go:756-757` does:
```go
r.Handle("/webdav", s.webdavRouter())
r.Handle("/webdav/*", s.webdavRouter())
```

So the route IS registered. Why is it returning 404? Possibilities (investigate each):

1. **Reverse proxy strip**: Caddy/nginx in front of basement.pq.io may be stripping `/webdav` before it reaches the Go server. Check `docs/deployment/` for proxy config snippets. The webdav listener may be on a different port that's not exposed.
2. **Gateway disabled in config**: WebDAV may need explicit enablement in `config.yaml` / OrgCapabilities. Check `internal/api/admin_gateways.go` and `internal/gateway/webdav/*.go` for an Enabled flag.
3. **404 from inside the webdav router itself**: GET `/webdav` (no path) may not match any handler in `webdavRouter()`. Some WebDAV impls require PROPFIND with depth and don't respond to plain GET with a useful body — but a 404 vs a 401 vs a directory listing matters. The browser may need either a friendlier landing page OR a 401 challenge to let the user mount.
4. **WebDAV mounted only behind auth and unauthenticated request lands at 404 instead of 401**: check the middleware chain.

### Fix

Whichever of the above is the actual cause:
- If proxy strip: document in `docs/deployment/proxy.md` (create or extend) AND, separately, when `r` receives `/webdav` GET from a browser, return either a 401 with a `WWW-Authenticate: Basic realm="basement WebDAV"` challenge OR a brief HTML page explaining "this URL is for WebDAV clients, not browsers — see /docs/integrations/webdav for setup".
- If gateway disabled: surface a clear "WebDAV gateway is not enabled in this deployment. Enable it in admin → gateways." message instead of bare 404.
- If router itself: fix the missing handler.

The bar: the operator should hit `/webdav` and get a USEFUL response, not a bare 404. Either a WebDAV challenge (so their Finder/Cyberduck client can connect), or an HTML page that explains what /webdav is for.

### Acceptance

- `curl -i https://basement.pq.io/webdav` returns a non-404 response (either 200 HTML, 401 with WWW-Authenticate, or a meaningful redirect to /docs/integrations/webdav)
- Document the actual root cause in the commit message
- If the fix is a config/proxy issue, add a note to docs/deployment/ that explains it

## Bug C — Public docs serve raw `.md` files; operator wants HTML

### Symptom

`https://basement.pq.io/docs/integrations/webdav.md` returns either 404 or raw markdown. Operator: "public docs should be html".

### Investigation needed

`docs/integrations/webdav.md` exists in the repo (confirmed). Check how docs are served at runtime:

1. Is there a `/docs/*` route handler in `internal/api/server.go`? (Earlier grep found none.) If not, the request is probably falling through to the SPA fallback which serves index.html — and the browser doesn't render .md as HTML.
2. Is there a separate static-file server for docs? Look in `internal/api/static.go` or similar.
3. Are docs embedded via `embed.FS`? Check `cmd/basement/main.go` and any `//go:embed docs` directives.

### Fix

Add a `/docs/*` handler that:
1. Resolves the request path to a file under `docs/` (embedded via `//go:embed docs/**/*.md`)
2. If the path is `*.md`, render the markdown to HTML using a minimal Go markdown lib (`github.com/yuin/goldmark` is the standard pick — already in go.mod? check first; if not, add it)
3. Wrap the rendered HTML in a SHARED minimal template that includes basement's skin CSS variables so the rendered docs match the app's look
4. If the path is a directory, list its contents (or render an index.md if present)
5. If the path is `*.html` already, serve as-is
6. URL form: `/docs/integrations/webdav` (without .md) should ALSO work — make .md optional
7. Cache headers: ETag based on file hash, allow proxy caching
