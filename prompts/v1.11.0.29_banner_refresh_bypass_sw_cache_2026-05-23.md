# Cycle: v1.11.0.29 ‚Äî NewVersionBanner Refresh must actually pick up new bundle

**Operator bug report**: "refresh on update banner updates but doesn't hide banner"

When user clicks Refresh on the NewVersionBanner, page reloads but the banner re-appears immediately. Cause: PWA service worker (registered in v1.8.0e via vite-plugin-pwa) intercepts the index.html fetch and serves the cached OLD bundle, so `__BUILD_COMMIT__` is still stale ‚Üí banner re-mismatches.

## Read first

- `frontend/src/shared/ui/NewVersionBanner.tsx` ‚Äî current Refresh handler is just `window.location.reload()`
- `frontend/src/shared/api/buildWatcher.ts` ‚Äî the watcher that flips `mismatched`
- `frontend/vite.config.ts` ‚Äî vite-plugin-pwa registration (service worker generation config)
- `frontend/src/main.tsx` or wherever the SW gets registered (look for `navigator.serviceWorker.register` OR `registerSW` from `virtual:pwa-register`)

## Fix

Update the Refresh handler in `NewVersionBanner.tsx` to:

1. Unregister the active service worker (so it stops serving cached index.html)
2. Clear the Cache Storage API entries that hold the precache (or all caches ‚Äî simpler + safer for v1)
3. Then `window.location.reload()`

Sketch:

```typescript
async function handleRefresh() {
  // Best-effort: tell the SW to skip waiting + take control of the new SW.
  // If vite-plugin-pwa's registerSW gave us a control function, use it.
  // Otherwise, manually unregister + clear caches.
  try {
    if ("serviceWorker" in navigator) {
      const regs = await navigator.serviceWorker.getRegistrations();
      await Promise.all(regs.map((r) => r.unregister()));
    }
    if ("caches" in window) {
      const keys = await caches.keys();
      await Promise.all(keys.map((k) => caches.delete(k)));
    }
  } catch {
    // best-effort; reload anyway
  }
  window.location.reload();
}
```

PREFERRED: if vite-plugin-pwa's `registerSW` exposes `updateSW(true)` callback (it does), use THAT instead ‚Äî it's the proper integration. Check the project's SW registration to see how `registerSW` is consumed. If `updateSW` is available, calling `updateSW(true)` will:
- Send `skipWaiting` to the new SW
- Reload the page once the new SW activates

That's the canonical PWA update flow + cleaner than manual unregister.

Either approach is acceptable; use whichever fits the existing registration pattern.
