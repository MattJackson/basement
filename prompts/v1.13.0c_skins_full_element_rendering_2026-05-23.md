# v1.13.0c: Skins Full Element Rendering

## Background
v1.13.0a shipped the Skin struct with all fields defined (skin.go). v1.13.0b will add upload handler. This cycle wires FE consumption of typography, density, borderRadius, footer, loginHero.

## Task Requirements

### 1. Implement FE Rendering for All Skin Elements
Update `frontend/src/routes/admin/system.tsx` and related components to render:

**Typography:**
- Apply `skin.typography.sans` as the CSS font-family for body text
- Apply `skin.typography.mono` for monospace surfaces (code blocks, timestamps)
- If `skin.typography.fontUrl` is set, inject `<link rel="stylesheet">` for webfont

**Density:**
- Map `skin.density` to layout tokens:
  - compact → padding: 4px, gap: 8px
  - comfortable → padding: 8px, gap: 12px  
  - spacious → padding: 12px, gap: 16px

**BorderRadius:**
- Apply `skin.borderRadius` to buttons, cards, modals, inputs

**Footer:**
- Show operator footer when non-nil
- Render `footer.text` below main content
- Render up to 5 links from `footer.links` (truncate if more)

**LoginHero:**
- Show on `/admin/login` when non-nil
- Display `loginHero.imageDataUri` as background or side panel
- Display `loginHero.tagline` under the image (optional)

### 2. Per-user Skin Override
When org policy permits (`orgCapabilities.SkinPolicy !== "locked"`):
- Add skin selector to user settings
- Allow users to pick from available skins in registry
- Apply selected skin at session level (overrides org default)

### 3. Create Cycle Report
Write `docs/cycle-reports/v1.13.0c-skins-element-rendering.md` documenting:
- Which elements were wired and how
- Any fallback behavior when skin fields are missing/empty
- Per-user override implementation details
