# v2.0.0-beta.34 — /admin/system save consistency

## What you're producing

ONE commit, ONE tag. Single UX bug fix:

**S2** — `/admin/system` mixes save patterns: some controls require an
explicit "Save changes" button click; others auto-save the moment you
toggle them. Operator wants consistency.

Operator-stated:
> "hate that some you have to click save changes others just save
> on change - consistency"

## Decision: explicit save for everything

Pick the "Save changes" pattern (button-driven) for every control on
this page. Rationale:
- `/admin/system` controls org-wide state (signup mode, allowed drivers,
  skin policy, admin session TTL, OIDC mapping, etc). Auto-saving on
  every toggle increases risk of accidental org-wide changes.
- The page already has at least one "Save changes" button (for signup
  mode + drivers); normalize on that pattern.
- The dirty-state UX is well-known: change → button enables → click
  → toast confirms + button disables.

## Investigation step (do this FIRST)

Open `frontend/src/routes/admin/system.tsx` and audit every form
control. Categorize them as either:
- **Explicit save** (already in a section with a Save button) — leave as is
- **Auto-save** (mutates on change, no button) — needs to be converted

Common auto-save patterns to look for:
- `onChange={(e) => mutation.mutate(...)}` directly
- `onValueChange={(v) => setActiveSkin.mutate(v)}` (radio groups)
- Toggle/switch components that PUT immediately

Build a list of which controls auto-save today.

## Then

For each auto-save control:

1. Replace the immediate mutation with local state update + dirty flag.
   ```tsx
   // before
   onChange={(e) => mutation.mutate(e.target.value)}
   // after
   onChange={(e) => setLocalState({...localState, fieldName: e.target.value})}
   ```
2. Surface the dirty state next to a Save button (either the existing
   page-level Save button or a section-scoped one — match what's
   already on the page).
3. Move the mutation call into the Save button's click handler.

If the page has multiple SECTIONS with their own state slices, keep
section-scoped Save buttons. If there's only one global form, one
page-level Save button is fine.

**Exception: skin selector** lives in the UserMenu dropdown (per-user
preference, not org config). Leave that as auto-save — different
surface, different concern. Don't touch UserMenu.tsx in this cycle.
This prompt is about `/admin/system` ONLY.
