# Cycle: v1.13.6 — Garage v2 ListBuckets fanout for admin-grade size/count/created

The v1.11.0.18 bug-hunt freshman flagged this:

> `GET /admin/clusters/{cid}/buckets` returns `objects: 0, bytes: 0, created: 0001-01-01` for buckets that the detail endpoint `/buckets/{bid}` reports as `objects: 1, bytes: 23, created: 2026-…`. Root cause: Garage v2 `/v2/ListBuckets` doesn't return counts; the driver's `ListBuckets` (`internal/drivers/garage/buckets.go:14`) populates only ID + Aliases on the Bucket struct, leaving the rest at Go zero values that marshal as misleading data.

Operator decision (now confirmed): **fanout** — per [[feedback_admin_grade_screens]] lists need sizes + counts + dates, not zero placeholders. Acceptable perf cost: typical cluster has 5-200 buckets; concurrent fanout via worker pool keeps total latency bounded to a single GetBucketInfo round-trip.

## Fix

In `internal/drivers/garage/buckets.go` `ListBuckets`:

1. After fetching the bucket-name list from `/v2/ListBuckets`, fanout `GetBucketInfo` calls in parallel via a bounded worker pool (e.g. 8 concurrent goroutines using `errgroup` or similar)
2. Each worker fetches the bucket's full info + populates Objects, Bytes, CreatedAt on the corresponding Bucket struct
3. If a per-bucket GetBucketInfo fails: log + leave that bucket's count/size as zero + continue (don't fail the whole list)
4. If the cluster is huge (>500 buckets), document a follow-up to add pagination
