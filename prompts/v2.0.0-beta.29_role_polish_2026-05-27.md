## T1 — Menu doesn't close when elevation modal appears

### Root cause (do NOT just revert to controlled state — that has a
deeper bug)

beta.27 used controlled `useState(menuOpen)` and passed `open={menuOpen}`.
THAT cycle had a bug: the DropdownMenu JSX is inside `useMemo` whose
deps list omits `menuOpen`. React cached the JSX with `open=false` and
clicks never propagated. beta.28 fixed THAT by going uncontrolled —
but then lost the ability to close the menu before opening the
elevation modal (T1).

The proper fix needs BOTH:
- Controlled open state so `handleRoleChange` can close it
- useMemo deps that include `menuOpen` OR the DropdownMenu wrapper
  moved OUT of the useMemo entirely.

**Pick option B** (move out of useMemo). It's structurally cleaner.
The useMemo currently builds the inner pill content (visual styling)
AND wraps it in DropdownMenu. Split them:
- `useMemo` returns just the inner `<span data-testid="persona-pill">`
- The DropdownMenu wrapper is rendered in the regular component body,
  conditional on `availableRoles.length > 1`, where `menuOpen` is in
  scope and changes trigger re-renders normally.

### Concrete edit

In PersonaPill.tsx around lines 366-394 (the `if (availableRoles.length > 1)`
branch currently inside the useMemo): MOVE that wrapper out of the
useMemo. The useMemo should always `return pillContent` (the inner
span). The DropdownMenu wrapping happens in the JSX returned by the
component's main `return (...)` (around line 397+).

Re-introduce `const [menuOpen, setMenuOpen] = useState(false);`.
Pass `open={menuOpen} onOpenChange={setMenuOpen}` to DropdownMenu.
In `handleRoleChange`, BEFORE calling `promptElevationFromAnywhere`,
call `setMenuOpen(false)` so the menu closes ahead of the modal.

### Then dedupe

Based on what you find, expected sources of multiple toasts:
1. `handleRoleChange` calls `toast.error(...)` on cancel/error path
2. `useSwitchActiveRole` may have implicit error handling
3. AuthModeHydrator may fire on the elevation success path
4. Some other queryClient effect

Plan: For a SUCCESSFUL role switch, NO toasts should fire (the
visible page navigation is the signal). For elevation-cancel, ONE
toast: "Stayed on previous role" (or similar). For non-423 error,
ONE toast: the error message.

Audit + remove duplicates. Common pattern: if both handleRoleChange's
catch AND the mutation's onError fire, you get 2 toasts. Pick one
canonical source.

(or exactly 1 if you keep an info toast for the success case).
