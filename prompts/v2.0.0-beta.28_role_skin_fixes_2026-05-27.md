## R1 ŌĆö Role switcher click is a no-op

### What's wrong

`PersonaPill` renders a DropdownMenu around the pill when
`availableRoles.length > 1`. The trigger renders correctly (visible
in DOM as `data-testid="persona-role-trigger"`) but clicking it
doesn't open the menu.

Verified via Playwright against live beta.27: trigger element exists
with `aria-haspopup=menu`, click fires, no menu appears.

### Root cause

The DropdownMenu JSX is built inside `useMemo(() => {...}, deps)`
where `deps = [mode, flashing, inAmberWindow, inRedWindow, activeRole]`
(see line ~395). The freshman for beta.23 added `const [menuOpen, setMenuOpen] = useState(false)`
and wired `<DropdownMenu open={menuOpen} onOpenChange={setMenuOpen}>`,
but `menuOpen` is NOT in the useMemo deps array.

So React caches the JSX with the initial `open={false}` value. Clicks
call `setMenuOpen(true)`, the component re-renders, but `useMemo`
returns the SAME cached JSX (deps haven't changed). The dropdown
keeps rendering as closed.

### Fix

Drop the controlled-open prop entirely. Let Base UI's DropdownMenu
manage its own open state internally (uncontrolled). The only reason
controlled state was added in beta.23 was so `handleRoleChange` could
close the menu before opening the elevation modal on a 423 ŌĆö that's
a rare path and Base UI's auto-close-on-item-select handles the
common one fine.

Concretely in `PersonaPill.tsx`:

1. Remove the `useState(false)` for `menuOpen` + `setMenuOpen` (around line 160).
2. Change `<DropdownMenu open={menuOpen} onOpenChange={setMenuOpen}>` to just `<DropdownMenu>` (around line 368).
3. Remove the `setMenuOpen(false)` call inside `handleRoleChange`'s
   423 branch (around line 136).

Don't add `menuOpen` to the useMemo deps ŌĆö that would technically work
but the uncontrolled approach is simpler + the controlled state has
no business being inside a memoized branch anyway.

## R2 ŌĆö Skin selection doesn't actually change anything

### What's wrong

When a user picks a skin via UserMenu ŌåÆ Skin submenu, the UI doesn't
visually change. Frontend mutation reports success, but a refetch of
`/api/v1/skins/active` returns the SAME skin as before.

### Root cause (TWO backend bugs)

**R2a** ŌĆö `internal/api/skin_upload.go:435-497` (`setUserSkinHandler`):
the handler validates the skin name and returns `{success:true, skinName:...}`
but **literally does not persist anything**. The comment at line 490
reads: `// Persist user's skin choice (stored in a per-user location; for now just echo back)`.
The endpoint is a stub.

**R2b** ŌĆö `internal/api/skin_upload.go:383-426` (`getActiveSkinHandler`):
this endpoint returns the ORG-wide active skin (`caps.ActiveSkin`),
not the calling user's preference. Even if R2a stored a per-user skin,
this getter wouldn't surface it. Also: the response body OMITS the
`palette` field, so the frontend's CSS-variable injection at
`frontend/src/shared/hooks/useSkin.ts:78-104` no-ops (early-returns
because `skin?.palette` is undefined).

### Fix

You'll need to:

**(a)** Add a per-user skin store. Look at how other per-user state is
persisted in this codebase ŌĆö likely `internal/store/`. Look for
patterns like `internal/store/user_regions.go` (per-user data). Mirror
that pattern.

The store needs:
- `Get(userID string) (skinName string, ok bool)` ŌĆö read user's choice
- `Set(userID string, skinName string) error` ŌĆö persist user's choice
- Backed by JSON file (consistent with other stores) at e.g.
  `data/user_skins.json`

**(b)** Wire `setUserSkinHandler` to actually call `s.userSkinStore.Set(claims.UserID, req.SkinName)`
after validation. Pull `claims` from the auth context (look at how
`listSyncsHandler` at `internal/api/user_syncs.go:185-190` reads
claims ŌĆö same pattern).

**(c)** Update `getActiveSkinHandler` to:
1. Read auth claims (don't require them ŌĆö anonymous fallback to org)
2. If authed and user has a skin set, use that name
3. Otherwise fall back to `caps.ActiveSkin` then `store.DefaultActiveSkin`
4. **ALWAYS include `palette` field** in the response. Look at the
   `skin.Skin` struct in `internal/skin/builtin.go` ŌĆö it has
   `Palette: PaletteSet{Light: Palette{...}, Dark: Palette{...}}`.
   The response currently has 8 fields; add a 9th: `"palette": skinObj.Palette`.

**(d)** Wire the new store into `cmd/basement-server/main.go` so it's
constructed at startup. Mirror how other stores are wired.

### Tests

Add Go tests in `internal/api/skin_upload_test.go` (extend if exists):
1. `TestSetUserSkin_PersistsAndGets` ŌĆö call setUserSkin for user A,
   then call getActiveSkin as user A, assert the response matches
   the set value (not the org default).
2. `TestGetActiveSkin_IncludesPalette` ŌĆö assert response has a
   `palette` field with at least `light` and `dark` sub-objects.
