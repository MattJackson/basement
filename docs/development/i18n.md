# Internationalization (i18n) Development Guide

## Overview

basement uses `react-i18next` for client-side internationalization. This guide covers the architecture, translation file structure, and workflow for adding new languages.

## Architecture

- **Library**: [`react-i18next`](https://react.i18next.com/) — most maintained, smallest bundle, full TypeScript support
- **Detector**: `i18next-browser-languagedetector` — auto-detects user language from localStorage or browser settings
- **Storage**: Language preference persists in `localStorage.basement_language`

## Translation File Structure

### Directory Layout

```
frontend/src/shared/i18n/
├── index.ts                    # i18next configuration
└── locales/
    └── <lang>/                 # One directory per language (en, es, etc.)
        ├── common.json         # Shared strings: buttons, navigation, auth
        └── pages.json          # Page-specific strings
```

### Namespaces

1. **`common`** — App-wide strings used across multiple pages:
   - Navigation links (`navigation.*`)
   - Buttons and actions (`buttons.*`)
   - Role labels (`roles.*`)
   - Authentication text (`auth.*`)
   - States (`states.*`)
   - Confirmations (`confirmations.*`)

2. **`pages`** — Route-specific strings:
   - Page titles and subtitles
   - Empty state copy
   - Action buttons unique to that page

### Adding a New Language

1. **Update the languages array** in `frontend/src/shared/i18n/index.ts`:
   ```typescript
   export const SUPPORTED_LANGUAGES = ["en", "es", "fr"] as const;
   ```

2. **Create translation files**:
   ```bash
   mkdir -p frontend/src/shared/i18n/locales/<lang>
   cp frontend/src/shared/i18n/locales/en/common.json frontend/src/shared/i18n/locales/<lang>/common.json
   cp frontend/src/shared/i18n/locales/en/pages.json frontend/src/shared/i18n/locales/<lang>/pages.json
   ```

3. **Translate the JSON files**:
   - Keep all keys identical to English
   - Translate only the values
   - For Spanish: use formal "usted" verb forms for buttons and actions
   - Preserve product name "basement" untranslated in all languages

4. **Add language picker option** if needed (see Language Switcher section)

## Using Translations in Components

### Basic Usage

```tsx
import { useTranslation } from "react-i18next";

export function MyComponent() {
  const { t } = useTranslation("common");
  
  return <button>{t("buttons.save")}</button>;
}
```

### Page-Specific Translations

```tsx
const { t } = useTranslation("pages");

<h1>{t("filesHome.pageTitle")}</h1>
<p>{t("filesHome.emptyStateDescription")}</p>
```

### Multiple Namespaces

```tsx
const { t: tc } = useTranslation("common");
const { t: tp } = useTranslation("pages");

<button onClick={handleSave}>{tc("buttons.save")}</button>
<h1>{tp("filesHome.pageTitle")}</h1>
```

## Translation Workflow

### 1. Identify Strings to Extract

- **Common UI strings**: buttons, labels, navigation items → `common.json`
- **Page-specific content**: titles, descriptions, empty states → `<page>.json` or `pages.json`

### 2. Create English Master File

Before translating:
1. Wrap all hardcoded strings with `t()` using the English key path
2. Ensure the English master file (`en/common.json`, `en/pages.json`) is complete for that component/page
3. Test that all keys resolve correctly in English

### 3. Add Spanish Translation

For v2.0 beta cycle, prioritize Spanish:
1. Copy English JSON files to Spanish locale directory
2. Translate values using credible translations (formal "usted" form)
3. Keep product name "basement" unchanged

### 4. Fallback Rules

- If a translation key is missing in the target language, i18next falls back to `en`
- The fallback chain: `{requestedLang} → en`
- Missing keys display as their key path (e.g., `"buttons.save"`) — use this during development to find untranslatable strings

### 5. Beta Cycle Timeline

- **beta.3**: Scaffolding + en/es anchor languages + representative extraction (UserMenu, /files page)
- **beta.4**: Full app string extraction — every UI string moves to `t()` / English master JSON complete
- **beta.5**: Mass language pour for all LTR locales (~20-30 languages)

## Language Switcher

The language switcher is implemented as a submenu in the UserMenu component:

```tsx
<DropdownMenuSub>
  <DropdownMenuSubTrigger data-testid="language-submenu-trigger">
    Language
  </DropdownMenuSubTrigger>
  <DropdownMenuSubContent>
    <DropdownMenuRadioGroup value={i18n.language} onValueChange={(lang) => i18n.changeLanguage(lang)}>
      {i18next.options.supportedLanguages?.map((lang) => (
        <DropdownMenuRadioItem key={lang} value={lang}>
          {lang === "en" ? "English" : lang === "es" ? "Español" : lang}
        </DropdownMenuRadioItem>
      ))}
    </DropdownMenuRadioGroup>
  </DropdownMenuSubContent>
</DropdownMenuSub>
```

### Display Names

Language display names are hardcoded in the switcher. For new languages, add a mapping:

```tsx
const languageNames: Record<string, string> = {
  en: "English",
  es: "Español",
  fr: "Français",
  de: "Deutsch",
};

<DropdownMenuRadioItem value={lang}>
  {languageNames[lang] || lang}
</DropdownMenuRadioItem>
```

## Testing

### Unit Tests for i18n Config

Test that translations load correctly and fallback works:

```tsx
import { describe, it, expect } from "vitest";
import i18next from "i18next";

describe("i18n configuration", () => {
  it("loads English and Spanish translations successfully", async () => {
    const i18n = await import("@/shared/i18n");
    
    expect(i18n.SUPPORTED_LANGUAGES).toEqual(["en", "es"]);
  });

  it("fallbacks to English when key is missing", async () => {
    await i18next.changeLanguage("es");
    expect(i18next.t("common:buttons.save")).toBe("Guardar");
  });
});
```

### Render Tests for Language Switcher

Test that the language switcher changes language and re-renders:

```tsx
import { render, screen, fireEvent } from "@testing-library/react";
import { I18nextProvider } from "react-i18next";
import i18n from "@/shared/i18n";

it("changes language when selecting Spanish", async () => {
  render(<UserMenu />, { wrapper: ({ children }) => <I18nextProvider i18n={i18n}>{children}</I18nextProvider> });
  
  fireEvent.click(screen.getByLabelText("Open admin menu"));
  const languageTrigger = await screen.findByTestId("language-submenu-trigger");
  fireEvent.click(languageTrigger);
  
  const espanolOption = await screen.findByText("Español");
  fireEvent.click(espanolOption);
  
  expect(i18n.language).toBe("es");
});
```

### Snapshot Tests

Snapshot test components in both languages to verify visual consistency:

```tsx
it("renders UserMenu in Spanish", () => {
  const { container } = render(<UserMenu />, { wrapper: ({ children }) => <I18nextProvider i18n={i18n}>{children}</I18nextProvider> });
  
  // Switch to Spanish first
  i18n.changeLanguage("es");
  
  expect(container).toMatchSnapshot();
});
```

## Best Practices

1. **Never hardcode UI strings** — always use `t()` with a namespaced key
2. **Use descriptive key paths** — `buttons.save` not just `save`
3. **Keep translation files organized** by namespace (common vs pages)
4. **Test in both languages** during development to catch layout issues
5. **Document new languages** in this guide when adding them

## Adding a New Language (Step-by-Step)

1. Add language code to `SUPPORTED_LANGUAGES` array in `index.ts`
2. Create locale directories: `locales/<lang>/common.json` and `pages.json`
3. Copy English files as template, then translate values only
4. Update any hardcoded language display names if needed
5. Write unit tests for the new language
6. Add snapshot tests for components in the new language

## Notes for Spanish Translations

- Use formal "usted" verb forms (e.g., "Guardar" not "Guarda")
- Product name "basement" stays untranslated
- Maintain technical terminology consistency with existing translations
- Keep ellipsis (…) and special characters intact (e.g., "Cargando…")
