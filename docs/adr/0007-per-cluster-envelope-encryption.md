# ADR-0007: Per-cluster envelope encryption (Cluster Secret Key)

- **Status**: Accepted
- **Date**: 2026-05-23
- **Decision-maker**: Operator
- **Builds on**: [ADR-0001](./0001-rbac-three-tier-creds.md) (three-tier creds), v1.0.0a connections at-rest encryption (JWT-derived AES-GCM)
- **Triggered by**: cluster-tier secrets (Garage admin tokens, future
  per-cluster S3 secretAccessKey persistence) currently share the
  JWT signing secret as their AES-GCM key. JWT secret is a hot env-var,
  rotates infrequently, and is identical for every cluster — a
  compromised JWT secret unlocks every backend basement ever held a
  credential for. The blast radius and the operational-rotation story
  are both wrong for the threat model.

## Context

basement v1.0.0a introduced at-rest encryption of sensitive
Connection.Config fields (admin_token, secret_key, s3_secret_key,
auth_token). The wrapping key is `sha256(cfg.JWT.Secret)` — see
`internal/store/crypto.go`. This is one key for the whole
deployment, derived from a value that:

- is shared with every JWT-signed cookie and every confirm-token,
- has no per-cluster scope,
- has no defined rotation procedure (rotating JWT_SECRET invalidates
  every cookie and every connections.json record at once).

v1.11.x is also poised to start persisting Garage `CreateKey`'s
`secretAccessKey` so the operator can recover the secret after the
shown-once dialog (today the only copy lives in the operator's
clipboard). Pushing more secrets under the JWT-derived key compounds
the blast radius and the rotation hazard.

We want:

1. **Per-cluster scope** — compromising one cluster's wrapping key
   doesn't touch another cluster's secrets.
2. **Operator-controlled unlock** — the secret-of-secrets is a
   password the cluster admin holds, not an env var the process can
   read at any moment.
3. **In-memory after unlock** — background jobs (federation, backup
   runner, metrics scheduler) need the secrets while the operator is
   away from the keyboard.
4. **Multi-admin** — N cluster admins, each with their own password,
   each able to unlock and re-lock independently.
5. **Recovery** — losing a password loses access from that admin's
   POV, but another admin still holding theirs can re-wrap with a
   new password.
6. **Safe migration** — the existing JWT-encrypted `admin_token`
   must keep working until the first successful CSK unlock.

## Decision

Adopt envelope encryption with a per-cluster **Cluster Secret Key
(CSK)**:

```
+--------------------+     Argon2id        +-----------+   AES-GCM
| admin password     | ------------------> | wrapping  | --------> wrappedCSK (per admin)
| (user-supplied)    |   (per-admin salt)  |   key     |
+--------------------+                     +-----------+
                                                |
                                                v
                            +---------------------------------+
                            | CSK (32 random bytes)           | AES-GCM
                            |   plaintext lives ONLY in       | --------> per-secret ciphertext
                            |   ClusterSecretManager.csks     |              on disk
                            |   in process memory after       |
                            |   unlock; never on disk         |
                            +---------------------------------+
```

### Mechanism

- **CSK** is 32 random bytes generated by `crypto/rand` the first
  time an admin sets a password for the cluster (or migrates from
  the legacy JWT-encrypted path).
- **Wrapping key** is `Argon2id(password, salt, time=3, memory=64MiB,
  threads=4) → 32 bytes`. Per OWASP 2026 recommendations for
  password hashing / KDF; memory cost dominates the GPU attacker.
- **WrappedCSK** is `AES-256-GCM(CSK, wrappingKey)` with a fresh
  12-byte nonce per write. Stored per `(clusterID, adminUserID)` so
  N admins can each have their own wrapped copy.
- **Plaintext CSK** is held in `ClusterSecretManager.csks[clusterID]
  []byte`. Never written to disk, never logged, never serialised
  into audit metadata.
- **Per-secret encryption** uses `AES-256-GCM(plaintext, CSK)`. Same
  wire format as `internal/store/crypto.go` — `nonce(12) || ct(N) ||
  tag(16)` — so callers see a familiar shape.

### Scope: what CSK protects vs. what stays where it is

| Secret | Current key | After ADR-0007 |
|---|---|---|
| Cluster `admin_token` | JWT-derived (v1.0.0a) | **CSK** — migrated on first unlock |
| Garage `CreateKey` `secretAccessKey` (newly persisted) | n/a (not persisted today) | **CSK** at create time |
| Per-user UserRegion S3 secrets | JWT-derived | unchanged — per-user scope; different trust model (the user logs in, the password they prove their identity with isn't the cluster admin's) |
| Service-account secret (bcrypt) | bcrypt one-way hash | unchanged — verification-only, never decrypt |
| JWT signing secret itself | env var | unchanged — JWT keeps signing |
| Webhook HMAC signing secret | plaintext on disk | unchanged for v1.12.0a; revisit if elevated to CSK-tier |

The shared rule: any secret whose plaintext basement must hand to a
backend at request time AND whose blast radius is "this cluster" is
on CSK. Per-user, per-grant, or signature-only secrets stay on their
existing path.

### Lifecycle

1. **First-admin bootstrap**: cluster admin clicks "Enable cluster
   encryption" → modal asks for a password → server generates a
   random CSK, wraps it under Argon2id(password, salt), stores
   `wrappedCSK + salt + kdfParams` per `(clusterID, adminUserID)`.
   CSK is cached in memory immediately so the admin can use it for
   the rest of the session.
2. **Subsequent unlocks**: admin enters password → server looks up
   their wrappedCSK record, derives wrapping key, AES-GCM decrypts
   to CSK, caches in memory. Wrong password fails AEAD auth → 401
   `INVALID_PASSWORD`.
3. **Add another admin**: an already-unlocked admin enters the new
   admin's user ID + the new admin's chosen password → server takes
   the in-memory CSK, re-wraps under the new admin's
   Argon2id(password, fresh salt), writes a new wrappedCSK record
   for the new admin user ID.
4. **Remove an admin**: the wrappedCSK record for that admin is
   deleted. The CSK itself is unchanged — other admins still have
   their wrapped copies. The removed admin can no longer unlock.
5. **Lock**: admin clicks "Lock cluster" → server zeroes
   `csks[clusterID]` and removes it from the map. Subsequent
   Encrypt / Decrypt calls return `ErrLocked` → API returns `423
   LOCKED`.
6. **Restart**: in-memory map is empty by definition. Any admin must
   unlock before background jobs touch the cluster's secrets.

### API surface

```
POST   /api/v1/admin/clusters/{cid}/unlock      {password}  → 200 {unlocked:true} or 401
POST   /api/v1/admin/clusters/{cid}/lock                    → 204
GET    /api/v1/admin/clusters/{cid}/lock-status             → {unlocked, requiresMigration, admins:[uid…]}
POST   /api/v1/admin/clusters/{cid}/admins      {adminUserId,password}  → 201 (caller must be unlocked)
DELETE /api/v1/admin/clusters/{cid}/admins/{adminUserId}    → 204
```

Any other endpoint that needs CSK to decrypt a stored secret returns
`423 LOCKED` with `{cluster_id, hint:"unlock required"}` instead of a
500 / 401. The FE intercepts `423` and presents the unlock modal,
then re-runs the request on success.

### Migration (legacy admin_token → CSK)

A Connection record may have its `admin_token` already encrypted
under the v1.0.0a JWT-derived key (Connection.ConfigEnc). The
migration is **lazy + on-first-unlock**:

1. Admin sets a CSK password for the cluster for the first time.
2. ClusterSecretManager.MigrateFromJWT (called by the unlock /
   set-first-password handler) is given the legacy key (JWT secret).
3. Manager decrypts the existing Config sensitive subset under the
   JWT key, re-encrypts under CSK, writes back. The Connection
   record's `requiresMigration` flag flips to false.
4. The legacy ciphertext is dropped from the on-disk record once
   the rewrite is fsynced — **not before**. If the rewrite errors,
   the legacy ciphertext stays on disk and the migration is retried
   on the next unlock. The bridge is never burned before the new
   path is verified.

### Multi-admin example

```
Cluster cid=abc has admins:
  user "matthew":  wrappedCSK_matt   = AES-GCM(CSK, Argon2id("hunter2", saltM))
  user "wife":     wrappedCSK_wife   = AES-GCM(CSK, Argon2id("eggcream", saltW))

Either admin can unlock with their own password.
Both admins decrypt to the SAME CSK.
Either admin can encrypt/decrypt the cluster's admin_token, Garage
secretAccessKey, etc.
```

### Background jobs (federation engine, backup runner)

When a tick fires for a federation whose primary or replica needs CSK
to read its credentials:

- `clustersecret.IsUnlocked(clusterID)` is the gate.
- Locked → log at `Info` level once per minute, set the engine's
  per-federation status to `"paused-locked"`, skip the tick.
- Auto-resume: on the very next unlock, the next scheduled tick
  proceeds normally — no manual re-registration needed.

This means an operator who restarts the process and walks away from
the keyboard for a week has all federations paused-but-recoverable
until they unlock; nothing breaks, nothing replicates with stale
credentials, nothing requires intervention beyond "type your
password."

### Recovery posture

- **Single admin loses password**: that admin's wrappedCSK is now
  un-openable. If they're the only admin, the cluster is bricked
  (no one can decrypt the secrets). Operator must DELETE the
  cluster's admin record + Connection's CSK material, re-create the
  connection from scratch with fresh credentials, set a new CSK
  password.
- **One of N admins loses password**: another admin unlocks
  normally, then runs `POST /admins` for the locked-out admin with
  a new password. Removed-then-added.
- **JWT secret rotation**: independent — JWT only matters during the
  migration window. Once CSK has taken over, rotating JWT doesn't
  affect cluster secrets at all.

### Audit events

Every state change emits to the existing append-only audit log:

| Action | When |
|---|---|
| `cluster:csk_unlocked` | successful unlock |
| `cluster:csk_locked` | explicit lock or process restart-detected |
| `cluster:csk_admin_added` | new admin wrappedCSK written |
| `cluster:csk_admin_removed` | admin wrappedCSK deleted |
| `cluster:csk_migrated` | legacy JWT-encrypted secret re-encrypted under CSK |

The actual password value is NEVER in any audit field. The CSK
plaintext is NEVER in any audit field.

## Out of scope (deferred)

- Per-cluster CSK rotation (re-generating a fresh CSK and
  re-encrypting every stored secret). The wrappedCSK is N-admin
  rotatable already; the CSK itself stays the same for the cluster's
  lifetime. Full CSK rotation is a v1.13+ cycle.
- Hardware-backed wrapping (HSM / YubiKey). Same Argon2id flow
  works with a KMS-backed wrapping key; v1.12.0a ships the
  password path only.
- UserRegion secrets migration to CSK. Different trust model —
  user logs in with their basement password (no separate cluster
  admin password to type), so the JWT-derived path is the
  appropriate one for v1.12.0a.
