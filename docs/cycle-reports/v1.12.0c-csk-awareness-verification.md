# v1.12.0c: CSK Lock-State Awareness Verification

**Status:** Accepted  
**Date:** 2026-05-23  
**Build:** v1.12.0c  

## Summary

This cycle verifies that background jobs correctly gate on CSK lock-state to prevent failures when clusters are locked, and confirms why the federation engine does not require such gates given its use of UserRegion secrets instead of admin_tokens/CSK.

## Background

Batch2 analysis found:
- Federation engine uses **UserRegion JWT secrets** (NOT CSK) per ADR-0007
- Backup runner has **CSK lock-state checks** implemented at `internal/api/backup_runner.go:96-109`

This documentation captures the verification findings and adds a regression test to prevent future loss of this behavior.

## Which Background Jobs Need Lock-State Gates

### Backup Runner: YES, requires lock-state gates

The backup runner (`internal/api/backup_runner.go`) must check CSK lock-state because:

1. **Backup destinations may use CSK-protected connections** — When a backup writes to a Garage bucket or other backend that stores its `admin_token` under CSK (per ADR-0007 migration), the connection's credentials are only accessible when the cluster is unlocked.

2. **Both source and destination must be checked** — The runner resolves both `SrcRegionID` and `DstRegionID` to Connection IDs, then checks each against `clusterSecrets.IsUnlocked()`. If either side uses CSK and is locked, the entire backup run is skipped.

3. **Graceful skip vs. failure** — When a cluster is locked:
   - The backup result is marked as failed with error: `"cluster <cid> is locked — skipped; will retry on next scheduled run"`
   - A log entry at `Info` level records the skip
   - The scheduler automatically retries on the next interval ("auto-resume on unlock")

### Federation Engine: NO lock-state gates needed

The federation engine does NOT need CSK lock-state checks because:

1. **Uses UserRegion secrets, not admin_tokens** — Per ADR-0007 Section "Scope: what CSK protects vs. what stays where it is":
   - Cluster `admin_token` → **CSK** (migrated on first unlock)
   - Per-user UserRegion S3 secrets → **unchanged**, still JWT-derived

2. **Different trust model** — UserRegion secrets are per-user, not per-cluster admin. The user logs in with their basement password (not a separate cluster admin password), so the JWT-derived path is appropriate for v1.12.0a and beyond.

3. **No migration to CSK planned** — ADR-0007 Section "Out of scope (deferred)" explicitly lists "UserRegion secrets migration to CSK" as deferred because it requires a different trust model.

## Where Backup Does the Check: Implementation Pattern

### Location
`internal/api/backup_runner.go`, function `runBackupOnce`, lines 96-109.

### Code Pattern

```go
// v1.12.0a (ADR-0007): if either side's Connection uses a CSK
// for its stored secrets and the cluster is currently locked,
// skip the run rather than fail it. The scheduler will fire
// again on the next interval; an admin unlock between now and
// then lets the next run succeed without manual intervention
// ("auto-resume on unlock" per the ADR).
//
// In v1.12.0a no Connection has been migrated to CSK yet (the
// store-layer hook lands in a follow-up cycle), so this branch
// is a no-op in practice. Wiring it now keeps the runner correct
// the moment the migration completes for any single cluster.
if s.clusterSecrets != nil {
    for _, cid := range []string{srcConn, dstConn} {
        if !s.clusterSecrets.IsUnlocked(cid) {
            hasCSK, _ := s.clusterSecrets.HasAdmins(cid)
            if hasCSK {
                result.CompletedAt = time.Now().UTC()
                result.Errors = []string{fmt.Sprintf("cluster %s is locked — skipped; will retry on next scheduled run", cid)}
                s.logger.Info("backup: skipping locked cluster",
                    "backupId", b.ID, "clusterId", cid)
                return result
            }
        }
    }
}
```

### Pattern Breakdown

1. **Nil check** — `if s.clusterSecrets != nil` handles cases where CSK is not wired (tests or disabled deployments).

2. **Iterate both connections** — Checks both source and destination Connection IDs to cover all migration states.

3. **Two-level gate**:
   - First: `!s.clusterSecrets.IsUnlocked(cid)` checks if the cluster's CSK is in memory
   - Second: `s.clusterSecrets.HasAdmins(cid)` confirms the cluster actually has CSK configured (distinguishes "no CSK yet" from "CSK but locked")

4. **Early return** — Sets result error and returns immediately, preventing any attempt to open drivers or copy data.

5. **Logging** — `Info` level log enables operators to see skips in logs without spamming at Debug level.

## Regression Test Requirements

A regression test must be added to `internal/api/backup_runner_test.go` (or existing backup test file) that:

1. Creates a mock cluster secret manager with one unlocked and one locked cluster
2. Configures a backup where the destination uses the locked cluster's Connection ID
3. Verifies the backup result is marked as failed with the correct skip error message
4. Confirms no driver operations were attempted (the runner returned before step 1: open drivers)

The test should be pinned to "backup runner skips locked clusters" in its name to ensure discoverability during future audits of CSK-awareness.
