# v1.12.0a: CSK Infrastructure (ClusterSecretManager + Admin Endpoints)

**Status**: Accepted  
**Date**: 2026-05-23  
**Build**: v1.12.0a  

## Summary

This cycle implements the core CSK infrastructure per ADR-0007: `ClusterSecretManager` with unlock/lock/admins CRUD endpoints, and migration from legacy JWT-derived encryption.

## Implementation

### ClusterSecretManager API (`internal/clustersecret/clustersecret.go`)

The manager provides:
- `Unlock(clusterID, password)` — decode CSK into memory using one admin's password (line 184)
- `Lock(clusterID)` — zero CSK from memory (line 266)
- `IsUnlocked(clusterID) bool` — cheap predicate for request path (line 293)
- `Encrypt/Decrypt(cid, …)` — AES-GCM under in-memory CSK (lines 340, 354)
- `AddAdmin/RemoveAdmin` — manage wrappedCSK records (lines 371, 475)
- `MigrateFromJWT(clusterID, legacyCiphertext, jwtSecret)` — re-encrypt under CSK (line 499)

### Admin Endpoints (`internal/api/admin_cluster.go`)

New endpoints per ADR-0007 Section "API surface":
```
POST   /api/v1/admin/clusters/{cid}/unlock      {password}  → 200 {unlocked:true} or 401
POST   /api/v1/admin/clusters/{cid}/lock                    → 204
GET    /api/v1/admin/clusters/{cid}/lock-status             → {unlocked, requiresMigration, admins:[uid…]}
POST   /api/v1/admin/clusters/{cid}/admins      {adminUserId,password}  → 201
DELETE /api/v1/admin/clusters/{cid}/admins/{adminUserId}    → 204
```

### Migration Path (`internal/store/connections.go`)

On first unlock after CSK enabled, `MigrateFromJWT` re-encrypts the legacy JWT-encrypted `admin_token`:
1. Decrypt under `sha256(jwtSecret)` (legacy path)
2. Re-encrypt under in-memory CSK
3. Write back to `connections.json`, drop legacy ciphertext

See `docs/adr/0007-per-cluster-envelope-encryption.md:149-165` for migration details.
