### Density (`skin.density`)

**Source field:**
- `Density` enum with literals `"compact"`, `"comfortable"`, `"spacious"` (lines 38-56)

**Implementation:**

1. **CSS Token Layer** (`frontend/src/index.css` lines 102-119):
   ```css
   --density-compact-padding: 4px;
   --density-compact-gap: 8px;
   --density-comfortable-padding: 8px;
   --density-comfortable-gap: 12px;
   --density-spacious-padding: 12px;
   --density-spacious-gap: 16px;
   ```

2. **Runtime Mapping** (`frontend/src/shared/hooks/useSkin.ts` lines 43-54):
   - Returns `densityTokens` object with `padding` and `gap` values based on current skin's density
   - Compact → `padding: "4px", gap: "8px"`
   - Comfortable (default) → `padding: "8px", gap: "12px"`
   - Spacious → `padding: "12px", gap: "16px"`

**Usage pattern:**
Components should consume density tokens like:
```tsx
<div style={{ padding: densityTokens.padding, gap: densityTokens.gap }}>
  {/* content */}
</div>
```

**Fallback behavior:**
- Missing/undefined `density` → defaults to `"comfortable"` (useSkin.ts line 20)

## Per-User Skin Override Implementation

### Org Policy Gate

**Policy field:** `OrgCapabilities.SkinPolicy` (store/org_capabilities.go lines 60-73)

**Valid values:**
- `"locked"` — Operator-selected skin locked for all users; no per-user override
- `"default"` — Standard behavior, no user override (same as locked)
- `"user-choice"` — Users can select their own skin from registry

**Check function:** `frontend/src/shared/api/queries.ts:780-782`
```typescript
export function canUserSelectSkin(skinPolicy?: string): boolean {
  return skinPolicy === "user-choice";
}
```

### User Menu Integration

**Location:** `/Users/mjackson/Developer/basement-ui/frontend/src/shared/ui/UserMenu.tsx` (lines 198-235)

**Implementation details:**
1. **Conditional rendering:** Skin submenu only appears when `canUserSelectSkin(orgCaps.data?.skinPolicy)` returns true
2. **Registry fetch:** `useSkinRegistry()` hook pulls available skins from `/api/v1/skins` endpoint
3. **Active skin tracking:** `useSkin()` provides current active skin name for radio selection checkmark
4. **Mutation:** `useSetActiveSkin()` POSTs to `/api/v1/user/skin` with `{skin: selectedName}` body
5. **Cache invalidation:** On success, invalidates `["skin", "active"]` and `["org-capabilities"]` query keys

**UI structure:**
```tsx
<DropdownMenuSub>
  <DropdownMenuSubTrigger data-testid="skin-submenu-trigger">Skin</DropdownMenuSubTrigger>
  <DropdownMenuSubContent>
    <DropdownMenuRadioGroup onValueChange={setSelectedSkin}>
      {skinRegistry.data.map(s => (
        <DropdownMenuRadioItem key={s.name} value={s.name}>
          {s.displayName || s.name}
        </DropdownMenuRadioItem>
      ))}
    </DropdownMenuRadioGroup>
  </DropdownMenuSubContent>
</DropdownMenuSub>
```

**Session-level override:** Per-user skin selection overrides org default at session level. When a user selects a skin:
- Backend persists the selection to user session store
- Skin remains active until user changes it or logs out
- Org admin can still change active skin globally (affects new sessions only)
