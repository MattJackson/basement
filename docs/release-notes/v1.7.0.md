# Basement v1.7.0 ŌĆö Release Notes

**Status**: Tagged

> **Next release**: [v1.8.0](v1.8.0.md) ŌĆö MCP server + mobile PWA
> + rebrand to AGPLv3.

## Headline

**Service accounts + webhooks ŌĆö M2M auth substrate + event-driven
workloads + federation upgrade.**

v1.6 gave basement its federation control plane: a `FederatedBucket`
is the same logical bucket living on multiple backends, kept in
lock-step by a polling-based replication engine. v1.7 is the upstream
substrate for everything beyond v1.x. **Service accounts** ship the
M2M auth primitive that v1.8's CLI binary + v1.8's MCP server + every
future automation will mint a credential against ŌĆö long-lived bearer
tokens scoped per-capability, attributed to the SA owner with a
distinguishable actor in audit. **Webhooks** ship the event-driven
substrate that turns "polling for changes" into "react to changes" ŌĆö
HMAC-signed HTTP callbacks on bucket events with retry policy +
auto-disable + admin verification snippet. The same primitive then
gets pointed inward: an **internal pub/sub** lets the v1.6 federation
engine subscribe to bucket events directly, so a write on the primary
propagates to replicas within seconds instead of waiting up to the 10s
polling tick. Polling stays as the fallback for backends without
webhook support ŌĆö both paths share the same convergence semantics.

- `/admin/service-accounts` ŌĆö list of the host's service accounts
  (admin-only); columns: name, `BMNT`-prefixed access key, capability
  count, last-used relative time, expiry, status pill
- `/admin/service-accounts/new` ŌĆö mint form (route, not modal) with a
  domain-grouped capability picker + scope editor + expiry preset
- Shown-once dialog on Create + Rotate ŌĆö refuses dismissal on Escape /
  outside-click, gated Done button, four copy paths (access key alone,
  secret alone, `Authorization: Bearer AKID:SECRET` header, AWS-CLI
  profile snippet)
- `/files/webhooks` ŌĆö list of the caller's webhook subscriptions
  (user-tier); empty state lands new operators on the wizard
- `/files/webhooks/new` ŌĆö subscription form with target URL +
  HMAC secret (auto-generate or operator-supplied) + event-type
  checkboxes + region/bucket/prefix scope picker
- `/files/webhooks/$id` ŌĆö detail page with Python verification snippet
  + recent delivery history + Enable/Disable/Test/Delete actions
- Bearer auth middleware ŌĆö `Authorization: Bearer AKID:SECRET` parallel
  to the JWT session cookie; cookie wins when both present; bearer
  cannot elevate to ADMIN (SA bundle is both floor and ceiling)
- Federation event-driven replication ŌĆö `webhook.Engine.Subscribe(name,
  cb)` exposes an in-process pub/sub; `federation.Engine.SubscribeToEvents`
  bridges it; `ObjectCreated/Modified/Deleted` envelopes drive
  per-replica streamPut / DeleteObject within seconds

## What changed since v1.6.0

### Service-account data layer + admin API (`v1.7.0a`)

- **New `internal/serviceaccount` package.** `ServiceAccount` is the
  first-class record: name + owner user ID + `BMNT`-prefixed access
  key + bcrypt-hashed secret + `Capabilities []string` (granted
  capabilities, AND'd against the outer `Scopes` envelope) +
  `Scopes []string` + `ExpiresAt` + `LastUsedAt` + `RevokedAt`. Atomic
  JSON store at `{dataDir}/service_accounts.json` (same tmp+fsync+rename
  pattern as the rest of the stores).
- **Plaintext secret generated server-side, returned exactly once on
  Create + Rotate.** Never persisted, never logged, never returned
  again. The `BMNT` prefix on access keys is greppable across audit
  logs so an operator can resolve a rogue bearer token back to a
  service account by access key alone.
- **Six admin endpoints under `/api/v1/admin/service-accounts`** ŌĆö
  list / create / get / update / delete / rotate ŌĆö gated on
  `host:manage_users`. Cross-user GET / PUT / DELETE collapse to 404
  so the wire shape can't enumerate IDs across owners. Delete is soft
  (`RevokedAt` set) so audit greps can resolve the access key back to
  a name + owner months after revocation.

### Bearer-auth middleware (`v1.7.0b`)

- **The auth middleware tries in order:** existing JWT session cookie,
  then `Authorization: Bearer {AccessKeyID}:{Secret}` against the
  v1.7.0a SA store, then falls back to `SESSION_REQUIRED`. Cookie
  wins when both are present (an attacker who can set the
  Authorization header still can't override an `HttpOnly` `Secure`
  cookie).
- **Bearer matches resolve the SA** via `GetByAccessKey`, screen for
  `SERVICE_ACCOUNT_REVOKED` + `SERVICE_ACCOUNT_EXPIRED` before the
  bcrypt compare, then debounce-touch `LastUsedAt` and inject a
  `Claims` whose `UserID` is the SA owner and whose new
  `ServiceAccountID` field is the SA row ID.
- **Policy gates branch on `ServiceAccountID`.** SA-authed requests
  route through a new `policy.ServiceAccountAllows` pure-function that
  AND's the SA's granted `Capabilities` list against its outer
  `Scopes` envelope ŌĆö the SA bundle is both floor and ceiling, the
  JWT user's role assignments never apply. A missing capability
  returns 403 `ELEVATION_NOT_AVAILABLE` (distinct from
  `ELEVATION_REQUIRED` ŌĆö bearer tokens cannot elevate to ADMIN, so
  the FE must not render an elevate CTA for M2M callers).
- **Audit attribution rewrites the actor** from the SA owner's
  username to `sa:{SA.ID}` so an operator greping audit can
  distinguish human cookie activity from machine bearer activity at
  a glance.

### /admin/\* auto-elevation entry guard (`v1.7.0a.1`)

- **`AdminEntryElevationGuard`.** Sits inside AppShell and opens the
  elevation modal whenever the operator lands on `/admin/*` in USER
  mode ŌĆö mirrors the UserMenu "Switch to admin view" behaviour, but
  for deep links / bookmarks / manual URL entry. Closes a URL-bar
  bypass where landing directly on an admin page rendered the chrome
  in USER mode while the URL said admin.
- **Cancel routes to `/files`** with an info toast (`"Cancelled ŌĆö
  staying in user view"`); success leaves the page in place with
  mode = ADMIN. A `useRef` latches the last prompted pathname so
  navigation within `/admin/*` doesn't fire N modals.
- **`AdminUserModeBanner` belt-and-braces.** Sticky amber banner
  whenever the operator is on `/admin/*` in USER mode ŌĆö distinct from
  the v1.3.0a.4 `ElevationExpiredBanner` (which only handles the
  falling-edge adminŌåÆuser case). Carries "Elevate to admin" and
  "Drop to /files" buttons.

### Drop-privileges UI cache invalidation (`v1.7.0a.2`)

- **`handleDrop`** (the ├Ś on the ADMIN persona pill) now invalidates
  the `["auth","me"]` query immediately after `setAuthMode`, so the
  follow-up refetch reads the freshly-rotated cookie and the
  `AuthModeHydrator` agrees with local state instead of fighting it.
  Before: pill stayed on ADMIN amber + countdown even after the
  backend logout-elevation returned 200.

### Auto-elevation guard hydration race (`v1.7.0a.3`, `v1.7.0a.4`)

- **The guard now defers** while `useUser()` is still loading and also
  short-circuits when the freshly-arrived `user.mode === "admin" ||
  "elevated"`. Before these hotfixes the guard fired on every
  `page.goto('/admin/*')` for sessions that were already in ADMIN ŌĆö
  the provider's conservative USER default + the one-render gap
  before `AuthModeHydrator` runs `setMode` caused the modal to pop
  every nav, sitting on top of subsequent clicks. The smoke caught
  this against v1.7.0e/f as three overlay-intercepted admin clicks.
  Two new tests pin the loading branch + the user.mode-says-admin
  branch.

### Service-account admin UI (`v1.7.0c`)

- **`/admin/service-accounts` list page.** Table with name, `BMNT`
  access key, capability count, last-used relative time, expiry, and
  an Active/Expired/Revoked pill rendered in the order the v1.7.0b
  bearer middleware checks them (revoked beats expired beats active).
  Per-row Rotate + Revoke actions are dropdown-gated and run through
  the existing elevation guard so a 403 pops the ADMIN-tier modal once
  + retries the mutation on success.
- **`/admin/service-accounts/new` mint route** (route page, not modal,
  per the popups-max-2-fields doctrine). The capability picker is a
  domain-grouped, searchable checkbox grid backed by the policy
  registry's `usePolicies()` payload, with a per-capability scope
  editor that pre-fills `host:*` / `cluster:*` / `bucket:{cid}:*` /
  `key:{cid}:*` defaults so operators don't have to memorize the
  six-form scope grammar. Expiry picks from "Never / 1 month /
  6 months / 1 year / Custom date".
- **`<SecretShownOnceDialog>` shared component.** Lives in `shared/ui/`,
  reused by both Create and Rotate. Refuses dismissal on Escape /
  outside-click, gates the Done button behind an acknowledgement
  checkbox, and surfaces four copy paths ŌĆö access key alone, secret
  (show/hide-toggled) alone, `Authorization: Bearer AKID:SECRET`
  header, and a ready-to-paste `~/.aws/credentials` profile snippet.

### Webhook subscription type + delivery engine (`v1.7.0d`)

- **New `internal/webhook` package.** `WebhookSubscription` is the
  first-class record: name + owner user ID + target URL + bcrypt-
  hashed HMAC secret + event-type filter + region/bucket/prefix
  scope + `LastFiredAt` + `ConsecutiveFailures` + `Disabled`. Atomic
  JSON store; types + handlers + dispatcher all in one package.
- **`/api/v1/user/webhooks` CRUD** ŌĆö list / create / get / update /
  delete / test / enable / disable. Secret handling matches the
  v1.7.0a SA mint-only pattern: the plaintext HMAC secret is returned
  on Create + on rotated Update, redacted from every List / Get
  response.
- **HMAC-SHA256 signed delivery.** Every outbound POST carries
  `X-Basement-Signature: sha256=<hex>` over the JSON body. Retry
  policy: 3 attempts with 1s / 5s / 15s exponential backoff. After
  10 consecutive failures the subscription is auto-disabled and an
  `webhook:auto_disabled` audit entry fires.
- **Per-delivery audit + mutation audit.** `webhook:fired_success` /
  `webhook:fired_failure` on each delivery attempt;
  `webhook:create/update/delete/test/enable/disable` on each
  mutation. `POST /test` emits a synthetic envelope so operators can
  validate target + secret without waiting for real traffic.
- **Engine is robust to per-delivery panics** (recover-shielded);
  saturated queue drops the oldest envelope rather than blocking
  emit; Stop drains in-flight deliveries cleanly.

### Webhook subscription UI (`v1.7.0e`)

- **`/files/webhooks` list page.** Table with name, target URL,
  event-type chips, last-fired relative time, failure count, and an
  Enabled / Disabled / Auto-disabled pill. Per-row Test + Toggle +
  Delete actions; empty state lands new operators on the wizard.
- **`/files/webhooks/new` subscription form.** Single-route form
  (not a wizard ŌĆö the surface fits comfortably on one page) with
  target URL + HMAC secret (auto-generate-or-supply) + event-type
  checkboxes + region/bucket/prefix scope picker. HTTPS warning
  surfaces inline when the target URL is non-HTTPS ŌĆö operator-allowed,
  not blocked, but the warning lands so a `http://` typo isn't silent.
- **`/files/webhooks/$id` detail page.** Configuration summary,
  per-delivery history (last 50), and a **Python verification
  snippet** with the operator's actual subscription ID ŌĆö copy-pasteable
  into a Flask / FastAPI receiver to verify the `X-Basement-Signature`
  header against the shared secret. Enable / Disable / Test (synthetic
  envelope) / Delete actions in the header.

### Federation event-driven replication (`v1.7.0f`)

- **`webhook.Engine.Subscribe(name, cb) -> unsubscribe`** ŌĆö new in-
  process pub/sub API. Subscribers fire synchronously inside the
  dispatcher BEFORE per-webhook HTTP delivery, so internal subsystems
  can react to bucket events without configuring external webhooks.
- **`federation.Engine.SubscribeToEvents`** bridges federation into
  the bus. When an envelope's (regionId, bucket) matches a federation's
  primary, the engine queues a single-object replicate task onto a
  per-federation buffered channel that a dedicated worker drains.
  `ObjectCreated` / `ObjectModified` envelopes drive a streamPut to
  each replica; `ObjectDeleted` envelopes trigger a `DeleteObject`
  instead (new method on the replication-client interface + on the
  production `federationwire` adapter).
- **Polling stays as fallback.** The 10s polling tick from v1.6
  continues to run alongside the event-driven path for backends
  without webhook-source coverage. Both paths share the same
  `recordSuccess` / `recordFailure` helpers so the broken-after-3 /
  auto-failover semantics stay identical across them.
- **Saturated event channels drop the oldest task** with a log
  warning rather than blocking the dispatcher; the dropped task's
  convergence falls back to the next polling pass ŌĆö no event loss
  beyond the polling lag.
- **Wired in `main.go` after both engines start.** The user-region
  object DELETE handler now fires a real `object.deleted` event after
  a successful server-side delete; PUT events follow when the v2.0
  gateway lands. With this cycle live an operator deleting an object
  via the `/files` browser sees the delete propagate to every
  federation replica within seconds instead of waiting up to the 10s
  polling lag.

## Configuration

**No new env vars.** Service accounts persist to
`{BASEMENT_DATA_DIR}/service_accounts.json`; webhooks persist to
`{BASEMENT_DATA_DIR}/webhooks.json` ŌĆö both alongside the existing
user-regions / syncs / backups / shares / federations stores. Service-
account capabilities + scope flow through the existing policy
registry; webhook delivery uses HMAC with operator-chosen secret
(auto-generated when omitted at Create / Rotate time).

The webhook delivery worker pool runs in-process with the rest of
basement; HTTP timeouts, retry intervals, and the auto-disable
threshold (10 consecutive failures) are compiled-in. Federation's
event-driven replication uses the in-process `webhook.Engine` pub/sub
ŌĆö no external receiver is needed for federation itself.

## Breaking changes

**None.** All v1.7 additions are purely additive:

- `user_regions` schema unchanged
- `internal/sync` engine unchanged ŌĆö federation reuses it as the
  copy primitive on both the polling and event-driven paths
- `internal/federation` package gains `SubscribeToEvents` but the
  existing polling-tick path stays as fallback
- Existing v1.6 record shapes (federations, replicas, policy) are
  untouched
- Driver interface unchanged ŌĆö no driver knows or cares about
  service accounts or webhooks
- JWT session cookie remains the primary auth path; bearer is
  additive

  an empty file if absent); the admin sidebar gets a new "Service
  accounts" link between Policies and Audit log
- The webhook store reads `webhooks.json` (creates an empty file if
  absent); the user nav gains a "Webhooks" link
- Existing federations automatically wire into the event-driven path
  once the webhook engine is online ŌĆö no operator action required;
  polling continues as a fallback for any (region, bucket) pair that
  the webhook source doesn't cover
- CLI clients can switch from cookie auth to bearer for M2M
  workloads by minting a service account at `/admin/service-accounts/new`
  and dropping the access key + secret into their tool's credential
  configuration

## Architecture

v1.7 leans **directly** on v1.6's federation engine and v0.8's
sync engine. No driver changes; no new copy code path; no new env
vars. The new ground is:

1. A first-class `ServiceAccount` record + bcrypt-hashed-secret store
   + bearer-auth middleware integrated into the existing chain.
2. A first-class `WebhookSubscription` record + atomic-JSON store +
   in-process delivery dispatcher with retry / backoff / auto-disable.
3. An in-process pub/sub API (`webhook.Engine.Subscribe`) that lets
   internal subsystems consume the same envelope stream that drives
   external HTTP webhooks.
4. Federation engine subscribes to the in-process pub/sub and queues
   single-object replicate tasks per matching event ŌĆö sub-second
   convergence on the happy path vs. up to 10s for the v1.6 polling
   tick (which stays as fallback).

This is by design. The competitive moat is the *control plane* ŌĆö
service accounts + webhooks unlock CLI / MCP / CI pipelines without
forking the admin UI; event-driven federation tightens the durability
window without changing the federation topology semantics.

### Architecture preview: v1.8 + v2.0

- **v1.8 ŌĆö MCP server.** Authenticates via v1.7 service accounts,
  consumes a curated subset of the same JSON API the FE uses today.
  Single Go binary at `cmd/basement-mcp/`, stdio-only JSON-RPC 2.0
  transport, ten tools at launch (seven read + two write + one
  forward-compatible placeholder). (A dedicated `basement` CLI was on
  the original v1.8 plan; v1.8.0d dropped it ŌĆö aws-cli covers the
  object-store CRUD path and the web UI covers the control plane,
  so a separate CLI binary would have been redundant.)
- **v1.8 ŌĆö Mobile PWA.** Bearer auth lets a phone-bound service
  account authenticate offline + come online with already-minted
  credentials.
- **v2.0 ŌĆö S3 gateway.** Service accounts mint the SigV4 keys the
  gateway will verify; webhooks emit the inbound-write events that
  drive event-driven federation; together they're the substrate for
  the gateway routing inbound S3 requests through the v1.6 federation
  topology.

## Known issues / coming in v1.8

- **MCP server not yet shipped.** v1.7 is the bearer-auth substrate;
  v1.8 lands the read-mostly MCP surface (and a small subset of
  mutations, with operator-confirmed gating). (Shipped in v1.8.0c as
  `cmd/basement-mcp/`. A dedicated `basement` CLI binary was on the
  original v1.8 plan but was dropped in v1.8.0d in favour of
  aws-cli + the web UI.)
- **Mobile PWA not yet shipped.** Same pattern ŌĆö v1.7's bearer auth +
  the v1.0 polished web shell are the substrate; v1.8 lands the
  installable-PWA wrapper + mobile-tuned chrome.
- **Webhook delivery is best-effort within the retry policy.** Three
  retries with 1s / 5s / 15s backoff covers most receiver hiccups,
  but a receiver down longer than ~21s misses the envelope. The
  auto-disable threshold (10 consecutive failures) protects basement-
  ui from infinite retry against a stale receiver; a future cycle
  may add a dead-letter queue.
- **Bearer tokens cannot elevate.** By design ŌĆö M2M callers can't
  prompt for a password. The 403 distinguishes
  `ELEVATION_NOT_AVAILABLE` from `ELEVATION_REQUIRED` so the FE
  doesn't render an elevate CTA for bearer-authed sessions, and the
  capability check fails loud rather than looping.
- **Block-scrub feature-detection fallback.** Carry-over from v1.4 ŌåÆ
  v1.5 ŌåÆ v1.6 ŌåÆ v1.7 ŌĆö the `/v1/worker` feature-detection fallback
  for legacy Garage admin APIs is still on the v1.x backlog. Not on
  the v1.8 critical path (MCP server + PWA take priority).
