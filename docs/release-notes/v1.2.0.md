# Basement v1.2.0 — Release Notes

**Status**: Tagged · superseded by [v1.3.0](v1.3.0.md)
**Date**: 2026-05-21
**Tracker**: [ADR-0003](../adr/0003-sudo-style-admin-elevation.md)

> **Superseded by [v1.3.0](v1.3.0.md)** — the three-mode state
> machine (USER / ADMIN / ELEVATED) introduced here collapsed to
> two modes (USER / ADMIN) per the [ADR-0003 amendment](../adr/0003-sudo-style-admin-elevation.md#amendment-v130a4--two-mode-simplification--operator-configurable-ttl).
> `mode="elevated"` cookies and `target_mode="elevated"` API calls
> silently migrate to ADMIN; the safety remains the per-elevation
> TTL (now operator-configurable). See v1.3.0 notes for current
> behaviour.

## Headline

**Sudo-style admin elevation + key-first user model.**

v1.1 sharpened the user persona around a region-tier keychain but left
one rough edge: a single account with `host_admin` + `cluster_admin` +
their own UserRegions could fire a destructive admin action one click
away from the casual "browse my files" flow. The blast radius for
"oh I thought I was in user view" was the entire host.

v1.2.0 lands the **USER → ADMIN → ELEVATED** session state machine
from [ADR-0003](../adr/0003-sudo-style-admin-elevation.md). One
identity, one cookie, but admin authority is **gated behind a fresh
re-auth** with a short TTL. Standard Linux `sudo` ergonomics on top
of the v1.1 role model.

Alongside, the user keychain becomes truly **key-first**: each access
key is its own card on `/files`, and a single user can register
multiple keys against the same S3 endpoint ("Work S3" vs "Personal
S3"). The old `unique(userId, endpoint)` constraint relaxes to
`unique(userId, endpoint, alias)`; existing data carries through
unchanged.

## What changed since v1.1.0

- New session state machine: every cookie carries `mode` +
  `modeExpiresAt` claims; the gate middleware downgrades expired
  modes per-request
- New `POST /api/v1/auth/elevate` endpoint mints ADMIN (15 min TTL)
  or ELEVATED (5 min TTL) cookies after password re-auth
- New `POST /api/v1/auth/elevate/oidc/start` +
  `GET /api/v1/auth/elevate/oidc/callback` for OIDC step-up via
  `prompt=login&max_age=0` (rejects cached IdP sessions that
  silently skip `prompt`)
- New `POST /api/v1/auth/logout-elevation` drops privileges back to
  USER without re-logging-in
- `GET /api/v1/auth/me` now echoes the live mode + expiry +
  `oidcUser` boolean so the FE can render the right elevation
  affordance
- Capability gate enforces `MinModeFor(capability)`: destructive ops
  (`cluster:delete`, `bucket:delete`, `key:delete`,
  `policy:edit_matrix`, `host:manage_*`, `cluster:edit_layout`)
  return **403 `ELEVATION_REQUIRED`** with `mode_required: "elevated"`
  until the session steps up
- Frontend `<AuthModeProvider>` mirrors the cookie state in React
  with a 1Hz auto-downgrade tick; `<ElevationProvider>` exposes
  `useElevationGuard()` for destructive click handlers; the
  openapi-fetch middleware opens the elevation modal eagerly on any
  unhandled 403 `ELEVATION_REQUIRED`
- `<PersonaPill>` lives in both AppShell + UserShell: USER neutral,
  ADMIN amber + mm:ss countdown, ELEVATED orange + lightning bolt;
  flash + toast at <30s; "drop privileges" button next to the
  countdown chip
- User keychain becomes key-first: multiple keys per endpoint are
  legal, each surfaced as a `UserKeyCard` on `/files` with alias
  prominent + endpoint hostname as small mono subtitle
- New canonical `/files/keys/new` "Add a key" form; the old
  `/files/regions/new` route 301-redirects; `/files/keys` becomes
  "My Keys"
- Pre-v1.2 cookies (no `mode` claim) are grandfathered to **ADMIN
  for 7 days** from the v1.2.0a deploy, so the existing matthew
  session keeps working across the deploy without re-login

## Three-mode state machine

Every session is in exactly one mode at any time:

| Mode | Capabilities | TTL | Persona pill |
|---|---|---|---|
| **USER** | UserRegion / key access + read-only shares/syncs | Session lifetime | Neutral |
| **ADMIN** | All `host:*`, `cluster:*`, `bucket:*`, `key:*`, `policy:*` non-destructive ops | 15 min idle → auto-revert to USER | Amber + countdown |
| **ELEVATED** | Destructive ops (delete cluster/bucket/key/user, edit policy matrix, edit cluster layout) | 5 min idle → back to ADMIN | Orange + lightning bolt |

TTLs are operator-tunable via `BASEMENT_ADMIN_TTL_SEC` (default 900)
and `BASEMENT_ELEVATED_TTL_SEC` (default 300).

### Transitions

```
USER ──(re-enter password OR OIDC step-up)──▶ ADMIN
ADMIN ──(re-enter password OR OIDC step-up)──▶ ELEVATED
ADMIN ──(15 min idle)──▶ USER
ADMIN ──(click "drop privileges")──▶ USER
ELEVATED ──(5 min idle)──▶ ADMIN
any ──(logout)──▶ unauthenticated
```

### Capability-to-mode mapping

The gate stays the same — capability checks first, then the mode
check on top. Examples:

| Capability | Min mode |
|---|---|
| `bucket:view`, `objects:*`, `share:*` | USER |
| `cluster:test`, `cluster:view_layout`, `bucket:edit_alias`, `key:edit_permissions` | ADMIN |
| `cluster:delete`, `bucket:delete`, `key:delete` | ELEVATED |
| `host:manage_users`, `host:manage_policies` | ELEVATED |
| `policy:edit_matrix`, `policy:assign_role` | ELEVATED |
| `cluster:edit_layout` | ELEVATED |

Full list in `internal/auth/policy/min_mode.go`.

## Elevation flows

### Local-password users

1. USER clicks "Switch to admin view" in the UserMenu, OR clicks any
   destructive button while in ADMIN
2. FE pops the password modal (2 fields max — current password +
   submit)
3. `POST /api/v1/auth/elevate` validates the password against the
   stored bcrypt hash, mints a new cookie with bumped `mode` +
   `modeExpiresAt`
4. FE invalidates `/auth/me`, persona pill flips, original click
   handler retries against the now-elevated session

### OIDC SSO users

1. Same UX trigger (UserMenu or destructive click)
2. FE detects `oidcUser: true` from `/auth/me` and renders an
   "Elevate via SSO" button (no password field — IdP owns the
   credential)
3. `POST /api/v1/auth/elevate/oidc/start` mints a state token (5 min
   TTL, same-session bound) and returns the IdP authorize URL with
   `prompt=<BASEMENT_OIDC_ELEVATION_PROMPT, default "login">` +
   `max_age=0`
4. Browser hops to the IdP, re-enters credentials (or hits a fresh
   MFA challenge), bounces back to
   `/api/v1/auth/elevate/oidc/callback?state=…&code=…`
5. Callback validates state, exchanges the code, checks the new ID
   token's `auth_time` is within 60s (rejects IdPs that silently
   ignore `prompt=login`), mints the elevated cookie, 302s to
   `/?elevated=<mode>`
6. `AuthModeHydrator` picks up the query param, fires a success
   toast, invalidates `/auth/me`, strips the param

The `BASEMENT_OIDC_ELEVATION_PROMPT` env var (default `login`) is
operator-configurable in case the IdP uses a non-standard prompt
keyword.

## Key-first user model

v1.1 enforced **one key per (user, endpoint)** in `UserRegion`
storage. That conflated "the keychain" with "the connection list" —
fine for the simple home case, but it broke the very common pattern
of one user with two relationships to the same S3 endpoint:

- A **Work S3** key on `s3.us-east-1.amazonaws.com` with the
  company's IAM-issued limited grants
- A **Personal S3** key on the same endpoint with the user's
  personal account credentials

v1.2 relaxes the unique constraint to `(userId, endpoint, alias)`.
The same `(userId, endpoint)` pair may now appear multiple times so
long as each row has a distinct alias. The alias is the user-facing
name on the card.

`/files` reframes itself accordingly:

- Subtitle changes from "Your regions" to "Each card is one of your
  access keys"
- Each card is a `UserKeyCard`: **alias** is the prominent line,
  **endpoint hostname** is small mono subtitle, **access-key-ID
  truncated** below
- Two keys on `s3.us-east-1.amazonaws.com` render as two cards side
  by side, each with its own audit trail
- `/files/keys` ("My Keys") and `/files/keys/new` ("Add a key") are
  the canonical routes; `/files/regions/*` still resolves and
  redirects for back-compat

### Dropped constraint

| | v1.1 | v1.2 |
|---|---|---|
| Unique | `(userId, endpoint)` | `(userId, endpoint, alias)` |
| Two keys same endpoint | 409 Conflict | Allowed (distinct aliases) |
| Same alias collision | n/a | 409 Conflict |
| Existing v1.1 data | — | Implicit alias preserved on first read |

No migration step runs — existing `user_regions.json` records load
unchanged. The implicit alias from v1.1 (or operator-assigned name)
satisfies the new tri-key, so no operator action is needed.

The sync resolver picks the **first match per endpoint** when
mapping a key to an admin Connection (all keys at one endpoint
bridge to the same admin Connection anyway), and emits a debug log
when multiple keys exist so operators can see which key won.

## Configuration

Three new environment variables, all optional:

| Var | Default | Purpose |
|---|---|---|
| `BASEMENT_ADMIN_TTL_SEC` | `900` (15 min) | Idle TTL for ADMIN mode before auto-revert to USER |
| `BASEMENT_ELEVATED_TTL_SEC` | `300` (5 min) | Idle TTL for ELEVATED mode before drop to ADMIN |
| `BASEMENT_OIDC_ELEVATION_PROMPT` | `login` | OIDC `prompt` parameter sent during step-up; some IdPs expect `consent` or a custom value |

See `docs/configuration.md` for the full env reference.

## Audit log additions

The existing `actorRole` field on audit Events now carries `user` /
`admin` / `elevated` to reflect the session's mode at action time.
Three new action codes:

| Action | When emitted |
|---|---|
| `auth:elevate_success` | `/auth/elevate` or OIDC callback mints a higher-mode cookie |
| `auth:elevate_failure` | Password mismatch, state mismatch, expired state, or `auth_time` outside the 60s freshness window |
| `auth:elevation_required` | Gate returned 403 `ELEVATION_REQUIRED` — the user attempted a capability above their current mode |

Every destructive action (cluster/bucket/key/user delete, policy
edit, layout edit) is now logged with `actorRole: "elevated"` for
incident forensics. A flood of `auth:elevation_required` against
one user is a clear signal that someone is mashing destructive
buttons faster than the 5-min ELEVATED window allows — a useful
ops signal.

## Upgrade notes

**Operators in normal config**: pull v1.2.0, restart. No migration
runs, no data shape changes, no config required. The existing matthew
session is **grandfathered to ADMIN for 7 days** from the v1.2.0a
deploy via cookie-claim back-compat (cookies issued pre-v1.2 have no
`mode` claim and are treated as ADMIN-with-grace). After the 7-day
grace expires (or on next logout), the normal USER-default flow
kicks in and the operator elevates per-action via the modal.

**Operators with OIDC**: confirm your IdP honors `prompt=login` (or
configure `BASEMENT_OIDC_ELEVATION_PROMPT` for a value your IdP
respects). The 60s `auth_time` freshness check will reject IdPs that
silently cache the user's session and skip the re-auth prompt — this
is intentional: silent re-issue defeats the purpose of step-up.

**Operators integrating against the legacy `/api/v1/user/regions/*`
shape**: nothing breaks. The endpoints are unchanged; the storage
just permits multiple records per `(userId, endpoint)` now. A
`POST /api/v1/user/regions` that previously 409'd on duplicate
endpoint now succeeds if the alias differs.

## Breaking changes

**None.** Back-compat preserved end-to-end:

- Pre-v1.2 cookies → ADMIN with 7-day grace
- `/files/regions/*` routes → still resolve, redirect to
  `/files/keys/*`
- `/api/v1/user/regions/*` shape → unchanged
- Existing `user_regions.json` records → load unmodified
- `useUserRegions` React hook → kept as-is (matches storage type
  name); only `useCreateUserRegion` was renamed to `useCreateUserKey`

## Verification

What's covered by automated tests and what's documented but
**unit-tested only**:

| Surface | Unit tests | Integration / live-verified |
|---|---|---|
| Mode JWT claims (issue / parse / expire) | yes | yes (`/auth/me` round-trip) |
| `POST /auth/elevate` password path | yes | yes (basement.pq.io smoke) |
| Capability → min-mode mapping enforcement | yes | yes (gate returns 403) |
| Auto-downgrade on expired `modeExpiresAt` | yes | yes |
| `POST /auth/logout-elevation` | yes | yes |
| Pre-v1.2 cookie grace path | yes (synthetic old cookie) | yes (matthew session) |
| `/files/keys/new` flow + multiple-keys-per-endpoint | yes | yes (Playwright) |
| **OIDC step-up callback** (`/auth/elevate/oidc/*`) | yes (mocked IdP) | **NOT live-verified** — requires a configured IdP. Operators with Authentik / Keycloak / Pocket-ID should smoke-test before relying on it for production. |

The OIDC gap is the only verification debt going into v1.2. It is
covered by unit tests against a mock IdP that exercises state
validation, `auth_time` freshness rejection, and cookie issuance —
but the integration against a real Authentik / Keycloak / Pocket-ID
has not been run against basement.pq.io because the test deployment
runs local password only. Operators standing up SSO should run a
manual elevation flow end-to-end before disabling local password as
the break-glass.

## Known issues / coming in v1.3

- **Per-cluster `cluster_admin` role assignment UI** — still pending
  from v1.1. Today the operator assigns
  `cluster_admin@cluster:{cid}` via the JSON editor at
  `/admin/policies/assignments`; a dedicated picker is on the v1.3
  backlog
- **Bulk import of access keys** — `/files/keys/new` is one key at
  a time; a paste-CSV or paste-JSON flow for users with N existing
  keys is on the v1.3 backlog
- **Rotate-key flow** — today, rotation is "delete + add"; a
  dedicated rotate flow that preserves the alias + audit chain is
  v1.3
- **OIDC group-claim → role auto-mapping** — operator-configured
  mapping from IdP group claims to basement roles, to avoid
  per-user `/admin/policies/assignments` clicks at scale
