# Basement v1.4.0 — Release Notes

**Status**: Tagged
**Date**: 2026-05-22

## Headline

**Scale + perf: virtualized object browsing, key-perms pagination,
batch ops, growth analytics, block scrub.**

v1.3 made basement comfortable to run with multiple humans in the
install. v1.4 is the scale-and-perf cycle: the surfaces that started
hitching against thousand-bucket / thousand-object real-world data
get virtualization, pagination, and batch affordances, and the
"is anything growing fast?" / "is anything corrupt?" questions get
direct answers on the admin side.

- Bucket browser virtualized — flat directories with 10K+ rows scroll
  smoothly at fixed-row height
- `Driver.PerBucketStatsAvailable()` capability flag — Garage v1
  hides the Size + Objects columns at the user-region tier instead of
  showing rows of em-dashes
- `/admin/audit` gains real pagination (Prev/Next + "Page N of M" +
  Export CSV)
- `/admin/clusters/{cid}/keys/{kid}` Edit mode hydrates the FULL
  cluster bucket list with filter + 50-per-page pagination + sticky
  Save bar
- Per-file checkboxes + sticky batch-action bar in the bucket
  browser — multi-select delete via `Promise.allSettled` fan-out
- `/admin/usage` gains `Growth (7d/30d/90d)` column, a "Buckets growing
  fastest" panel, an amber anomaly banner when any tracked bucket grew
  more than 100% in the window, and a 7d / 30d / 90d range selector
- `/admin/clusters/{cid}/scrub` — Garage block-scrub UI with live
  status (Running / Idle), progress %, blocks scanned/corrupt,
  last-completed timestamp, free-form driver message, and a Run
  scrub button. AWS S3 + MinIO render a "Not supported" card with
  the capability reason text

## What changed since v1.3.0

### Scale: virtualized bucket browser (`v1.4.0a`)

- **`/files/{regionId}/b/{bid}` virtualized via
  `@tanstack/react-virtual`.** A flat directory with 10K+ rows scrolls
  smoothly at fixed-row-height (48px). Folders sort to the top, files
  in S3 order, sticky header, scroll resets on prefix change. The
  virtualizer mounts inside `[data-testid="virtual-object-list-scroll"]`
  and renders absolute-positioned row divs (`data-row-kind="folder"`,
  `"file"`, or `"loadMoreSentinel"`) — not `<tbody><tr>` anymore.
- **Infinite scroll.** The bucket browser auto-fetches the next
  continuation token when the user nears the bottom of a truncated
  page. No more "load more" button — the sentinel row tells the
  virtualizer to fan out a `useInfiniteQuery` `fetchNextPage()` call.
- **`Driver.PerBucketStatsAvailable()` capability flag.** Gates the
  Size + Objects columns on the per-region bucket list. Garage v1
  returns `false` at the user-region tier (no public stats endpoint
  there); AWS S3 / MinIO / Garage v2 return `true`. When the flag is
  false the columns hide rather than rendering rows of em-dashes — a
  driver-honest fallback per the driver parity doctrine.

### Scale: paginated audit (`v1.4.0a`)

- **`/admin/audit` pagination.** Default page size dropped from 200 →
  50 with Prev/Next + "Showing X-Y of Z (Page N of M)" footer. New
  `offset` query param backed by `Audit.QueryWithTotal` on the file
  logger. Disabled affordances on out-of-bounds clicks so the operator
  can't paginate past the result set.
- **Export CSV.** Client-side button that dumps the currently filtered
  page to a downloaded CSV. No server round-trip — the file always
  matches what's on screen.

### Scale: key permissions editor (`v1.4.0b`)

- **`/admin/clusters/{cid}/keys/{kid}` Edit mode hydrates the FULL
  cluster bucket list** (granted + ungranted) instead of just
  `key.buckets`. Operators grant access to new buckets without
  bouncing through the "+ Grant access" dialog.
- **Filter input.** `Filter buckets...` narrows by alias substring
  client-side. Mounts via `[data-testid="key-perms-filter"]`.
- **50-per-page pagination.** Previous/Next + "Showing X-Y of Z (page
  N of M)" indicator. Checkbox state survives pagination — the edit
  array is mutated in-place, not rebuilt per page.
- **"Show only granted" toggle.** Default off — shows ALL buckets; on
  hides ungranted rows. Mounts via
  `[data-testid="key-perms-only-granted"]`.
- **Sticky Save bar.** Pins to the bottom of the card via
  `[data-testid="key-perms-sticky-save"]` so the operator never has to
  scroll a long list to commit.

### Scale: batch object operations (`v1.4.0b`)

- **Per-file checkbox column** at the left edge of every file row.
  Folders are excluded — recursive deletes need explicit per-folder
  confirmation, not a tick.
- **"Select all visible" checkbox** in the table header with an
  indeterminate state when some-but-not-all visible files are
  selected. Mounts via `[data-testid="select-all-visible"]`.
- **Sticky bottom batch-action bar** appears when ≥1 object is
  selected — "N selected | Delete N objects | Cancel". Delete fires
  parallel DELETE requests via `Promise.allSettled`; partial failure
  surfaces a per-row error indicator (destructive icon + "delete
  failed" label + the backend's error message as a `title` attribute)
  and leaves survivors selected for retry.
- **Row height held at 48px** so virtualization perf from v1.4.0a
  stays intact.
- **Move/copy punted to v1.5** (needs the server-side copy + delete
  pattern wired in).

### Cluster health: block scrub UI (`v1.4.0c`)

- **`Driver.ScrubSupport()` / `ScrubState()` / `StartScrub()`** added
  to the driver interface. Garage v1 implements against
  `/v1/worker` + `/v1/worker/scrub-blocks`; Garage v2 against
  `/v2/GetWorkers` + `/v2/LaunchRepairOperation`. AWS S3 + MinIO
  advertise `Supported=false` with the operator-facing reason
  "Backend manages durability via internal mechanisms; no
  operator-initiated scrub".
- **New API endpoints**:
  - `GET /api/v1/admin/clusters/{cid}/scrub` — capabilities + state
  - `POST /api/v1/admin/clusters/{cid}/scrub` — kick off a scrub
  Both gated on `cluster:edit @ cluster:{cid}` — same gate as
  layout apply/revert. Unsupported drivers return 409
  `SCRUB_UNSUPPORTED` on POST; GET returns 200 with capabilities
  carrying the reason text so the UI can render the explanation card.
- **`/admin/clusters/{cid}/scrub` route**. Renders `ScrubState`
  (Running badge or Idle badge, blocks scanned/corrupt, progress %,
  last-completed timestamp, free-form driver message). Run scrub
  button POSTs to the kick endpoint; the `useClusterScrub` hook polls
  every 5s while a scrub is running, every 30s otherwise.
- **Cluster detail page gains "Maintenance — Scrub" link** above the
  Layout link, so operators land on the scrub surface from the same
  place they manage everything else about the cluster.

### Storage analytics: growth + anomalies (`v1.4.0c`)

- **`Growth (Nd)` column on `/admin/usage` per-cluster table.**
  Weighted-by-bytes rollup of contributing top-bucket growth %.
  Tooltip explains "% change vs Nd ago, weighted by current bytes".
- **"Buckets growing fastest" panel.** Top 10 by growth %; only
  buckets the operator already cares about (currently in top-bytes)
  appear here, which keeps the fan-out bounded.
- **Amber anomaly banner** when any tracked bucket grew more than 100%
  in the window. Surfaces buckets with per-bucket "Investigate?" links
  to the bucket detail page so the "what changed?" question lands
  before the operator scrolls.
- **7d / 30d / 90d range selector** feeds both the growth calc and
  the inline trend charts. Pure FE work on the existing
  `/admin/usage/series` + `/admin/usage/overview` endpoints — server
  clamps `?from=&to=` at 90d.

## Breaking changes

**None operator-facing.** All additions.

The Driver interface gains three new methods —
`ScrubSupport()`, `ScrubState(ctx)`, `StartScrub(ctx)`,
`PerBucketStatsAvailable()`. External `driver.Driver` implementations
(out-of-tree drivers, if any) need to add stubs:

```go
func (d *MyDriver) PerBucketStatsAvailable() bool { return true }
func (d *MyDriver) ScrubSupport() driver.ScrubCapabilities {
    return driver.ScrubCapabilities{
        Supported: false,
        Reason:    "scrub not implemented for this driver",
    }
}
func (d *MyDriver) ScrubState(ctx context.Context) (driver.ScrubState, error) {
    return driver.ScrubState{}, nil
}
func (d *MyDriver) StartScrub(ctx context.Context) error {
    return driver.ErrScrubUnsupported
}
```

All four in-tree drivers (garage_v1, garage, aws_s3, minio) updated
like-for-like per the driver parity doctrine — Garage gets the real
implementation, AWS / MinIO get clean unsupported responses.

`/files/{regionId}/b/{bid}` switched from `<tbody><tr>` rows to
virtualized div containers. Smoke / regression tests that selected
rows via `tbody tr` need to switch to `[data-row-kind="folder"]` /
`[data-row-kind="file"]` (the smoke script in `scripts/postdeploy-ui-smoke.ts`
has the updated selectors).

## Upgrade notes

**Operators in normal config**: pull v1.4.0, restart. No migration
runs, no data shape changes, no config required.

**Wire shape**: `/api/v1/user/regions/{rid}/buckets` now returns
`{buckets: [...], perBucketStatsAvailable: bool}` instead of a bare
array. The FE unwraps both shapes (`Array.isArray(body)
? body : body?.buckets`) so v1.3-era clients still parse. Custom
integrations against this endpoint should switch to the new envelope
shape.

**Block scrub against older Garage builds**: the v1 driver hits
`/v1/worker` on the cluster admin API. Some older Garage builds (or
v2 clusters mistakenly configured as v1 in basement) respond
"Unknown API endpoint: GET /v1/worker", in which case the scrub
page renders the error banner ("Couldn't load scrub status.") rather
than the Status card. The page still mounts cleanly with a Back link
to the cluster — no crash. If you see this, double-check the
cluster's driver field matches the actual Garage admin API version.

## Known issues / coming in v1.5

- **Block-scrub status against legacy Garage admin APIs.** The v1
  driver assumes the `/v1/worker` admin endpoint is present, which
  isn't true on every Garage build. We render the error banner
  branch gracefully but the operator can't kick a scrub from
  basement until the driver gets a back-compat probe. v1.5 will
  add a feature-detection fallback so the Run scrub button is
  hidden (not just error-bannered) when the endpoint is absent.
- **Backup / restore story.** basement itself runs as a single
  binary with JSON state under `BASEMENT_DATA_DIR`, but there's no
  in-product backup / restore flow yet. v1.5's headline is going to
  be the cross-backend backup wizard — S3 → S3 with retention
  policies and a "restore bucket from backup" surface — plus a
  documented backup path for the basement state directory itself.
- **Move / copy in the bucket browser.** Multi-select delete shipped
  this cycle; multi-select move and copy slipped to v1.5 — they need
  the server-side copy + delete pattern wired through the driver
  interface, and that work pairs naturally with the backup wizard.
- **More drivers.** B2, R2, Wasabi are all S3-API-compatible but ship
  with their own admin quirks. The shared `BuildS3Client` + driver
  capability flags should make these cheap drop-ins; planned for
  v1.5.x.
