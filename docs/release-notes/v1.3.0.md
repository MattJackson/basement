# Basement v1.3.0 — Release Notes

**Status**: Tagged
**Date**: 2026-05-22
**Tracker**: [ADR-0003 amendment](../adr/0003-sudo-style-admin-elevation.md#amendment-v130a4--two-mode-simplification--operator-configurable-ttl)

## Headline

**Multi-user onboarding + key-first model refinement + sudo elevation polish.**

v1.2 landed the headline architecture — region-tier user persona,
sudo-style admin elevation, key-first user keychain. v1.3 is the
follow-on cycle that takes those new surfaces from "shipped" to
"comfortable to run with multiple humans in the install":

- OIDC group claims auto-map to basement roles, so admins stop
  hand-editing `/admin/policies/assignments` every time a new
  teammate signs in via SSO
- Driver-aware endpoint hints in cluster + key forms — operators
  paste an AWS S3 host and the region label auto-suggests `us-east-1`,
  paste a Garage host and the placeholder reads `http://garage:3903`
- Per-region S3 addressing toggle + drop-in-place rotate-key flow
- Folder navigation in the bucket browser (delimiter-based
  `commonPrefixes`)
- Invite-token polish + bulk-import access keys at `/files/keys/new`
- Per-cluster `cluster_admin` assignment UI right on the cluster
  detail page
- ADR-0003 simplifies from three modes (USER / ADMIN / ELEVATED) to
  two (USER / ADMIN), with an operator-configurable TTL and a
  drop-in-place expiry banner that preserves form state

Plus a stack of hotfixes that fall out of running a real multi-user
deploy end-to-end: graceful key-revoked errors, the region label
honored in S3 signing, login no longer auto-redirects admins to
`/admin`, presign keys no longer double-encoded.

## What changed since v1.2.0

### Auth + onboarding

- **OIDC group-claim → role auto-mapping** (`v1.3.0a`). New
  operator-configured mapping from IdP group claims to basement
  role IDs. Applied on every OIDC login — matching mappings
  auto-assign, stale ones revoke, manual assignments never touched.
  Operator manages the table at `/admin/system` &rarr; "OIDC Group
  &rarr; Role Mapping". No more per-user click work when a new
  teammate joins the SSO group.
- **Invite tokens go persistent** (`v1.3.0d`). Backed by
  `{dataDir}/invites.json` with bcrypt-hashed tokens, atomic write,
  RWMutex. Default 30-day expiry (operator-tunable per invite).
  `/admin/users` gains a "Pending invites" section with create /
  label / revoke / rotate / copy-full-URL affordances. The optional
  label feeds the auto-generated username (sanitized to lowercase +
  alnum + dash), so an invite labelled "wife" provisions the user as
  `wife` instead of `user-abcd1234`. The public redemption endpoint
  (`POST /invites/{token}/redeem`) now verifies against the persistent
  store instead of accepting any well-formed input.

### Two-mode elevation (ADR-0003 amendment)

The v1.2.0 three-mode state machine (USER / ADMIN / ELEVATED)
collapses to two modes (`v1.3.0a.4`). ELEVATED was cognitive overhead
without real protection — the TTL is the safety, not a sub-mode.

- **`ModeElevated` dropped** from `MinModeFor()`. Every previously-
  ELEVATED capability (`cluster:delete`, `bucket:delete`,
  `key:delete`, `host:manage_users`, `host:manage_policies`,
  `policy:edit_matrix`, `policy:assign_role`, `cluster:edit_layout`)
  collapses into the same ADMIN tier as `cluster:edit` and friends
- **`ModeElevated` survives as a string alias for `ModeAdmin`** for
  one release cycle so v1.2 call sites compile unchanged
- **v1.2-era cookies with `mode="elevated"` silently migrate to
  ADMIN on read** in `currentMode()` — no logout required across the
  upgrade
- **`POST /auth/elevate` accepts `target_mode="elevated"` as a synonym
  for `"admin"`** — same back-compat treatment as the cookie path
- **New `OrgCapabilities.AdminSessionTTLSec`** (default 900s, range
  60–86400s, clamp on write) is the operator-configured per-elevation
  TTL. Set from `/admin/system` &rarr; "Admin session timeout"
  (dropdown of 5m / 15m / 30m / 1h / 2h / 8h presets + Custom
  seconds input). `/auth/elevate` reads this on each call instead of
  the hardcoded 15-min default
- **`PersonaPill` loses its ELEVATED variant** — one ADMIN variant
  with a wider warning ramp (amber at <2 min, red + flashing +
  "Stay admin" extend toast at <30s) gives the operator real lead
  time and a one-click extend
- **New `ElevationExpiredBanner`** renders at the top of `/admin/*`
  after an ADMIN→USER expiry transition. Offers a Re-elevate button
  that pops the standard elevation modal; the page itself stays
  mounted so any in-progress form keeps its state

### Driver-aware UX

- **Public `GET /api/v1/system/driver-defaults`** (`v1.3.0b`) returns
  the curated `EndpointDefaults` catalogue (admin URL, S3 endpoint,
  region label, one-sentence hints, optional docs link) for every
  registered driver; FE caches forever
- **Add Cluster + Edit Cluster forms** now render driver-specific
  placeholders (`http://garage-host:3903` for Garage,
  `https://s3.us-east-1.amazonaws.com` for AWS S3,
  `http://minio-host:9000` for MinIO) and inline hints under each
  input
- **Add Key form** gains a "Common endpoints" expandable with
  one-click "Use this" buttons for each driver, plus an auto-suggest
  that fills the region label when the operator pastes an endpoint
  matching a known pattern (`amazonaws.com` → `us-east-1`, `garage`
  → `garage`, `minio` → `us-east-1`) — never overwrites a region the
  operator has already typed

### Per-region S3 addressing toggle + rotate-key flow

- **`UserRegion.AddressingStyle` field** (`v1.3.0c`) threads
  `"path"` (default) or `"virtual_host"` from the keychain through
  `Registry.ForUserRegion` and the shared `driver.BuildS3Client`
  helper. New `driver.NewS3VirtualHostClient` mirrors
  `NewS3PathStyleClient`; both are picked through `BuildS3Client`
  which enforces the IP-host smart default — an IP literal forces
  path-style regardless of the requested toggle (virtual-host
  requires wildcard DNS for the bucket subdomain)
- **Backwards-compat**: every UserRegion persisted before this cycle
  reads back as path-style via `applyReadDefaults`; no migration
- **`POST /api/v1/user/regions/{regionId}/rotate`** updates accessKey
  + secret in place (preserving alias / endpoint / region /
  addressingStyle / `lastUsedAt`), audits as `region:rotate`, and
  invalidates the cached S3 client for the old (endpoint,
  accessKey) tuple so the next ListBuckets uses the fresh secret.
  Wrong-owner attempts collapse to 404
- **Frontend**: `/files/keys/new` gains an Advanced expandable with a
  "Use virtual-host addressing" toggle, auto-disabled when the host is
  an IP literal; AWS row in "Common endpoints" auto-checks the toggle
  (AWS prefers virtual-host for S3-tool compatibility); `/files/keys`
  cards show a "via path-style" / "via virtual-host" subtitle and
  gain a "Rotate key" button next to Delete

### Folder navigation in the bucket browser

- **`/files/{regionId}/b/{bid}` defaults `delimiter="/"`** (`v1.3.0c.1`)
  and returns `commonPrefixes` (sub-folder rows) alongside `objects`
  (files at this level). Folder rows render first, alphabetical; file
  rows after. Breadcrumb across the top is the bucket alias + each
  prefix segment, all clickable; an "Up to parent folder" affordance
  sits beneath
- **Driver interface change**: `ListObjects` gains a trailing
  `delimiter string` param — cascades through all four drivers
  (garage_v1, garage, aws_s3, minio), each setting
  `s3.ListObjectsV2Input.Delimiter` only when non-empty. The sync
  engine + bucket-access probe continue to pass `""` for flat
  recursive listing
- **Wire-shape rename**: the JSON field on `ObjectPage` is
  `commonPrefixes` (matches the S3 nomenclature), replacing the prior
  `prefixes`

### Bulk-import access keys

- **`/files/keys/new` "Bulk import" toggle** (`v1.3.0d`) swaps the
  single-key form for a paste-area that auto-detects three formats —
  CSV (with snake_case or camelCase header variants), TSV, and
  aws-cli credentials-file profile blocks
- **Per-row preview table** with client-side validation errors before
  submit
- **New `POST /api/v1/user/regions/bulk` endpoint** creates rows
  independently — a per-row failure (`DUPLICATE_REGION`,
  `INVALID_ENDPOINT`, `INVALID_REQUEST`) lands in the response's
  `errors` array with the original index but doesn't abort the rest
  of the batch
- `addressingStyle` rides along when present (default `path`)
- Gated only on USER-tier auth — anyone authenticated can bulk-add
  their own keys

### Per-cluster cluster_admin assignment UI

- **Cluster detail pages gain a "Cluster admins" section** above
  Buckets (`v1.3.0e`). Table renders user (display name + username),
  role, source (manual / OIDC / inherited from global), and a Remove
  button
- **Wildcard inheritance** (`cluster:*` or the `*` superuser scope)
  shows up with an amber "inherited from global" badge and a
  tooltip-explained disabled Remove — those have to be managed from
  `/admin/policies` because they affect more than this one cluster
- **`+ Add cluster admin` button** opens a two-field modal (user
  picker eagerly fetched from `/admin/users`, role picker defaulted
  to `cluster_admin` with other non-deprecated roles available) that
  POSTs to the existing `/admin/policies/assignments` endpoint with
  scope `cluster:{cid}`
- **New convenience endpoint**
  `GET /api/v1/admin/clusters/{cid}/admins` gated on
  `policy:view_matrix @ host:*` filters the global assignment list
  to scopes matching `cluster:{cid}` (exact, wildcard, and superuser
  via `ScopeMatches`), joins user `Name` from the store
  server-side, and marks inherited rows with a boolean so the FE
  doesn't have to re-do the matching. 404 for stale cluster links

### Hotfixes that fell out of running a multi-user deploy

- **`v1.3.0a.1`** — Region endpoints
  (`/api/v1/user/regions/{id}/buckets`, `/objects`,
  presign-get/put, multipart init/part/complete/abort, delete-object)
  now detect S3 auth-rejection codes (`InvalidAccessKeyId`,
  `SignatureDoesNotMatch`, `AccessDenied`, `Forbidden`,
  `InvalidSignature`) and surface them as 401 `USER_KEY_REJECTED`
  with the offending region + alias + endpoint + accessKeyId — FE
  renders an inline alert with "Delete this region" / "Add a fresh
  key" actions instead of a generic "internal error" toast
- **`v1.3.0a.2`** — Forces path-style S3 addressing across every
  driver via the shared `driver.NewS3PathStyleClient` helper. Garage
  requires path-style; IP-addressed MinIO requires it (no DNS
  wildcard for `bucket.10.x.y.z`); AWS S3 accepts it everywhere.
  Inline copies in all four drivers collapse into one call site so
  future driver work cannot drift back to virtual-host. Fixes a
  `404 NotFound` on user-region ListObjects against Garage
- **`v1.3.0a.3`** — Elevation UX hotfix: wrap destructive admin
  handlers with `useElevationGuard()` + auto-elevate on persona
  switch in the UserMenu, so admins flipping into /admin no longer
  get stuck staring at a 403 toast
- **`fcb7e0d`** — Login no longer auto-redirects admins to /admin.
  Every user lands on `/files` (user shell); operators step into
  admin via the UserMenu "Switch to admin view" affordance per the
  ADR-0003 amendment
- **`f1f0bc3` / `81b4928`** — Garage v1 driver actually honors the
  `region` field in its admin config (was hardcoded empty →
  defaulted to `us-east-1`, broke S3 signing for Garage-region
  deployments)
- **`9a4fd28`** — Folder navigation re-renders on URL prefix change
  (TanStack Router cached the same hook scope across the prefix
  param, so navigating into a sub-folder showed stale rows)
- **`e94f4f2`** — Object browser file last-modified renders; presign
  URLs no longer double-encode the key path (was breaking download
  for any key with a slash in the name)
- **`1a73bf2`** — PersonaPill latches the "session ended" toast on
  the mode transition, not on `expiresAt` — fixes a stale-toast
  flash when the cookie was refreshed mid-tick

## Breaking changes

- **`ModeElevated` is gone** as a distinct tier (silently migrates to
  ADMIN). v1.2 code that imported the constant still compiles —
  `ModeElevated` is a string alias for `ModeAdmin` for one cycle —
  but tests asserting the distinction need updating
- **Login no longer auto-redirects admins to `/admin`**. Every user
  lands on `/files`; admins flip via UserMenu. Bookmarks pointing at
  `/admin/clusters` (or any other admin route) still resolve
  unchanged

That's it for hard breaks. Everything else is back-compat preserved.

## Upgrade notes

**Operators in normal config**: pull v1.3.0, restart. No migration
runs, no data shape changes, no config required.

**Cookie / session format**: unchanged on the wire. Existing
sessions get the ADMIN 7-day grace from v1.2.0a until they expire
naturally or the operator logs out. `mode="elevated"` cookies
silently migrate to `"admin"` on read.

**OIDC group mappings**: only apply on next IdP login — existing
sessions keep their currently-assigned roles. New mappings registered
at `/admin/system` &rarr; "OIDC Group &rarr; Role Mapping" take effect
the next time the affected user signs in.

**Admin session TTL**: defaults to 900s (15 min) and is now editable
at `/admin/system` &rarr; "Admin session timeout". The previous
hardcoded 15-min value rolls into the new config field unchanged
on first read; no operator action needed.

## Known issues / coming in v1.4

- **Bucket browser performance** at 10k+ objects — the flat
  `tbody tr` render starts hitching above a few hundred rows.
  Virtualizing the table (windowed render via `@tanstack/react-virtual`
  or similar) is the v1.4.0a slot
- **Backup story** — `basement` itself runs as a single binary
  with JSON state under `BASEMENT_DATA_DIR`, but there's no
  in-product backup/restore flow. Operators today `tar` the data
  dir; a documented restore path is on the v1.4 backlog
- **More drivers** — B2, R2, Wasabi are all S3-API-compatible but
  ship with their own admin quirks. The shared `BuildS3Client` +
  driver capability flags should make these cheap drop-ins; planned
  for v1.4.x
