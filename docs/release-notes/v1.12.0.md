**Status**: Tagged. Current `v1.x` tip; subsequent patch releases tracked in [`../../CHANGELOG.md`](../../CHANGELOG.md).

## Headline

**CSK envelope encryption per ADR-0007: per-cluster secret key with multi-admin unlock, lock-state awareness in background jobs.**

v1.12 is the security-hardening minor that introduces per-cluster envelope encryption (ADR-0007): each cluster now has its own 32-byte Cluster Secret Key (CSK) wrapped under Argon2id-derived keys per admin user. The CSK lives only in memory after unlock; background jobs (backup runner, federation engine) gate on lock-state to prevent failures when clusters are locked. Three sub-cycles:

- **v1.12.0a**: CSK infrastructure with `ClusterSecretManager`, admin endpoints for unlock/lock/admins CRUD, and migration from legacy JWT-derived encryption (`docs/cycle-reports/v1.12.0a.md`).
- **v1.12.0b**: ADR-0007 compliance verification confirming federation uses UserRegion secrets (JWT-derived), not admin_tokens/CSK; backup runner implements lock-gate at `internal/api/backup_runner.go:96-109` (`docs/cycle-reports/v1.12.0b.md`).
- **v1.12.0c**: Backup runner lock-state awareness regression test and documentation capturing why federation needs no gates while backup does (`docs/cycle-reports/v1.12.0c-csk-awareness-verification.md`).

## Architecture context — CSK envelope encryption story

### Why per-cluster CSK? (ADR-0007 rationale)

v1.0.0a introduced at-rest encryption of Connection Config fields (`admin_token`, `secret_key`, etc.) under a JWT-derived key: `sha256(cfg.JWT.Secret)` as the AES-GCM wrapping key. This is one key for the whole deployment, shared with every JWT-signed cookie and every connection record at once. The blast radius and rotation story are both wrong for the threat model:

- Compromising the JWT secret unlocks **every backend** basement ever held a credential for.
- Rotating `JWT_SECRET` invalidates every cookie and every connections.json record simultaneously.
- Garage `CreateKey`'s `secretAccessKey` (newly persisted) would compound this blast radius by adding more secrets under the same key.

ADR-0007 (accepted 2026-05-23, see [`docs/adr/0007-per-cluster-envelope-encryption.md`](../adr/0007-per-cluster-envelope-encryption.md)) adopts envelope encryption with a per-cluster CSK:

```
+--------------------+     Argon2id        +-----------+   AES-GCM
| admin password     | ------------------> | wrapping  | --------> wrappedCSK (per admin)
| (user-supplied)    |   (per-admin salt)  |   key     |
+--------------------+                     +-----------+
                                             |
                                             v
                            +---------------------------------+
                            | CSK (32 random bytes)           | AES-GCM
                            |   plaintext lives ONLY in       | --------> per-secret ciphertext
                            |   ClusterSecretManager.csks     |              on disk
                            |   in process memory after       |
                            |   unlock; never on disk         |
                            +---------------------------------+
```

### Key design decisions

1. **Per-cluster scope**: Compromising one cluster's wrapping key doesn't touch another cluster's secrets (`clustersecret.go:290` — `IsUnlocked(clusterID string) bool`).
2. **Operator-controlled unlock**: The secret-of-secrets is a password the cluster admin holds, not an env var the process can read at any moment.
3. **In-memory after unlock**: Background jobs (federation, backup runner, metrics scheduler) need the secrets while the operator is away from the keyboard; the CSK cache in `ClusterSecretManager.csks[clusterID]` serves this purpose (`clustersecret.go:154`).
4. **Multi-admin**: N cluster admins, each with their own password, each able to unlock and re-lock independently via wrappedCSK records per `(clusterID, adminUserID)` (`clustersecret.go:100-106`).
5. **Safe migration**: Existing JWT-encrypted `admin_token` keeps working until the first successful CSK unlock; lazy migration on-first-unlock (`clustersecret.go:499-528` — `MigrateFromJWT`).

### What CSK protects vs. what stays where it is (ADR-0007 Section "Scope")

| Secret | Current key | After ADR-0007 |
|---|---|---|
| Cluster `admin_token` | JWT-derived (v1.0.0a) | **CSK** — migrated on first unlock (`internal/store/connections.go:190`) |
| Garage `CreateKey` `secretAccessKey` (newly persisted) | n/a (not persisted today) | **CSK** at create time |
| Per-user UserRegion S3 secrets | JWT-derived | **unchanged** — per-user scope; different trust model (`docs/adr/0007-per-cluster-envelope-encryption.md:97`) |
| Service-account secret (bcrypt) | bcrypt one-way hash | unchanged — verification-only, never decrypt |
| JWT signing secret itself | env var | unchanged — JWT keeps signing |

### Background jobs: lock-state gates

When a tick fires for a background job whose credentials need CSK:

- `clustersecret.IsUnlocked(clusterID)` is the gate (`clustersecret.go:293`).
- **Backup runner**: YES, requires lock-state gates because backup destinations may use CSK-protected connections (per ADR-0007 migration). Implementation at `internal/api/backup_runner.go:96-109` checks both source and destination Connection IDs; if either is locked AND has CSK admins configured, the runner skips with error `"cluster <cid> is locked — skipped; will retry on next scheduled run"`.
- **Federation engine**: NO lock-state gates needed. Per ADR-0007 Section "Scope: what CSK protects vs. what stays where it is", federation uses **UserRegion JWT secrets** (not admin_tokens/CSK). UserRegion secrets are per-user, not per-cluster admin; the user logs in with their basement password (not a separate cluster admin password), so the JWT-derived path is appropriate for v1.12.0a and beyond (`docs/cycle-reports/v1.12.0c-csk-awareness-verification.md:36-44`).

## API surface

New endpoints per ADR-0007 Section "API surface" (admin-tier, unlocked cluster required for most operations):

```
POST   /api/v1/admin/clusters/{cid}/unlock      {password}  → 200 {unlocked:true} or 401 INVALID_PASSWORD
POST   /api/v1/admin/clusters/{cid}/lock                    → 204
GET    /api/v1/admin/clusters/{cid}/lock-status             → {unlocked, requiresMigration, admins:[uid…]}
POST   /api/v1/admin/clusters/{cid}/admins      {adminUserId,password}  → 201 (caller must be unlocked)
DELETE /api/v1/admin/clusters/{cid}/admins/{adminUserId}    → 204
```

Any endpoint that needs CSK to decrypt a stored secret returns `423 LOCKED` with `{cluster_id, hint:"unlock required"}` instead of a 500/401; the FE intercepts and presents the unlock modal (`docs/adr/0007-per-cluster-envelope-encryption.md:144-147`).

## Configuration

**New env vars**: None for v1.12.0a/b/c — CSK is opt-in via admin UI action "Enable cluster encryption". The JWT secret remains unchanged; it's only used during the migration window when `MigrateFromJWT` re-encrypts legacy secrets under CSK (`clustersecret.go:499`).

**New data files**: `{DATA_DIR}/cluster_secrets.json` stores the list of `WrappedCSK` records per cluster (one entry per admin user ID). File mode 0600; never logged, never serialized into audit metadata (`docs/adr/0007-per-cluster-envelope-encryption.md:11`).

## Breaking changes

**None.** The v1.12 cycle is additive throughout:

- CSK is opt-in via admin UI action ("Enable cluster encryption"); existing clusters without CSK continue using JWT-derived encryption until migrated.
- Migration from JWT to CSK is lazy + on-first-unlock; the legacy ciphertext stays on disk until the rewrite is fsynced (`docs/adr/0007-per-cluster-envelope-encryption.md:162-165`).
- Background jobs that don't need CSK (federation) continue unchanged.
- No driver interface changes, no API wire-shape changes for non-admin endpoints, no storage migrations except new `cluster_secrets.json` file.

## Upgrade notes

**Existing operators**: Pull the new image; zero action required unless you want to enable per-cluster encryption:

```bash
docker pull ghcr.io/mattjackson/basement:v1.12.0
# or whatever tag corresponds to v1.12.0
```

Once running, navigate to any cluster's admin settings and click "Enable cluster encryption". The UI will prompt for a password; on submit:

1. `ClusterSecretManager.BootstrapFirstAdmin` generates a fresh 32-byte CSK (`clustersecret.go:440-442`).
2. Wraps the CSK under Argon2id(password, salt) and stores `WrappedCSK` for your admin user ID.
3. Caches the plaintext CSK in memory so existing connections keep working immediately.

**New operators**: Fresh installs start without CSK; enable it via the admin UI when ready. The backup runner is already wired with lock-gates (`internal/api/backup_runner.go:96-109`), so once you unlock a cluster, scheduled backups will run normally and auto-resume on subsequent unlocks after restarts.
