# Basement v1.9.0 — Release Notes

**Status**: Superseded by v1.10.0 — see
[`v1.10.0.md`](v1.10.0.md) for the compliance + integrity cycle
that completes the v1.x roadmap.

**Date**: 2026-05-22

## Headline

**WebDAV gateway + pluggable gateway architecture (SMB / NFS / FTP /
S3 ready for v1.10+).**

v1.8 lit up two new client surfaces on top of v1.7's M2M substrate:
Claude / Cursor / Code drive basement over MCP, and household users
install basement as a PWA on their phone. v1.9 lights up the third —
the native filesystem mount — and lays the architecture for the rest.

A new `internal/gateway/` package introduces a per-protocol `Gateway`
interface and a shared `Backend` data-plane contract, with a `Registry`
that the boot path populates and `/admin/system` reads from. WebDAV
ships as the first real implementation: mount basement as a folder in
Finder, Explorer, Nautilus, iOS Files, Android, or any rclone target,
authenticated with the same `username:password` an operator uses for
the web UI or a `BMNT...:secret` minted at `/admin/service-accounts`.
Four stub gateways (SMB, NFS, FTP, S3) register at boot so the full
roster surfaces in the UI from day one, marked "coming soon" until
their implementations land in v1.10+.

The Gateways card on `/admin/system` is registry-driven from
`GET /api/v1/admin/gateways`: capability chips per protocol
(read / write / delete / move / lock / basic-auth / bearer-auth /
sigv4-auth), live status (running, last activity, total requests),
mount URL with Copy + per-platform connect hints for implemented
gateways, and "Coming soon" affordances in place of toggles for stubs.
Adding a new gateway in v1.10+ takes one file in
`internal/gateway/{name}/` plus a `Register()` call in `main.go` — no
UI changes, no API changes; the registry-driven card and the
`/admin/gateways` endpoint pick it up automatically.

- **WebDAV gateway built-in.** `/webdav/` tree on the same chi router
  as `/api/v1` surfaces a user's regions + buckets + objects to any
  WebDAV client. Auth is HTTP Basic; LOCK / UNLOCK return 501
  (Finder + Explorer + rclone tolerate it); MOVE / COPY use
  ServerSideCopy; MKCOL at the region root creates a bucket.
- **Gateways card on `/admin/system`.** Registry-driven from
  `/api/v1/admin/gateways`; one row per protocol with capability chips,
  live status, mount URL, connect hints, and an Enabled toggle for
  implemented gateways. Auto-refresh on a 30s tick keeps the counters
  honest.
- **Pluggable architecture.** `internal/gateway/{Gateway,Backend,Registry}`
  interfaces; `ProductionBackend` composes `config.Admin`,
  `store.UserRegions`, `serviceaccount.ServiceAccounts`,
  `driver.Registry`, and `store.Connections` into a single S3-shaped
  data-plane surface. The `Backend` contract is what every Gateway
  (WebDAV today, SMB / NFS / FTP / S3 / future) implements against —
  no protocol code reaches into the driver registry directly.
- **Four stub gateways registered.** SMB, NFS, FTP, S3 ship as
  registered protocols with `Implemented()=false` so the
  `/admin/gateways` roster + UI cards exist from day one. Each
  carries a real description, the would-be capability list, and a
  v1.10+ implementation tracker.
- **Time Machine docs.** `docs/integrations/time-machine.md` is
  honest about basement not shipping native SMB: explains the absence
  of a production-grade pure-Go SMB server, recommends the basement
  BACKUP wizard against a real NAS, and documents the Samba + s3fs-
  fuse sidecar pattern as a community-supported workaround.
- **Plugin doc.** `docs/integrations/adding-a-gateway.md` walks
  through the Gateway + Backend interfaces, the boot-time wiring in
  `main.go`, the per-protocol Enable toggle pattern, and the testing
  recipe; points at `internal/gateway/webdav/` as the reference
  implementation.

## What changed since v1.8.0

### WebDAV gateway (`v1.9.0a`)

- **New `/webdav/` tree on the chi router.** Mounted at the root,
  alongside `/api/v1` and the SPA. Path namespace:
  `/webdav/` lists region aliases, `/webdav/{alias}/` lists buckets
  (Garage admin bridge applied when an admin Connection matches the
  region endpoint), `/webdav/{alias}/{bucket}/` lists objects
  (delimiter `/`), `/webdav/{alias}/{bucket}/key` is the per-object
  surface (GET / PUT / DELETE / HEAD / MOVE / COPY).
- **Auth: HTTP Basic, two creds shapes.** Username + password against
  the env-admin / store users (same `VerifyPassword` path as
  `/api/v1/auth/login`), OR a service-account `BMNT...:secret` pair —
  the same wire format the v1.7.0b bearer middleware accepts, just
  relocated into the Basic user/pass slots. A Finder mount can use the
  same key minted at `/admin/service-accounts`.
- **Built on `golang.org/x/net/webdav`.** Custom `FileSystem`
  implementation backed by `driver.ForUserRegion` — the same per-user
  S3 plumbing that powers `/api/v1/user/regions/*`. MKCOL at a region
  root creates a bucket; deeper MKCOL drops a zero-byte folder
  marker. MOVE / COPY use ServerSideCopy. LOCK / UNLOCK return 501
  (Finder + Explorer + rclone tolerate the absence of locking).
- **Middleware location moved.** The root-level
  `AllowContentType("application/json")` middleware moved into the
  `/api/v1` sub-router so WebDAV requests (XML on PROPFIND, arbitrary
  content types on PUT) aren't rejected with 415 before reaching the
  gateway. All existing `/api/v1` endpoints still enforce JSON.

### Gateways settings + Time Machine docs (`v1.9.0b`)

- **`OrgCapabilities.Gateways` nest** with a `WebDAV.Enabled` toggle
  (default on) and an optional `BaseURL` override. Legacy
  `org_capabilities.json` files that predate the nest migrate
  transparently in `OpenOrgCapabilities()` so a v1.9.0a → v1.9.0b
  upgrade does NOT silently disable existing WebDAV mounts.
- **Gateway kill switch.** `webdav.Handler` honours the toggle: when
  `OrgCaps.Get()` reports `WebDAV.Enabled=false`, every verb
  (including OPTIONS) short-circuits with a typed `403
  GATEWAY_DISABLED` before auth runs.
- **First Gateways card** on `/admin/system`: WebDAV section with
  Enabled toggle, mount-URL display + Copy button, optional BaseURL
  override, and per-platform connect hints for macOS Finder, Windows
  Explorer, Linux Nautilus, and iOS Files.
- **Explicit SMB "not supported natively" panel** links to the new
  Time Machine integration doc. `docs/integrations/time-machine.md`
  is honest about basement not shipping native SMB: explains why (no
  production-grade pure-Go SMB server), recommends the basement
  BACKUP wizard against a real NAS, and documents the Samba+s3fs-fuse
  sidecar pattern as a community-supported workaround with caveats.
- **`docs/integrations/webdav.md`** walks through the per-platform
  mount flow (Finder Cmd-K, Explorer Map network drive, Nautilus
  `dav://`, iOS Files), the auth surface, the v1.9 limitations
  (no LOCK / UNLOCK, no per-bucket ACLs at the WebDAV layer, slow
  large-bucket listings), and a troubleshooting matrix.

### Gateway architecture (`v1.9.0c`)

- **New `internal/gateway/` package.** Holds the
  `Gateway` + `Backend` + `Registry` interfaces. `Gateway` is the
  per-protocol contract (`Name`, `DisplayName`, `Description`,
  `Capabilities`, `Status`, `Implemented`, `Start`, `Stop`,
  `HTTPHandler`, `ListenAddress`). `Backend` is the shared data-access
  contract every Gateway calls (`AuthBasic` / `AuthBearer` / `AuthSigV4`
  plus the S3-shaped data plane: `ListRegions`, `ListBuckets`,
  `ListObjects`, `HeadObject`, `GetObject`, `PutObject`,
  `DeleteObject`, `CopyObject`, `CreateBucket`, `DeleteBucket`).
  `Registry` is the boot-time bag (`Register` / `Get` / `All` /
  `Enabled` / `StartAll` / `StopAll`).
- **`ProductionBackend`** composes existing primitives (`config.Admin`,
  `store.Users`, `serviceaccount.ServiceAccounts`, `store.UserRegions`,
  `driver.Registry`, `store.Connections`) into the `Backend` contract
  — the Garage admin bridge that powers `/webdav/{alias}` PROPFIND
  moved here verbatim. `mapDriverError` translates driver sentinels to
  gateway sentinels so protocol code can `errors.Is` without a
  transitive driver dep.
- **WebDAV moved** from `internal/webdav` into
  `internal/gateway/webdav` and now drives every storage call through
  `Backend`. `fs.go` no longer reaches into `driver.Registry` or
  `store.UserRegions`; `auth.go` delegates to `Backend.AuthBasic` /
  `Backend.AuthBearer`; `handler.go` preserves the v1.9.0a/b
  kill-switch + OPTIONS pre-auth + LOCK/UNLOCK 501 contract unchanged.
  Mount path unchanged; every existing verb test migrated to the new
  shape (in-memory fake `Backend` instead of driver+regions plumbing).
- **Four stub gateways register at boot.** SMB, NFS, FTP, S3 register
  with `Implemented()=false`, real `DisplayName`s + descriptions, the
  would-be `Capabilities` shape, and no-op `Start`/`Stop`. v1.10+
  cycles drop in real implementations behind the same interface.
- **New endpoint `GET /api/v1/admin/gateways`** returns the registry
  roster (`name`, `displayName`, `description`, `capabilities`,
  `status`, `implemented`, `enabled`) for the v1.9.0d generalised UI
  cycle. Gated on `host:manage_org_caps` via the existing `uiAdmin`
  group; returns `503 GATEWAYS_NOT_WIRED` when
  `SetGatewayRegistry` hasn't been called.

### Generalised Gateways UI + plugin doc (`v1.9.0d`)

- **Registry-driven Gateways card** on `/admin/system`. The
  WebDAV-hardcoded card from v1.9.0b is replaced by a card that
  renders one row per Gateway returned from `/api/v1/admin/gateways`:
  capability chips (read / write / delete / move / lock / basic-auth /
  bearer-auth / sigv4-auth), live status (running, active connections,
  last activity, total requests), mount URL with Copy button +
  per-platform connect hints for implemented gateways, "Coming soon"
  badge in place of an enable toggle for stubs. Auto-refresh on a 30s
  tick keeps the status counters honest.
- **`useGatewaysRegistry()` hook** wraps the endpoint; the card writes
  both the legacy `webdav` field AND the new generic
  `protocols["webdav"]` entry so v1.9.0b operators who flipped the
  WebDAV kill switch keep their state through the upgrade.
- **`OrgCapabilities.Gateways.Protocols`** is the new generic
  `map[string]GatewayConfig` nest (`Enabled` + `BaseURL` + `Options`)
  so v1.10+ gateways can ship per-protocol settings without a new Go
  field per gateway. A legacy v1.9.0b file with only
  `gateways.webdav.{enabled,baseUrl}` auto-migrates into
  `Protocols["webdav"]` on read.
- **Plugin doc** `docs/integrations/adding-a-gateway.md` walks
  through the Gateway + Backend interfaces, the boot-time wiring in
  `main.go`, the per-protocol Enable toggle pattern, the testing
  recipe, and points at `internal/gateway/webdav/` as the reference
  implementation. Stub docs (`smb.md`, `nfs.md`, `ftp.md`, `s3.md`)
  ship as implementation-tracking placeholders the card's per-row
  docs links point to.

## Architecture preview: v2.0

The S3 gateway in v2.0 becomes "just another Gateway implementation".
The `Backend` contract is already S3-shaped — `AuthSigV4` slot,
`PutObject` / `GetObject` / `HeadObject` / `DeleteObject` /
`CopyObject` / `ListObjects` / `CreateBucket` / `DeleteBucket` — so
the v2.0 implementation is a SigV4 verifier + an S3 request parser
on top of the same data plane that powers WebDAV today. The
federation topology from v1.6 ("read → nearest healthy replica;
write → primary") is reused unchanged because both layers call the
same `Backend`. Service-account-minted credentials gate ingress;
webhooks emit inbound-write events that drive event-driven
federation.

## Configuration

**No new env vars** for the server. WebDAV is enabled by default;
flip off at `/admin/system → Gateways → WebDAV → Enabled`. The
toggle takes effect on the next request — no re-deploy.

**Optional `BaseURL` override** per gateway. Defaults to the request
origin (`window.location.origin` on the FE, `r.Host` on the BE) so
the per-platform connect hints render the right URL automatically.
Operators behind a reverse-proxy chain where the public hostname
differs from the request origin can pin the BaseURL explicitly.

**Existing `org_capabilities.json` files** auto-migrate on read:
v1.8 files get `Gateways.WebDAV.Enabled=true` by default; v1.9.0a/b
files with `gateways.webdav.{enabled,baseUrl}` get the same values
mirrored into `Gateways.Protocols["webdav"]` on read so the v1.9.0d
registry-driven card sees them.

## Breaking changes

**None.** The v1.9 cycle is additive throughout:

- `/webdav/` is a new tree on the chi router; existing `/api/v1`
  routes unchanged in shape and behaviour.
- `OrgCapabilities.Gateways` is a new nest; legacy
  `org_capabilities.json` files migrate transparently.
- The v1.9.0b → v1.9.0c → v1.9.0d refactors are wire-stable: the
  `/webdav/*` path namespace, auth shapes, and kill-switch contract
  carry through unchanged. The `Gateways.WebDAV` field is preserved
  on the wire alongside the new `Gateways.Protocols["webdav"]` entry;
  the backend tie-breaks toward the legacy field on divergence so
  operators rolling forward from v1.9.0b keep their state.
- The internal-only `internal/webdav/` package moved to
  `internal/gateway/webdav/` in v1.9.0c. Any external Go consumer
  (rare — basement isn't a library) importing that path needs to
  update; the public HTTP surface is unchanged.

## Upgrade notes

Pull the new image:

```bash
docker pull ghcr.io/mattjackson/basement:v1.9.0
```

Existing operators with WebDAV enabled in v1.9.0a/b see no change:
the toggle state, the BaseURL, and the per-user mount path all
carry through. The Gateways card on `/admin/system` will now show
five rows instead of one — four of them ("SMB", "NFS", "FTP",
"S3") marked "Coming soon" with the toggle replaced by an em-dash.
That's expected.

Operators who want to mount basement as a folder:

1. Open the Files app (or Finder, Explorer, Nautilus, rclone).
2. Connect to `https://basement.example.com/webdav/`.
3. Use either your basement username + password, or a service-account
   access-key + secret minted at `/admin/service-accounts`.
4. Drag-and-drop into the resulting region/bucket folder writes
   straight to basement.

See `docs/integrations/webdav.md` for the full per-platform walkthrough,
the limitations (no LOCK / UNLOCK; no per-bucket ACLs at the WebDAV
layer; slow large-bucket listings), and a troubleshooting matrix.

**Reverse-proxy caveat.** Some proxy stacks (older Caddy on HTTP/2,
Cloudflare, certain Nginx configs) filter non-standard HTTP verbs at
the edge — PROPFIND / PROPPATCH / MKCOL / COPY / MOVE — and return
synthetic 405 responses before the request reaches basement. If
Finder reports "the server isn't responding" but the basement web UI
works, your proxy is the suspect; see `docs/integrations/webdav.md`
for the per-proxy fix matrix.

## Known issues / coming in v1.10

- **WebDAV: no LOCK / UNLOCK.** v1.9 returns 501 for both. Read+write
  clients (Finder, Explorer, Nautilus, rclone, iOS Files) tolerate
  the absence; a few legacy Office-via-WebDAV setups complain. Adding
  real lock state across long-running clients risks a memory leak;
  v1.10 may add a TTL-backed lock token map if operator demand
  surfaces.
- **WebDAV: large-bucket PROPFIND can be slow.** v1.9 walks the full
  bucket listing on every PROPFIND of `/webdav/{alias}/{bucket}/`.
  Clients that aggressively re-list (Finder column view, certain
  search-indexing daemons) will feel it on 10K+ object buckets.
  v1.10's pagination work mirrors the v1.4 bucket-browser
  virtualization onto the WebDAV `FileSystem`.
- **WebDAV: reverse-proxy verb filtering.** Some operator proxies
  (Caddy ≤ v2.8 on HTTP/2, Cloudflare default ruleset, certain
  Nginx configs) reject non-standard verbs at the edge. The smoke
  exercises OPTIONS + the auth challenge as the wire-reachable proof
  of the gateway being live; PROPFIND / MOVE / COPY require the
  operator's proxy to pass verbs through. Documented in
  `docs/integrations/webdav.md`.
- **SMB / NFS / FTP / S3 stubs.** Visible in `/admin/system` and the
  `/api/v1/admin/gateways` roster but unusable until implemented.
  SMB lands as a Samba-sidecar pattern (a pure-Go SMB server is
  still not production-grade); NFSv4 is the v1.10 lead; FTP / SFTP
  in v1.11; the S3 gateway is the v2.0 major-version slot.
- **Versioning + Object-Lock + SSE.** The big v1.10 cycle: ransomware
  shield + compliance posture. Complements v1.6 federation by making
  replicas immutable — a compromised primary can't poison the chain
  if every replica honours WORM. Versioning + Object-Lock are the
  S3-API-level primitives; SSE-S3 / SSE-C / SSE-KMS are the at-rest
  encryption tiers. Each backend (Garage v2, AWS S3, MinIO / OpenMaxIO)
  exposes a different subset of these via capability flags — the UI
  + driver work tracks the v1.4-style "honest capability gating"
  pattern.
- **Block-scrub feature-detection fallback.** Carry-over from
  v1.4 → v1.5 → v1.6 → v1.7 → v1.8 → v1.9 — the `/v1/worker`
  feature-detection fallback for legacy Garage admin APIs is still
  on the backlog. Not on the v1.10 critical path (versioning +
  Object-Lock + SSE take priority).
