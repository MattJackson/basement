# Basement v1.6.0 — Release Notes

**Status**: Tagged
**Date**: 2026-05-22

## Headline

**Federation + multi-backend replication — DR story for
cross-backend mirroring.**

v1.5 made backup a first-class product: "I have a copy I can restore
from." v1.6 is the other half of the durability story: federation.
A federated bucket lives on multiple backends as the *same logical
bucket* — home Garage + an off-site B2 copy stay in lock-step
automatically, with one promotion away from a failover when the
primary goes dark. basement owns the replication map, the
continuous-mirror engine, and the per-replica health view; the
clients keep hitting whichever backend they were already pointed at.
No bandwidth bottleneck, no new SDK, no driver changes — the
existing v1.5 sync engine is the copy primitive, federation just
gives it new polarity.

- `/files/federated-buckets` — list of the caller's federations
- `/files/federated-buckets/new` — 5-step wizard: primary, replicas,
  policy, initial-sync confirmation, review
- `/files/federated-buckets/$id/` — detail page with configuration,
  per-replica health table, Resync now, Promote-to-primary, Delete
- `/files/{regionId}/b/{bucketId}` — bucket browser surfaces a
  "Federated · N replicas, M in-sync" badge when the bucket is part
  of a federation; clicks through to the federation detail page
- Manual failover with old + new primary in a Confirmation dialog;
  opt-in auto-failover watchdog promotes the healthiest replica
  after `AutoFailoverSec` consecutive primary outages
- Per-replica health calc (`in-sync` / `lagging` / `stale` /
  `broken`) updated every engine tick; lag tracked in bytes +
  objects + seconds
- New audit events: `federation:create`, `federation:update`,
  `federation:delete`, `federation:failover`, `federation:resync`,
  `federation:replicate_object`

## What changed since v1.5.0

### Federation data layer (`v1.6.0a`)

- **New `internal/federation` package.** `FederatedBucket` is the
  first-class record: owner user ID + canonical name + `Primary
  ReplicaTarget` + `Replicas []ReplicaTarget` + `Policy
  FederationPolicy` + timestamps. `ReplicaTarget` carries `RegionID`,
  `Bucket`, and runtime fields (`LastSync`, `Health`, `LagBytes`,
  `LagObjects`) populated by the engine.
- **Atomic JSON store at `{dataDir}/federated_buckets.json`** — same
  tmp+fsync+rename pattern as `backups.json` / `user_regions.json`.
  Uniqueness enforced on `(ownerUserId, name)` so two users can each
  own a federation called `lsi` but one user can't own two.
- **`FederatedBuckets.FindByTarget(regionId, bucket)`** — substrate
  for the v1.6.0e bucket-browser reverse-lookup endpoint: given a
  (region, bucket) pair, return the federation it belongs to (if
  any). Walks both primary and replica positions.

### Replication engine (`v1.6.0b`)

- **Per-federation goroutines** with a 10s polling tick. Each tick
  the engine lists the primary's recent objects (modified-since
  cursor), compares against each replica, and queues missing /
  out-of-date objects for replication. The 10s default is
  conservative; webhook upgrade in v1.7 makes this event-driven.
- **Per-replica worker pool** (default 4 workers per replica). Each
  task is a single-object PUT from primary to replica using the v1.5
  sync engine's stream / server-side-copy primitive — no new copy
  code path.
- **Health derivation.** After each replicate-tick the engine
  recomputes `ReplicaTarget.Health` from lag: `in-sync` (current),
  `lagging` (behind by more than `Policy.LagAlertSec`), `stale`
  (10× the alert threshold), `broken` (sync errored repeatedly).
  Surfaced on the FE detail page as colored pills.
- **Audit-per-object.** Every `federation:replicate_object` is an
  audit entry. High volume by design (one entry per object × N
  replicas) — `/admin/audit` default filter excludes it; visible via
  `?action=federation:replicate_object` query param.

### API endpoints (`v1.6.0c`)

- **Six user-tier endpoints under `/api/v1/user/federated-buckets`**
  — CRUD (`POST`, `GET list`, `GET id`, `PUT`, `DELETE`) plus
  `/failover` (manual primary swap) and `/resync` (force full
  re-replicate). All gated on USER auth; 404-on-not-owner to avoid
  existence leaks. Audited per ADR-0005.
- **`GET /api/v1/user/federated-buckets/by-target?regionId=X&bucket=Y`**
  — reverse-lookup that powers the v1.6.0e bucket-browser badge.
  Returns the federation record (200) or null (204). Owner-scoped.
- **DELETE preserves replica data.** The basement federation
  record is removed, replicas keep their data on each backend.
  Documented in the wizard + confirm dialog.

### Frontend (`v1.6.0d`)

- **`/files/federated-buckets` — list page.** Table with name,
  primary (region + bucket), replica count, health summary, last
  activity, actions. Empty state lands new operators on the wizard.
  10s auto-refresh via the `useFederations()` hook so a freshly
  failed-over federation reflects within a tick.
- **`/files/federated-buckets/new` — 5-step wizard** (per ADR-0005
  § Federation wizard): Primary, Replicas (1-10), Policy
  (continuous/scheduled sync + lag alert + write quorum + opt-in
  auto-failover), Initial-sync confirmation, Review + save. Each
  step validated before Next; duplicate-replica warning surfaces
  inline on step 2.
- **`/files/federated-buckets/$id/` — detail page** with three
  sections: Configuration (primary, policy summary, timestamps),
  Replicas (per-target health table with lag + last-sync +
  Promote-to-primary), Actions header (Resync now, Delete). 5s
  auto-refresh on the federation record. Failover is a
  confirmation Dialog (two read-only fields per the
  popups-max-2-fields rule).

### Bucket-browser federation badge (`v1.6.0e`)

- **`useFederationByTarget(regionId, bucket)` hook** queries the
  reverse-lookup endpoint and caches per (region, bucket) tuple.
  Returns null when the bucket isn't federated.
- **`<FederationBadge>` on `/files/{regionId}/b/{bucketId}`** —
  rendered next to the bucket name when the hook returns a record.
  Text: "Federated · N replicas, M in-sync". Clicks through to
  `/files/federated-buckets/$id` for the federation's detail page.
  Carries `data-testid="federation-badge"` for smoke coverage.
- **No badge when not federated** — the bucket browser must render
  identically for non-federated buckets so the badge is purely
  additive. Smoke gates accept both "badge present + says
  Federated" and "no badge" as valid.

### Auto-failover watchdog (`v1.6.0f`)

- **Opt-in via `Policy.AutoFailover` + `Policy.AutoFailoverSec`.**
  When enabled, a per-federation watchdog goroutine pings the
  primary every 30s. After `AutoFailoverSec` consecutive failures,
  promotes the healthiest replica (lowest lag + `in-sync` health,
  falling back to `lagging` if none are clean). Default: off.
- **Healthiest-replica election.** The watchdog ranks replicas by
  `(health, lagBytes, lagObjects)` ascending. Ties broken by
  `LastSync` (most recent wins). If every replica is `broken` the
  watchdog logs an audit warning and refuses to promote — operator
  intervention required.
- **Audit + alert on every promotion.** `federation:failover`
  carries `actor=system`, `reason=auto_watchdog`, old primary, new
  primary. Operator notices via `/admin/audit` and the federation
  detail page's primary swap.

## Configuration

**No new env vars.** Federation runs in the same process as the
rest of basement, persists state to
`{BASEMENT_DATA_DIR}/federated_buckets.json` alongside the existing
user-regions / syncs / backups / shares stores, and uses the
caller's existing `UserRegions` (per-user encrypted S3 credentials,
AES-GCM keyed off the JWT signing secret) for every backend
touch — no separate "federation user" or service account.

The polling tick defaults to 10s. Auto-failover health probes
default to 30s when the policy is enabled. Both intervals are
compiled-in; tunable as policy fields in a future cycle if an
operator's home-network NAT-pinhole patterns argue for it.

## Breaking changes

**None.** All v1.6 additions are purely additive:

- `user_regions` schema unchanged — federations layer on top
- `internal/sync` engine unchanged — federation reuses it as the
  copy primitive
- `internal/backup` package unchanged — backup and federation are
  separate concepts; operators may use both ("federated for live
  mirror + scheduled backup for historical snapshots")
- Existing v1.5 record shapes (regions, buckets, keys, syncs,
  shares, backups) are untouched
- Driver interface unchanged — no driver knows or cares about
  federation; the engine drives it from the basement side

## Upgrade notes

Pull v1.6.0, restart. No migration runs, no data shape changes, no
config required.

The first time basement starts on v1.6.0 the federation store
reads `federated_buckets.json` (creating an empty file if absent)
and registers any existing records with the polling engine. There's
no "first run" side effect — federations are opt-in per user via
the wizard, so a fresh upgrade has zero federations registered and
the engine ticks idle.

## Architecture

Federation builds **directly** on v1.5's sync engine. No driver
changes; no new copy code path; no new env vars. The new ground is:

1. A first-class `FederatedBucket` record (atomic JSON store).
2. A polling-based replication engine that uses the existing sync
   engine as the copy primitive.
3. Six API endpoints + one reverse-lookup endpoint.
4. Three new frontend routes (list, wizard, detail) + one badge on
   the existing bucket browser.
5. An opt-in watchdog goroutine for auto-failover.

This is by design. The competitive moat is the *control plane* —
not a new client SDK, not a new bandwidth-hogging proxy. Clients
keep hitting whichever backend they were already pointed at;
basement keeps the backends in lock-step from the side.

### Architecture preview: v2.0 S3 gateway

This is the substrate for v2.0's S3 gateway. When the gateway
lands, it routes inbound requests **using the federation
topology**: reads go to the nearest healthy replica (latency- or
geo-ranked), writes go to the primary. The federation map this
cycle establishes is the routing map that the v2.0 gateway will
consume — and the auto-failover watchdog is what keeps the
"primary" pointer honest while the gateway is in flight. See
[ADR-0005](../adr/0005-federation.md) § "Enables" and § "Lays the
routing map for v2.0 gateway".

## Known issues / coming in v1.7

- **Polling-based replication has primary-side load.** The 10s tick
  lists the primary's recent objects on every cycle. v1.7's webhook
  upgrade flips this to event-driven (engine subscribes to bucket
  event webhooks from the primary backend, polling becomes the
  fallback for backends without webhook support). Drops the
  primary-side load to zero on the happy path.
- **No M2M auth yet.** Federation is a user-tier construct, so
  every API call goes through the same USER session as the rest of
  basement. v1.7 lands **service accounts** for machine-to-
  machine auth + a CLI / MCP surface — both upstream-of-federation
  features (a CI pipeline can mint a federation against its own
  service account credentials).
- **Eventual consistency window.** Writes confirmed by the primary
  may not be on replicas for up to 10s (the polling tick).
  `Policy.WriteQuorum > 1` can require multiple backends to confirm
  before considering a write durable, but introduces write latency.
  The v1.7 webhook upgrade tightens this to sub-second on the happy
  path.
- **Delete-marker propagation.** This cycle treats deletes as
  replicated PUTs of "tombstone" objects. Real S3 delete-marker
  support requires versioning, which lands in v1.10. v1.6
  documents this as a known limitation in the wizard's policy step.
- **Two-way / multi-primary federation.** Every replica accepts
  writes, eventually consistent across all (CRDT-ish). Out of scope
  for v1.6; punted to v2.x once the v2.0 gateway is solid.
- **Block-scrub feature-detection fallback.** Carry-over from v1.4
  → v1.5 → v1.6 — the `/v1/worker` feature-detection fallback for
  legacy Garage admin APIs is still on the v1.x backlog. Not on
  the v1.7 critical path (service accounts + webhooks take
  priority).
