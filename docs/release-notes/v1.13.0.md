## Architecture context — Skin system story

### Why pluggable skins? (v1.13 rationale)

The current `frontend/src/index.css` hardcodes all visual tokens: colors, spacing, fonts, border radius. Every org gets identical chrome regardless of brand requirements. The blast radius and customization story are both wrong for the threat model:

- **Zero customization**: Operators cannot change colors, spacing, or branding without modifying source code.
- **No per-org variation**: Each deployment shares the same visual identity; white-label deployments require fork + rebuild.
- **No progressive enhancement**: New operators inherit all existing styles even when they want minimal chrome.

v1.13 adopts a registry-driven skin contract:

```
+------------------+     Registry      +-------------+   FE Render
| basement-default | ----------------> | All()       | -> CSS variables
| (built-in)       |                   | Get(name)   |    Typography
+------------------+                   +-------------+    Density tokens
     ^                                    ^               borderRadius
     |                                    |              OperatorFooter
     |                                    |                  LoginHero
     v                                    v
+---------------------------------------------------------------+
| User-uploaded skins: {dataDir}/skins/*.basement-skin.json   |
| Policy files: .policy.json (public/private, CORS)           |
+---------------------------------------------------------------+
```

### Key design decisions

1. **Immutable registry after boot**: `internal/skin/registry.go` mutex-protects concurrent reads; registrations happen once at boot (`main.go`) plus optional runtime uploads via admin API.
2. **Built-in fallback**: `basement-default` ships with every deploy, mirroring current `index.css` tokens exactly for zero-diff upgrade (`internal/skin/default.go:28-93`).
3. **Typed contract**: `internal/skin/skin.go:63-212` defines `Skin` struct with all fields (palette, typography, density, borderRadius, footer, loginHero). FE types mirror this shape (`frontend/src/shared/api/queries.ts:33-75`).
4. **Per-user override gate**: Org-level `skinPolicy` field controls whether users can select their own skin from registry (`store/org_capabilities.go:60-73`): `"locked"` (no override), `"default"` (same as locked), `"user-choice"` (users can switch).
5. **Optional rendering**: Missing fields fall back to defaults; `loginHero === nil` renders current login layout unchanged, `footer === nil` shows no footer row (`v1.13.0c-skins-element-rendering.md:138-140`).

### What skins protect vs. what stays where it is

| Field | Current key | After v1.13 |
|---|---|---|
| `basement-default` palette + typography | Hardcoded in `index.css` (@theme block) | **Skin contract** — exposed via CSS variables (`--skin-sans-font`, etc.) |
| User-uploaded skin files | n/a (doesn't exist) | `{dataDir}/skins/*.basement-skin.json` with policy sidecars |
| Org-level active skin | `OrgCapabilities.ActiveSkin` string | **unchanged** — references skin name from registry |
| Per-user override | n/a (no user choice) | Session-persisted selection when `skinPolicy === "user-choice"` |

## API surface

New endpoints per v1.13.0b (`internal/api/skin_upload.go`):

```
GET    /api/v1/admin/skins                  → 200 SkinWithPolicy[] (admin-only)
POST   /api/v1/admin/skins/upload           → 201 {success, name, path} (multipart form)
PUT    /api/v1/admin/skins/:id/activate     → 200 {activeSkin: string} (persist to store)
DELETE /api/v1/admin/skins/:id              → 200 {success} (delete file + policy)
GET    /api/v1/admin/skins/:id/policy       → 200 SkinPolicy
PUT    /api/v1/admin/skins/:id/policy       → 200 SkinPolicy

GET    /api/v1/skins                        → 200 Skin[] (public registry, respects policy.public)
GET    /api/v1/skins/active                 → 200 Skin (org's active skin)

PUT    /api/v1/user/skin                    → 200 {success} (per-user override; requires skinPolicy=user-choice)
```

Any endpoint that references a non-existent skin name returns `404 NOT_FOUND`; the FE handles this gracefully by falling back to `basement-default` (`internal/api/skin_upload.go:317-319`).

## Configuration

**New env vars**: None for v1.13.0a/b/c — skin system is opt-in via admin UI action "Upload Skin". The built-in `basement-default` ships with every deploy; operators can choose to leave it active or upload custom skins.

**New data files**: `{DATA_DIR}/skins/*.basement-skin.json` stores uploaded skin JSON; `{DATA_DIR}/skins/*.policy.json` stores policy sidecars (public/private, CORS origin). File mode 0644 for skin JSON, 0644 for policy (admin-only read via authenticated endpoints) (`internal/api/skin_upload.go:159-180`).

## Breaking changes

**None.** The v1.13 cycle is additive throughout:

- `basement-default` mirrors current `index.css` tokens exactly; visual diff against pre-cycle deploy is zero (operator-locked acceptance gate per ADR-0008 roadmap).
- FE renders through CSS custom properties (`--skin-sans-font`, etc.) with fallback to existing hardcoded values when skin fields are missing.
- No API wire-shape changes for non-admin endpoints, no storage migrations except new `{dataDir}/skins/` directory.

## Upgrade notes

**Existing operators**: Pull the new image; zero action required unless you want to customize visual theming:

```bash
docker pull ghcr.io/mattjackson/basement:v1.13.0
# or whatever tag corresponds to v1.13.0
```

Once running, navigate to any cluster's **/admin/system** page and scroll to the "Skins" card:

1. Click **"Upload Skin"** to open upload dialog.
2. Drag-and-drop a `.basement-skin.json` file or click to browse.
3. File must be valid JSON with required `palette.light.primary`, `palette.dark.primary`, and `typography.sans` fields (`internal/api/skin_upload.go:58-86`).
4. On success, skin appears in registry list; click **"Activate"** to set as org-wide default.

**New operators**: Fresh installs start with `basement-default` active. The visual identity is identical to pre-v1.13 deploys (same color tokens, spacing, fonts). When ready to customize, upload a custom skin via admin UI.
