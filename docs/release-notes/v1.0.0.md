# Basement v1.0.0 — Release Notes (DRAFT)

**Status**: DRAFT — not tagged yet. Operator review pending.
**Target tag date**: 2026-05-21
**Tracker**: ADR-0001 + v0.9 polish backlog

## What is Basement?

A multi-backend admin UI for self-hosted S3-compatible object storage. One pane of glass over **Garage v1, Garage v2, AWS S3, MinIO/OpenMaxIO** — pick the backend per cluster; all four drivers stay in lockstep on feature support.

If you've used the MinIO web console and want something with broader backend coverage + portable identity + per-user accountability, this is for you.

## Headline changes since v0.4.x

### Three-tier role model (ADR-0001)

Three orthogonal roles, any combination per user:

- **Host Admin** — manages basement itself. No backend creds.
- **Cluster Admin** — manages a specific cluster's buckets/keys/layout via `admin_url + admin_token`.
- **User** — reads/writes specific buckets they've been granted on, via **their own scoped S3 key**.

Per-user S3 keys mean backend audit logs now attribute requests correctly. No more "all activity by GKclaude…" across every user.

### Flexible policy matrix

Roles + capabilities are **data**, not code. Edit at `/admin/policies`:

- 27 capabilities (host:manage_users, cluster:edit, bucket:delete, objects:list, policy:edit_matrix, …) registered in code
- Roles map capabilities to scopes (`host:*`, `cluster:{cid}`, `bucket:{cid}:{bid}`, `key:{cid}:{kid}`)
- Assignments are `(userId, roleId, scope)` triples
- Three seeded roles ship: `host_admin` (superuser), `cluster_admin`, `bucket_user`
- Operator can clone seeded roles, define new ones, and reassign

What-if simulator at `/admin/policies` answers "can user X do capability Y on scope Z" with a reasoning trace.

### Per-user encrypted bucket credentials

Cluster config now holds ONLY admin creds (`admin_url + admin_token + s3_endpoint`). User-tier S3 access keys live in `bucket_grants.json`, AES-GCM encrypted with the JWT signing secret.

Users connect a bucket via `/files/buckets/connect` with their own S3 access key — bucket-centric, not cluster-centric. Each bucket they want access to is a separate grant.

### Migration helper for existing operators

If you upgrade from v0.8.x with existing `access_key_id`/`secret_key` in your cluster configs, a banner appears on `/admin/*` offering one-click conversion to BucketGrants. Strips the creds from disk after.

### Operator polish

- **Lifecycle wizard** at `/admin/clusters/{cid}/buckets/{bid}` — express "after 30 days, delete" without writing JSON. Capability-gated per backend (Garage v1 returns Unsupported honestly; AWS / MinIO / Garage v2 have real implementations).
- **Usage overview** at `/admin/usage` — current snapshot: total bytes/objects per cluster, top 10 buckets by size, top 10 by object count, grants total.
- **Migrate wizard** at `/admin/migrate` — 3-step bulk cluster-to-cluster copy. Fans out the existing sync engine across every selected bucket; deep-linkable from cluster detail.
- **Persona switcher** in UserMenu — host admins can flip between admin view (/admin/*) and user view (/files/*) without re-logging.
- **All forms with >2 fields are routes**, not dialogs (popups-max-2-fields rule). Edit Cluster, Add Cluster, New Share, New Sync, New User, Connect a Bucket — all dedicated pages.

### Sync engine (v0.8.x)

Cross-cluster pull / push between any two clusters via the four drivers' StreamObject + ServerSideCopy capabilities. Job state persisted in `syncs.json` for resume + retry.

### Driver parity

Every backend touch goes through the Driver interface. Capability flags on the driver (e.g. `LifecycleSupport().Supported`) drive UI gating — **no driver-name checks in the UI**. All four drivers update like-for-like on every feature cycle: AWS S3 + MinIO/OpenMaxIO define "complete," Garage v1 + v2 fill in what they support.

Basement is one of the first admin UIs to ship real Garage v2 support.

## Security posture

- JWT-signed `__Host-` cookies, `SameSite=Strict`
- Bcrypt password hashing for local users
- AES-GCM encryption for stored S3 secrets in `bucket_grants.json`
- AES-GCM encryption for `admin_token` + `secret_key` in `connections.json` (v1.0.0a — silent migration on first boot)
- HMAC-signed share-auth cookies (no plaintext passwords in cookies)
- Per-user S3 signing for backend accountability
- OIDC support for identity federation (configurable per deployment)

## Two deployment postures

Gated by Host Admin via `/admin/system`:

- **Company mode** (default, `allowUserBackends=false`): Host Admin curates the cluster list. Users see exactly what they're granted on Host-Admin-approved clusters.
- **Multi-tenant mode** (`allowUserBackends=true`): Users can BYO buckets via their own S3 credentials, using whichever drivers Host Admin has enabled.

## Upgrading from v0.8.x

The migration helper auto-runs on first Host Admin login post-upgrade. If you have existing clusters with `access_key_id` + `secret_key` in their config:

1. Banner appears at top of any `/admin/*` page
2. Click "Migrate now →" → opens `/admin/migrations`
3. Per cluster, pick which buckets the existing key should grant access to (default: all)
4. Click Migrate → BucketGrants are minted for the current Host Admin, creds stripped from `connections.json`

For new operators: use `/admin/clusters/{cid}/keys` "+ New key" to mint a Garage key — the secret is shown ONCE at creation with copy buttons + aws-cli snippet. Save it; you can't read it again from basement's UI.

## Storage

Default data dir `/var/lib/basement`:

| File | What |
|---|---|
| `connections.json` | Cluster configs (admin URL + admin token + s3 endpoint) |
| `users.json` | Local-user records (bcrypt hashes; OIDC users have no hash) |
| `bucket_grants.json` | Per-user × per-bucket S3 keys (AES-GCM encrypted secrets) |
| `policies.json` | Roles + assignments matrix |
| `org_capabilities.json` | Signup mode, enabled drivers, allow-user-backends, OIDC-only |
| `shares.json` | Public share-link records |
| `syncs/{jobId}/state.json` | Sync job state |
| `audit/` | Audit log files (rotated daily) |

## Docker

```yaml
services:
  basement:
    image: ghcr.io/mattjackson/basement:v1.0.0
    user: "65532:65532"      # nonroot in distroless image
    volumes:
      - basement-data:/var/lib/basement
    environment:
      BASEMENT_ADMIN_USER: matthew
      BASEMENT_ADMIN_HASH: $$2a$$10$$...   # bcrypt hash
      BASEMENT_JWT_SECRET: ${JWT_SECRET}   # 32+ bytes; also used to encrypt bucket_grants
    ports:
      - "8080:8080"
volumes:
  basement-data:
```

**unraid operators**: the bind-mount volume must be chowned to UID 65532. Single line: `sudo chown -R 65532:65532 /mnt/user/appdata/basement`.

## Audit log (v1.0.0c)

Every mutating handler — cluster/bucket/key CRUD, share create/revoke,
user create/delete, policy edits, org-cap edits, sync create/delete —
records a JSON-Lines event into `{dataDir}/audit/YYYY-MM-DD.log`.
Files rotate by UTC date and never delete; operators rm or
cron-archive on their own cadence. Failed login attempts and
forbidden-capability checks also record (with `actor=""`,
`detail=…`) so privilege-escalation surfaces in the timeline.

Browse at `/admin/audit` — filter by actor / action / resource /
result / date range; right-click a row "Copy as JSON" for incident
write-ups. Gated on `host:manage_policies`.

## Usage time-series (v1.0.0d)

Per-bucket bytes + objects are now snapshotted hourly to
`{dataDir}/metrics/YYYY-MM-DD.jsonl` by a background scheduler.
Files rotate by UTC date and never delete — same retention contract
as the audit log (operator rm or cron-archive).

The `/admin/usage` page picks up an inline trend chart. Toggle
**Trend** in the header; clicking any bucket row in the two
top-N panels expands a 7-day sparkline drawn straight from the
snapshots (no external charting dep — single inline SVG).

API: `GET /api/v1/admin/usage/series?cid={cid}&bid={bid}` returns
`{snapshots:[{time,bytes,objects}], bucketAlias, range}`. Default
range 7 days, capped at 90 days. Gated on `host:manage_users` —
same Host Admin gate as the snapshot overview.

First boot after upgrade shows an empty-state "Recording started —
check back in an hour" until the first hourly tick lands.

## Known limitations going into v1.0

- Cluster Admin role-assignment UI is per-cluster only; per-zone / per-region scopes not yet supported
- OIDC group-claim → role mapping is manual (operator assigns via `/admin/policies`); auto-mapping in v1.x
- Audit log retention is operator-managed (rm or cron-archive old daily files); programmatic retention sweep in v1.x
- Metrics snapshot retention is operator-managed (same pattern as audit log); programmatic sweep in v1.x
