# Basement v1.1.0 — Release Notes

> **Superseded by [v1.2.0](v1.2.0.md). Sudo elevation + key-first
> model live.**

**Status**: Tagged
**Date**: 2026-05-21
**Tracker**: [ADR-0002](../adr/0002-region-tier-user-model.md)

## Headline

**Region tier replaces phantom Connections at the user persona.**

In v1.0.0 the user persona still spoke the language of "clusters" —
every time a user connected a bucket, basement spun up a fresh
phantom `Connection` record and showed it as a second cluster row at
`/files`. That conflated two concepts the user shouldn't see: a
**Cluster** (a backend instance with admin authority) and a **Region**
(an S3 endpoint where their buckets live).

v1.1.0 introduces a **region-tier abstraction** at the user persona.
The Cluster concept survives unchanged at the admin tier; the user
just sees Regions (their keychain) and the buckets each key can reach.

## What changed since v1.0.0

- New per-user keychain at `/files/keys` — one credential per endpoint
  the user is granted on, encrypted at rest with AES-GCM
- New `/files/regions/new` route replaces the old
  `/files/buckets/connect` flow — five fields (alias, endpoint, region
  name, access key, secret), all on one page
- `/files` lists Regions, not Connections — one row per endpoint
- Tap a region → basement signs `ListBuckets` with the region's S3
  key → backend returns whatever buckets the key can see (no more
  per-bucket grant explosion)
- For Garage backends, an admin bridge supplements S3 `ListBuckets`
  (Garage's data plane doesn't implement it) and filters the result
  down to buckets the user's key can actually reach
- Object-tier operations (list, presign-get/put, multipart_*, delete)
  all sign with the region's key — backend audit logs attribute every
  request to the actual user
- Sync + share creation accept either a real Connection ID or a
  region ID; the resolver translates regions to admin Connections on
  demand
- The `bucket_user` role in the policy matrix is deprecated — the
  region's S3 key is the permission; basement no longer invents
  per-bucket roles
- New `region:*` audit actions wired across the region tier:
  `region:create`, `region:delete`, `region:list_buckets`,
  `region:list_objects`, `region:presign_get`, `region:presign_put`,
  `region:multipart_init`, `region:multipart_part`,
  `region:multipart_complete`, `region:multipart_abort`,
  `region:delete_object` — Resource carries `region:{id}:{bucket}` and
  Detail carries `accessKey=<id>` so operator forensics can correlate
  with backend access logs

## The region model

See [ADR-0002](../adr/0002-region-tier-user-model.md) for the full
write-up. Brief summary:

- **Cluster Admin** (e.g. matthew on the home Garage) owns the
  cluster, creates buckets + S3 keys via `/admin/clusters/{cid}`, and
  hands `(endpoint, access_key, secret)` to a user out-of-band.
- **User** (e.g. wife) goes to `/files/regions/new`, pastes the three
  values, and basement stores a `UserRegion` record. Tap into the
  region, see the buckets that key can reach, browse + upload +
  download as normal.
- **Host Admin** (also matthew) flips between `/admin/*` and `/files`
  via the persona pill — but in `/files` only sees their own
  UserRegions, not other users'.

One vocabulary per persona. The Cluster concept survives unchanged at
the admin tier; the user just never has to think about it.

## Per-user keychain at `/files/keys`

Replaces the old per-bucket "view your cluster keys" mishmash with a
single page: one row per UserRegion, columns for alias, endpoint,
access key prefix, created, last used. Add a new region (link to
`/files/regions/new`), delete a region (with confirm), copy the
access key ID to clipboard. The secret is never read back from the
server — set-once at create time, then encrypted at rest.

## Sync + share are now region-aware

Both `/files/syncs/new` and `/files/shares/new` accept a region as
the source/destination picker — the backend resolver translates the
region's endpoint to the matching admin Connection at request time.

If a region has no admin bridge (the user has an S3 key for an
endpoint that basement has no admin authority on), the API returns
**400 `NO_ADMIN_BRIDGE`** with the offending endpoint + field name in
the response details. The UI renders a region-tier–specific block
explaining the gap and pointing at `/admin/clusters/new` for any
operator with the admin token to fill it.

This is a non-bug: sync + share intrinsically need admin authority
(cross-cluster copy, share-link auth), so a region-only key isn't
enough.

## `bucket_user` role deprecated

ADR-0002 step 6 deprecated the policy matrix's `bucket_user` role.
Bucket-level access is now controlled by **the S3 key the Cluster
Admin assigns to the bucket** — basement no longer invents
parallel grants in its own role matrix.

The role still exists in `policies.json` for back-compat (an
assignment with `bucket_user` no-ops rather than 404s), but the
matrix editor at `/admin/policies` shows a banner: *"Bucket-level
access is now controlled by S3 key assignment on the cluster admin
side, not by basement roles."* Operators upgrading don't need to
do anything; the role just becomes inert.

For new deployments, the seed only includes `host_admin` and
`cluster_admin`.

## Migration

`bucket_grants.json → user_regions.json` runs **automatically on first
boot of v1.1.0**. The migrator dedupes existing BucketGrants by
`(userId, endpoint)` (in practice the same user only has one key per
endpoint anyway), writes the corresponding UserRegion record with the
secret re-encrypted under the same JWT key, and renames the old file
to `bucket_grants.json.migrated-v1.1.0` so a downgrade can recover.

**Idempotent**: if `user_regions.json` already exists and the
migration ran once, the second boot is a no-op (logged to slog).

A summary appears in the startup log:

```
INFO  bucket-grants migration: read 4 grants, deduped to 2 regions, wrote 2 records
INFO  bucket-grants migration: renamed bucket_grants.json → bucket_grants.json.migrated-v1.1.0
```

If you have no `bucket_grants.json` (fresh v1.1.0 install) the
migrator silently skips.

## Breaking changes

The following legacy `/api/v1/user/*` endpoints are **deleted** in
v1.1.0e:

- `POST   /api/v1/user/buckets/connect`
- `GET    /api/v1/user/clusters`
- `GET    /api/v1/user/clusters/{cid}/buckets`
- `GET    /api/v1/user/clusters/{cid}/buckets/{bid}/objects`
- `GET    /api/v1/user/clusters/{cid}/buckets/{bid}/objects/{key}`
- `*      /api/v1/user/clusters/{cid}/buckets/{bid}/objects/{key}/presign-get`
- `*      /api/v1/user/clusters/{cid}/buckets/{bid}/objects/{key}/presign-put`
- `*      /api/v1/user/clusters/{cid}/buckets/{bid}/multipart/*`
- `GET    /api/v1/user/keys`

Their replacements all live under `/api/v1/user/regions/*`. The
basement frontend ships with the new URLs in the same release, so
in a normal upgrade nothing user-visible breaks.

## Upgrade notes

**Operators in normal config**: nothing to do. Pull v1.1.0, boot,
the migration runs once, everything keeps working.

**Anyone integrating directly with the legacy `/user/clusters/*` or
`/user/buckets/connect` API** (custom scripts, CI integrations): you
need to migrate to `/user/regions/*`. The new shape:

```http
POST /api/v1/user/regions
Content-Type: application/json
Cookie: __Host-basement_session=...

{
  "alias":       "home",
  "endpoint":    "https://s3.pq.io",
  "accessKeyId": "GK...",
  "secretKey":   "...",
  "region":      "garage"
}

→ 201 { "id": "ur_...", "userId": "...", "alias": "home", ... }
```

Then `GET /api/v1/user/regions/{regionId}/buckets`,
`GET /api/v1/user/regions/{regionId}/buckets/{bid}/objects`, etc. See
`docs/api.md` for the full surface.

## Known issues / coming in v1.2

- **Per-cluster-admin role assignment UI** still pending — today the
  operator assigns `cluster_admin@cluster:{cid}` via the JSON editor
  at `/admin/policies/assignments`; a dedicated picker is on the
  v1.2 backlog
- **Sudo-style admin elevation** lands in v1.2 per
  [ADR-0003](../adr/0003-sudo-style-admin-elevation.md) — the
  persona pill becomes a state machine (USER → ADMIN → ELEVATED) with
  re-auth at each transition. Lowers the blast radius for an operator
  who holds host_admin + cluster_admin + their own UserRegions on
  the same account.
