# Basement v1.5.0 — Release Notes

**Status**: Tagged
**Date**: 2026-05-22

## Headline

**Backup story: scheduled S3 → S3 backups with GFS retention +
point-in-time restore.**

v1.4 made the operator-facing surfaces comfortable at thousand-bucket
/ thousand-object scale. v1.5 is the durability cycle: basement
now owns the cross-bucket backup story end-to-end. Named, scheduled
backups run on a cron engine; snapshot mode writes timestamped
history with Grandfather-Father-Son retention; the restore wizard
walks the operator from "pick a snapshot" through "land it in a
target bucket" without a CLI in sight. Mirror mode stays available
as a continuous one-shot for operators who just want the latest copy
mirrored offsite.

- `/files/backups` — list of the caller's scheduled backups
- `/files/backups/new` — 5-step wizard: source, destination, mode +
  retention, schedule, name + review
- `/files/backups/$id/` — detail page with config, snapshot table
  (snapshot mode), and run history with prune counts
- `/files/backups/$id/restore` — 3-step wizard: pick snapshot,
  pick destination, confirm + run
- `Backup.Mode = mirror | snapshot` — mirror overwrites the
  destination on every run (default; back-compat with v1.5.0a); snapshot
  writes to `{dst}/{slug(name)}/{YYYY-MM-DD_HH:MM:SS}/`
- GFS retention (`{KeepDaily, KeepWeekly, KeepMonthly}`, default
  `{7, 4, 12}` ≈ 14 months / 23 stored snapshots) — pure-function
  `PlanPrune` with 17 table-driven tests
- Snapshot-level restore deep-link from the detail page's
  per-snapshot "Restore →" button (`?ts=YYYY-MM-DD_HH:MM:SS`)

## What changed since v1.4.0

### Backup CRUD + cron engine (`v1.5.0a`)

- **New `internal/backup` package.** `backup.Backups` is an atomic
  JSON store at `{dataDir}/backups.json` (same atomic-write pattern as
  the rest of basement's store). `backup.Scheduler` wraps
  `github.com/robfig/cron/v3` with panic-recovery per job so a
  malformed cron expression or a driver-side hiccup can't take down
  the scheduler goroutine.
- **Six user-tier endpoints under `/api/v1/user/backups`** — CRUD +
  `/run`, gated on the same USER auth as `/user/syncs` with 404-on-
  not-owner to avoid existence leaks. Backup runs go through the
  existing v0.8.x sync engine via a runner closure — no duplication
  of pull semantics, no new copy implementation.
- **`/files/backups` (list)** + **`/files/backups/new` (wizard)** +
  **`/files/backups/$id/` (detail)**. Wizard accepts manual / daily
  HH:MM / weekly day+HH:MM / monthly day-of-month+HH:MM / custom cron;
  detail page shows last-10 run history, edit-schedule-inline,
  enable+disable toggle, run-now button, delete.

### Snapshot mode + GFS retention (`v1.5.0b`)

- **`Backup.Mode` enum.** `mirror` (default for back-compat with
  v1.5.0a records) or `snapshot`. Snapshot runs write to
  `{dstBucket}/{slug(name)}/{YYYY-MM-DD_HH:MM:SS}/`; mirror runs keep
  the v1.5.0a behaviour of overwriting the destination.
- **`Backup.Retention = RetentionPolicy{KeepDaily, KeepWeekly, KeepMonthly}`.**
  Default when none specified is `{7, 4, 12}` — roughly 14 months of
  history with 23 stored snapshots. Empty-input, dense-day, sparse-
  day, ancient-snapshot, clock-skew, and sort-invariant edge cases
  are all covered by the 17 table-driven `PlanPrune` tests in
  `internal/backup/retention.go`.
- **`BackupResult` gains `SnapshotPrefix`, `SnapshotsPruned`,
  `BytesReclaimed`.** Surfaced on the detail page's run history table
  so the operator can see "this run kept history, pruned 3 old
  snapshots, reclaimed 1.2 GiB" without grepping the audit log.
- **New endpoint `GET /api/v1/user/backups/{id}/snapshots`** returns
  the most recent 10 snapshots with object + byte counts. Detail page
  renders this as a snapshot table with "Browse →" links to the
  destination bucket browser (deep-link with prefix pre-filled).
- **Wizard grows a 3rd step (Mode + retention)** with the GFS
  defaults pre-populated; existing 2-step (Source / Destination) and
  closing 2-step (Schedule / Name+review) flank it for the 5-step
  total.

### Snapshot restore wizard (`v1.5.0c`)

- **`POST /api/v1/user/backups/{id}/restore`.** Synchronous endpoint
  (request stays open until the copy finishes); body is
  `{snapshotTimestamp, dstRegionId, dstBucket, dstPrefix?,
  overwriteExisting}`. `snapshotTimestamp = "latest"` resolves at
  request time. Response carries the per-object summary
  (`{success, resolvedSnapshot, objectsCopied, objectsSkipped,
  bytesCopied, startedAt, completedAt, errors?}`).
- **`/files/backups/$id/restore` — 3-step wizard.** Step 1: snapshot
  selector (radio: "Latest" / explicit timestamp + dropdown of the
  recent snapshots). Step 2: destination region + bucket + optional
  sub-prefix, defaulted to the backup's original source so an
  operator who's just had a bucket nuked can restore in place with
  one click. Step 3: confirm with the `overwriteExisting` toggle —
  default off, on says "Overwrite existing objects in the destination"
  and surfaces a warning that this is destructive.
- **Result view** in place of step 3 once the restore returns —
  Success / Completed with errors header, per-object counts,
  bytes copied, started / completed timestamps, top-10 errors
  if any.
- **Snapshot-level deep-link.** The detail page snapshot table
  gains a per-row "Restore →" link that navigates to
  `/files/backups/$id/restore?ts=...` and pre-fills the snapshot
  selector. The wizard reads the `ts` search param via the
  `validateSearch` hook.
- **Mirror-mode short-circuit.** When `backup.mode === "mirror"` the
  restore route renders a `Restore is only available for snapshot-
  mode backups. This backup is in mirror mode — its destination
  already holds the latest copy.` notice instead of the wizard, since
  there's no history to roll back to.

### Routing hotfix (`v1.5.0c.1`)

- **`/files/backups/$id/restore` was unreachable in v1.5.0c.**
  v0.3.1-class regression: `frontend/src/routes/files/backups/$id.tsx`
  (the detail page) coexisted with `frontend/src/routes/files/
  backups/$id/restore.tsx` (the wizard). TanStack treats this as a
  parent-with-children configuration, but `$id.tsx` had no
  `<Outlet />` in its component body, so the restore route mounted
  under the detail page's content and never displayed. Fixed by
  renaming `$id.tsx` → `$id/index.tsx` so both routes are leaves
  under the `/files/backups` layout. The detail page still mounts at
  `/files/backups/$id/` (handled identically by TanStack); the
  restore wizard now mounts at `/files/backups/$id/restore` as
  intended.

## Configuration

**No new env vars.** Backup runs use the caller's existing
`UserRegions` (per-user encrypted S3 credentials, AES-GCM keyed off
the JWT signing secret); no separate "backup user" or service
account is involved. The cron scheduler runs in the same process
as the rest of basement and persists job state to
`{BASEMENT_DATA_DIR}/backups.json` alongside the existing user-
regions / syncs / shares stores.

## Breaking changes

**None.** All v1.5 additions are purely additive:

- `user_regions` schema unchanged
- `internal/sync` engine unchanged — backup runs reuse the existing
  pull semantics via a runner closure, no new copy code path
- Existing v1.4 record shapes (regions, buckets, keys, syncs,
  shares) are untouched
- Mirror mode is the default for `Backup.Mode` so any record persisted
  by an earlier dev cut (e.g. a v1.5.0a backup) keeps working
  unchanged after upgrading to v1.5.0b / v1.5.0c

## Upgrade notes

Pull v1.5.0, restart. No migration runs, no data shape changes, no
config required.

The first time the scheduler wakes after an upgrade it reads
`backups.json` (creating an empty file if absent) and registers any
non-`disabled` records with the cron loop. There's no "first run"
side effect — scheduled jobs only fire on their next cron tick, not
at process start.

## Known issues / coming in v2.0

- **Large restores are synchronous.** v1.5.0c's restore endpoint
  keeps the HTTP request open until every object has been copied.
  Works fine for the typical "I lost a few hundred MB, give me last
  Tuesday's snapshot" path but doesn't scale to "I lost a 50 GB
  bucket, restore from last month". v2.0 will move long-running
  restore (and long-running first-time syncs) to a background job
  with a poll-able progress endpoint and a streaming status row in
  the detail page.
- **Cross-driver restore.** A snapshot taken from a Garage source can
  restore to any S3-compatible destination, but B2 / Wasabi / R2 are
  not yet first-class drivers in v1.5 — they work today as generic
  S3 endpoints (via a UserRegion against the provider's S3 API) but
  there's no provider-specific quirk handling. v2.0 will land them
  as proper drivers behind the existing capability-flag interface.
- **Backup of basement's own state directory.** The cycle's
  internal backlog mentioned a "documented backup path for
  `BASEMENT_DATA_DIR`" — that work landed as documentation in
  `docs/configuration.md` (rsync the atomic-write JSON files, no
  database freeze needed) rather than an in-product surface. v2.0
  will revisit if an operator-facing backup-config surface becomes
  worth the additional UI weight.
- **Block-scrub status against legacy Garage admin APIs.** Carry-over
  from v1.4 — the feature-detection fallback for `/v1/worker` slipped
  out of v1.5 scope when the backup story took the headline. Still
  on the v2.0 list.
