## What changed since v1.7.0

### Rebrand cutover + AGPLv3 (`v1.8.0b`, `v1.8.0b.1`)

- **Project renamed `basement-ui` â†’ `basement`.** The original name
  outgrew itself â€” basement is no longer just a UI, it's a
  control-plane substrate with a CLI substrate (`internal/clilib`),
  an MCP server (`cmd/basement-mcp`), a federation engine, a backup
  engine, a webhook delivery worker, a bearer-auth middleware. Go
  module path moved to `github.com/mattjackson/basement`; all
  internal import paths updated in one sweep; OpenAPI spec file
  renamed from `basement-ui.yaml` to `basement.yaml`; Docker image
  moved from `ghcr.io/mattjackson/basement-ui` to
  `ghcr.io/mattjackson/basement`; `cmd/basement-server` binary keeps
  its name (already correct since v0.x).
- **Relicensed MIT â†’ AGPLv3.** Self-hosted control-plane software
  benefits from the AGPL's network-use clause: any modification
  served over a network (a fork-as-SaaS, an embedded admin UI in a
  proprietary product) must publish the modified source. The
  commercial-license escape hatch â€” operator emails matthew@pq.io â€”
  covers operators who need to embed or modify without publishing
  (proprietary embedded contexts, hosted SaaS reselling). The
  rebrand commit (`6de6a01`) sweeps LICENSE, package.json,
  README.md, and the v1.7 release notes in one pass;
  `v1.8.0b.1` (`d293dee`) is a follow-on OpenAPI filename rename.

### MCP server binary (`v1.8.0c`)

- **New `cmd/basement-mcp/` binary.** Stdio-only JSON-RPC 2.0 server
  on newline-delimited stdin/stdout (no HTTP variant); protocol
  version 2024-11-05; advertises `tools` capability only. Built as
  a separate binary so an operator drops it onto the same path as
  `basement-server` (or installs alongside Claude Desktop / Claude
  Code / Cursor) without coupling the MCP surface to the web
  server's HTTP lifecycle.
- **Ten tools at launch.** Seven read tools (`basement_list_regions`,
  `basement_list_buckets`, `basement_list_objects`,
  `basement_get_object_metadata`, `basement_list_backups`,
  `basement_list_federations`, `basement_list_audit`), two write
  tools (`basement_create_share`, `basement_create_backup_run`),
  and one forward-compatible placeholder (`basement_search` returns
  `NOT_IMPLEMENTED` until the v1.9 search-index cycle). Each tool
  consumes a curated subset of the same JSON API the FE uses today;
  no new server-side surface ships.
- **Bearer-only auth.** The MCP server reads
  `~/.config/basement/config.yaml` (the `internal/clilib` shape) for
  the bearer credentials minted at `/admin/service-accounts/new`
  and inherits whatever capabilities the SA was granted â€” it never
  defines its own role model. Capability failures surface to the
  agent as MCP tool errors with the same `ELEVATION_NOT_AVAILABLE`
  semantics as the FE.
- **New `internal/clilib` shared package.** Factors out the YAML
  config loader + HTTP client so future out-of-process clients
  (homebrew tap, alternate language SDK) can reuse the same
  `$BASEMENT_PROFILE` + `$BASEMENT_SECRET_KEY` precedence without a
  fork. The CLI binary plan for v1.8.0a originally drove this
  factoring; v1.8.0d dropped the CLI but the shared package stays.
- **Install paths documented** for Claude Desktop, Claude Code, and
  Cursor in `cmd/basement-mcp/README.md`. Tool calls log to stderr
  (JSON) so the stdout transport stays clean for the RPC frames.

### Drop the CLI plan + "Use with MCP" affordance (`v1.8.0d`)

- **`cmd/basement-cli/` deleted before it shipped.** The v1.8.0a CLI
  binary that landed on `main` was reverted in v1.8.0d â€” operator
  audit found the surface mostly duplicated aws-cli (object CRUD)
  and the web UI (control plane). A third surface didn't earn its
  keep. Stale references in README, `internal/clilib/config.go`
  comments, and the v1.7.0 release notes are scrubbed in the same
  cycle.
- **`<McpConfigSection>` shared component.** Lives in
  `frontend/src/shared/ui/McpConfigSection.tsx`; renders two
  snippets per service account: (1) `config.yaml` matching
  `internal/clilib.Config` (top-level `profiles` map keyed by
  profile name with `endpoint` / `access_key_id` / `secret_key`
  fields) with Copy + Download buttons, (2) Claude / Cursor host
  JSON config with `command: basement-mcp` and
  `env.BASEMENT_CONFIG` pointing at the YAML file. Endpoint defaults
  to `window.location.origin` so an operator standing in front of
  their own basement gets the right URL without typing.
- **Two call-sites for the same component.** On the mint-success
  dialog (`<SecretShownOnceDialog>`), the YAML inlines the plaintext
  secret â€” same shown-once contract the rest of the dialog
  enforces. On the new detail page, the YAML carries a
  `<SECRET_FROM_ROTATE>` placeholder + an amber hint that the
  operator must rotate the secret to refill it (the plaintext is
  gone forever after mint by design).
- **New `/admin/service-accounts/$id` detail page.** Shows identity
  (access key ID, created, last used, expires, revoked-at if
  revoked), capabilities (grouped by domain), and the "Use with
  MCP" card. The list page's name column links to this detail page
  so operators can grab the snippet after the shown-once dialog has
  closed. Rotation + revocation deliberately stay on the list
  page's row dropdown â€” keeping the detail page focused on the
  reveal-config use case avoids growing a second affordance for the
  shown-once secret flow.

### Mobile PWA + bucket browser card layout (`v1.8.0e`)

- **vite-plugin-pwa wired into the build.** Generates
  `dist/sw.js` + `dist/manifest.webmanifest` on every `pnpm build`;
  the service worker precaches the static app shell (HTML, JS, CSS,
  images, fonts) so offline shell loads work; `/api/*` is
  denylisted from the navigation fallback so auth-scoped responses
  always hit the network instead of being served stale from the
  cache.
- **Two manifest paths during the cutover.** The new
  `/manifest.webmanifest` is plugin-generated and matches the
  `internal/web/dist/` shape; the existing `/site.webmanifest`
  (frontend/public/site.webmanifest, hand-written) is kept for
  back-compat with iOS devices that already installed a v1.7-or-
  earlier shortcut â€” they'll find their bookmarked manifest URL
  still resolves. A unit test (`frontend/src/__tests__/manifest.test.ts`)
  pins both shapes' required fields.
- **iOS standalone hooks in `index.html`.**
  `apple-mobile-web-app-capable=yes` +
  `apple-mobile-web-app-status-bar-style=default` make Safari iOS
  render the installed app full-bleed without browser chrome (the
  web manifest alone isn't enough â€” Safari iOS only honors these
  legacy meta tags); apple-touch-icon at 180Ã—180 supplies the home-
  screen icon; theme-color `#C9874B` (basement amber) tints the
  address bar on Chrome / Edge Android.
- **Bucket browser card layout below 640px.** `useIsMobile()` hook
  watches `window.matchMedia("(max-width: 639px)")` and flips the
  virtualized row renderer between the existing 5-column table grid
  and a stacked card layout. On mobile, row height bumps from 40px
  to 56px so checkbox + name tap targets meet iOS HIG's 44px
  minimum; size + last-modified columns hide so the file name is
  the primary visual element; the layout flag surfaces as
  `data-layout="card"` on the scroll container for E2E
  observability.
- **`<InstallToHomeScreenHint>` banner.** Renders on `/files` when
  three conditions are all true: localStorage flag not set,
  `(display-mode: browser)` matches (i.e. the page is NOT already
  running standalone), and `(max-width: 767px)` matches. Once

  operator clears storage. Explains Share â†’ Add to Home Screen for

  banner renders on Android even though Chrome/Edge auto-prompt via

## Configuration

**`BASEMENT_CONFIG`** â€” read by `basement-mcp` (and by
`internal/clilib` generally); points at the YAML profile file
generated from the `<McpConfigSection>` snippet, default
`~/.config/basement/config.yaml`. The shape:

```yaml
profiles:
  default:
    endpoint: https://basement.example.com
    access_key_id: BMNT<...>
    secret_key: <plaintext>
```

Operators with multiple basements add additional `profiles:` blocks
keyed by name and select with `$BASEMENT_PROFILE`.

**No other new env vars** for the server. PWA precache assets ship
inside the existing `internal/web/dist/` embed; service-worker
registration is automatic on `/files` and the user-shell routes.

Operators who want to surface the MCP server to Claude / Cursor:

   detail page later â€” but the secret is shown exactly once, so

3. Copy the Claude / Cursor JSON snippet into your
   `~/Library/Application Support/Claude/claude_desktop_config.json`
   (or the equivalent for Cursor / Claude Code) and restart the
   host app.
4. Claude / Cursor will now expose the basement tools in its
   tool-use list. Try "list my regions" or "show me the last 10
   audit events" to verify.

Operators on mobile:

1. Open basement.example.com on Safari (iOS) or Chrome (Android).
2. iOS: tap Share â†’ "Add to Home Screen". The hint banner on
   `/files` walks through this.
3. Android: Chrome auto-prompts via the `beforeinstallprompt` event
   (or use the menu's "Install app" entry).
4. Once installed, the icon launches basement standalone (no
   browser chrome), with offline shell loads cached by the service
   worker.

## Architecture

v1.8 has three independent landing zones that share v1.7's bearer-
auth substrate:

1. **MCP server (`cmd/basement-mcp` + `internal/clilib`).** Single
   binary, stdio transport, no HTTP server. Talks to a running
   basement-server over the FE's JSON API using bearer credentials.
   The `internal/clilib` package factors out the YAML loader + HTTP
   client; the MCP binary is just a JSON-RPC 2.0 framing layer +
   tool dispatch on top of that.
2. **Service-account config UX
   (`<McpConfigSection>` + `/admin/service-accounts/$id`).** Pure
   FE work. The shared component renders YAML + JSON snippets the
   operator pastes into config files; the detail page is a
   read-only view that reuses the v1.7 admin list + mint
   endpoints.
3. **PWA (vite-plugin-pwa + iOS hooks + `<InstallToHomeScreenHint>`
   + mobile bucket browser).** Pure FE work that lifts the app
   shell into installability and tunes the bucket browser to
   phone-sized viewports. The service worker handles offline
   shell loads; the manifest + iOS meta tags drive the install
   affordance.

This is by design. v1.8 is a leverage cycle on top of v1.7's
substrate â€” every new surface re-uses an existing API and an
existing auth path. The platform itself didn't grow new state
(no new JSON stores, no new env vars, no migrations).

### Architecture preview: v1.9 + v2.0

- **v1.9 â€” WebDAV + SMB gateways.** Network-drive surfaces for
  non-tech users â€” the same household members who installed the
  PWA in v1.8 will be able to mount a basement bucket as a
  Finder / Explorer drive in v1.9. WebDAV is the cross-platform
  default (Finder, Explorer, GVFS, mobile WebDAV clients);
  SMB picks up Windows-shop and macOS-finder cases where WebDAV's
  latency is too high. Both gateways authenticate via v1.7 service
  accounts and route through v1.6 federation topology + v0.8 sync
  engine.
- **v2.0 â€” S3 gateway.** Major-version slot. Inbound S3 requests
  routed through basement's gateway, which dispatches via the v1.6
  federation topology (read â†’ nearest healthy replica; write â†’
  primary). Service-account-minted SigV4 keys gate ingress;
  webhooks emit inbound-write events that drive event-driven
  federation. The PWA + MCP surfaces from v1.8 are already
  gateway-aware (they hit the JSON API, which will be a thin
  wrapper over the gateway in v2.0).

## Known issues / coming in v1.9

- **MCP server's write tools are operator-confirmed only.** The two
  write tools that shipped in v1.8.0c
  (`basement_create_share`, `basement_create_backup_run`) gate on
  the service account having the corresponding capability. The
  pattern for "AI agent proposes; operator confirms in the web UI"
  is still being shaken out â€” a future cycle may add a per-tool
  `require_confirmation: true` flag the FE renders as a sticky
  pending-approval banner on `/admin/audit`. v1.8 ships without
  that affordance; operators who don't want the AI minting backup
  runs unsupervised should mint a read-only SA.
- **PWA precache is the static shell only.** v1.8 doesn't ship an
  offline read mode for actual bucket data â€” the service worker
  caches the JS/CSS/HTML/fonts so the shell loads instantly even
  on a flaky connection, but every `/api/*` call still requires
  network. A future cycle may add IndexedDB-backed read-through
  caching for the user's region + bucket lists.
- **Block-scrub feature-detection fallback.** Carry-over from
  v1.4 â†’ v1.5 â†’ v1.6 â†’ v1.7 â†’ v1.8 â€” the `/v1/worker`
  feature-detection fallback for legacy Garage admin APIs is still
  on the backlog. Not on the v1.9 critical path (WebDAV + SMB
  gateways take priority).
- **v1.8.0a CLI binary was never published past the `v1.8.0a` tag
  on `latest`.** Operators who pulled `:latest` between the v1.8.0a
  push and the v1.8.0d push had the CLI binary briefly available
  in the image; the v1.8.0d image and later don't. No further
  action required â€” the CLI was experimental and the API surface
  it exposed (`basement keys list`, etc.) is covered by aws-cli for
  S3 ops and the web UI for control-plane ops.
