## Headline

**Launch readiness — first-run wizard + 5-min install + observability +
production deployment guide.**

v1.11 is the launch-readiness minor: every gap a self-hoster hits
between "saw a Show HN post" and "running basement in production"
gets filled in this cycle. The functional surface barely changes —
v1.10 already shipped the compliance + integrity primitives that
close the v1.x roadmap. What v1.11 adds is the on-boarding,
deployment, observability, and credibility scaffolding around it.
Six primary cycles plus one hotfix:

- **First-run onboarding wizard** (`v1.11.0a`). Fresh deploys no
  longer drop the operator into an empty admin UI. A five-step
  wizard at `/admin/first-run` (Welcome -> Cluster -> OIDC (optional)
  -> Team invite (optional) -> Done) walks through the minimal
  cluster + identity setup with a Skip on every step and a
  permanent dismissal latch reachable from "I'll set up later".
  AppShell auto-routes admin entries to the wizard when
  `needsOnboarding && !completed`. Existing operators upgrading
  from v1.10.x with a configured deploy auto-promote to
  `completed=true` on first load — they never see the wizard.
- **Production deployment guide** (`v1.11.0b`). New
  `docs/deployment/` directory with seven docs covering the full
  operator path from `docker run` to a production-grade install
  behind TLS with backups, hardening, and a documented upgrade
  story: topology overview + Docker recipe + reverse-proxy
  matrix (Caddy / Nginx / Traefik with the WebDAV verb-passthrough
  workaround flagged in v1.9.0e) + TLS (auto-ACME / Cloudflare /
  proxy-terminated) + hardening (network bind, secret management,
  data-dir permissions, container UID/caps) + backup-basement
  (what's in `DATA_DIR`, snapshot patterns, restore drills) +
  upgrade (Watchtower, version-compat contract, migration notes).
- **5-minute install** (`v1.11.0c`). `docker run` against the
  default image with zero env vars now comes up working: basement
  auto-bootstraps a JWT secret + admin password on first boot,
  prints the password to stdout for `docker logs basement | grep`,
  and persists both to `{DATA_DIR}/.jwt-secret` +
  `{DATA_DIR}/.initial-admin-password` (mode 0600) for reuse on
  restart. New `scripts/install.sh` curl-pipe-bash one-liner pulls
  the image, drops a minimal `docker-compose.yml`, brings the
  container up, polls logs, and prints the password banner. Zero
  behavioural change for operators with explicit env (every var
  short-circuits the bootstrap).
- **Trust + credibility docs** (`v1.11.0d`). GitHub-canonical files
  so a self-hoster evaluating basement can answer "how do I report
  a vuln, how do I contribute, what is the supply-chain story"
  without reading source. `SECURITY.md` (disclosure policy +
  threat model + crypto claims tied to implementing files);
  `CONTRIBUTING.md` (dev loop + DCO contract); `DCO.md`
  (Developer Certificate of Origin verbatim);
  `.github/ISSUE_TEMPLATE/` + `PULL_REQUEST_TEMPLATE.md`; new SBOM
  GitHub Actions workflow that publishes a CycloneDX SBOM per
  release tag.
- **Screenshots gallery + README polish** (`v1.11.0e`). 15-shot
  Playwright-driven capture under `docs/screenshots/v1.10/` plus
  a 2x4 README embed so visitors see what they're getting before
  they install. README "Why basement" trimmed from a multi-
  paragraph history into a 5-sentence pitch; new Quickstart code
  block matching the v1.11.0c auto-bootstrap pattern; Features
  table compressed from a 60-bullet wall into a 13-row matrix
  with per-release-notes links.
- **Observability — Prometheus + Grafana + slog** (`v1.11.0f`).
  `/metrics` endpoint publishes 14 metric families (HTTP rate +
  latency histogram, auth attempts, audit volume, federation
  replicate + per-replica lag, backup runs + last-success
  timestamp, webhook deliveries, active sessions, service
  accounts, buckets, objects, build info). Unauthenticated by
  default; bearer-token gate via `BASEMENT_METRICS_TOKEN`. New
  `docs/observability/` ships a ten-panel Grafana dashboard +
  six-rule alert set + scrape-config + import steps. Server
  logging now flows through `log/slog` with JSON (default) /
  text format selectable via `BASEMENT_LOG_FORMAT`.
- **Garage v2 admin-tier connections hotfix** (`v1.11.0.1`).
  Real Garage v2 testing during the v1.11 cycle surfaced a bug
  where the v2 driver's admin-tier connection path was broken;
  fixed in `internal/drivers/garage/` with parity to v1.

## Architecture context — completing the launch-readiness arc

v1.0 -> v1.10 shipped the feature surface. v1.11 ships the
*adoption* surface: the friction between "interesting project"
and "this is in production" is what the six cycles target,
ordered from most-blocking to most-polish:

- **Bootstrap friction (v1.11.0c)** — every previous release
  required the operator to bcrypt their own admin password and
  generate a 32-byte JWT secret before basement would even start.
  That's three CLI dependencies (`htpasswd`, `openssl`,
  shell-quoting) and four steps in front of an evaluator who just
  wanted to see the UI. v1.11.0c collapses this to `docker run +
  docker logs | grep` and persists the generated secrets so
  restarts are seamless.
- **Empty-UI friction (v1.11.0a)** — a fresh deploy with the
  admin password in hand still landed on an empty cluster list
  with no signal about what to do next. The wizard names the
  three minimum-viable setup steps (cluster + identity + team)
  and surfaces a Skip on each so the operator can dismiss the
  whole thing in two clicks if they prefer to wire by hand.
- **Production-posture friction (v1.11.0b)** — the previous
  README mentioned "use a reverse proxy" without naming one or
  explaining the WebDAV-verb-passthrough caveat that bit v1.9
  operators. `docs/deployment/` is now seven docs covering the
  paths an operator actually walks: Docker recipe, three reverse-
  proxy configs with the verb matrix, TLS configurations, the
  hardening checklist, the backup-the-metadata-not-the-data
  pattern, and the upgrade procedure.
- **Trust friction (v1.11.0d)** — self-hosters evaluating a
  control plane that holds their KMS key IDs + S3 secrets care
  about disclosure policy and supply-chain transparency before
  they will commit. v1.11.0d ships the GitHub-canonical files
  + the SBOM workflow so the answers are one click away.
- **Visual-evaluation friction (v1.11.0e)** — "looks great, but
  what does it look like?" is the question the previous README
  couldn't answer without an install. The screenshots gallery
  + README embeds answer it before the operator decides whether
  to commit a `docker run`.
- **Observability friction (v1.11.0f)** — operators running
  basement in production need to plug it into the same
  Prometheus + Grafana stack they already run for everything
  else. `/metrics` + the Grafana dashboard + the alert set make
  basement a first-class citizen of standard ops tooling.

The v1.11.0.1 hotfix is the recurring driver-parity lesson: every
non-trivial driver change needs end-to-end testing against the
real backend, not just the unit tests. The Garage v2 admin-tier
bug was visible the moment a real Garage v2 cluster was connected.

## Configuration

**New env vars**:

| Var | Default | Notes |
|---|---|---|
| `BASEMENT_METRICS_TOKEN` | (unset) | When set, gates `/metrics` behind bearer-token auth. Unset = unauthenticated exposition (standard Prom convention). |
| `BASEMENT_LOG_FORMAT` | `json` | `json` (one parseable object per line) or `text` (key=value for local dev). Invalid values rejected at boot. |
| `BASEMENT_ADMIN_PASSWORD` | (unset) | Plaintext bootstrap password — bcrypted at boot, never persisted. Mutually exclusive with `BASEMENT_ADMIN_PASSWORD_HASH`. |

**Bootstrap behaviour** (v1.11.0c): when both
`BASEMENT_JWT_SECRET` and `BASEMENT_ADMIN_PASSWORD_HASH` /
`BASEMENT_ADMIN_PASSWORD` are unset, basement mints both on first
boot:

- `{DATA_DIR}/.jwt-secret` — 32 random bytes (hex), mode 0600,
  reused on restart.
- `{DATA_DIR}/.initial-admin-password` — 24-char password from
  an unambiguous alphabet, mode 0600. Printed once to stdout via
  `fmt.Println` so `docker logs basement | grep "INITIAL ADMIN
  PASSWORD"` works regardless of `LOG_LEVEL`. Persisted plaintext
  on disk so a forgotten password is recoverable (the file is
  the bootstrap latch — `BASEMENT_DATA_DIR` is the operator's
  trust boundary).

Idempotency: explicit env vars always short-circuit the bootstrap;
existing `.jwt-secret` / `.initial-admin-password` files are
reused, not rotated. Operators upgrading from v1.10.x with full
env see zero behavioural change.

## Breaking changes

**None.** The v1.11 cycle is additive throughout:

- All bootstrap paths are opt-in via the absence of env vars.
- The first-run wizard is gated on the `onboardingCompleted`
  latch which auto-promotes to `true` on first load for any
  pre-v1.11 deploy that has clusters configured.
- `/metrics` is a new endpoint; the bearer-token gate is opt-in
  via `BASEMENT_METRICS_TOKEN`.
- `BASEMENT_LOG_FORMAT` defaults to `json`, which matches the
  pre-v1.11 stdlib default for `log/slog`. Operators tailing
  plaintext logs can set `BASEMENT_LOG_FORMAT=text` to restore
  the previous format.
- No driver interface changes, no API wire-shape changes, no
  storage migrations.

## Upgrade notes

Pull the new image:

```bash
docker pull ghcr.io/mattjackson/basement:v1.11.0
```

**Existing operators**: zero action required. The first-run
wizard's dismissal latch auto-promotes to `completed=true` for any
deploy with clusters configured; the bootstrap path no-ops on the
presence of explicit env vars; `/metrics` is a new endpoint that
doesn't change existing routes.

**New operators**: enjoy the 5-min install:

```bash
docker run -d --name basement -p 8080:8080 \
  -v basement-data:/var/lib/basement \
  ghcr.io/mattjackson/basement:v1.11.0

docker logs basement 2>&1 | grep "INITIAL ADMIN PASSWORD"
# INITIAL ADMIN PASSWORD: <24-char string>
```

Open the UI, log in, and the first-run wizard walks the cluster +
OIDC + team-invite setup. Production posture (explicit secrets,
reverse-proxy, TLS, backups): see
[`docs/deployment/`](../deployment/README.md).

**Scraping `/metrics`**: see
[`docs/observability/README.md`](../observability/README.md) for
the Prometheus scrape config, Grafana dashboard import steps, and
the alert ruleset.

## End-of-v1.x summary — what shipped across the v1.x line

v1.x took basement from "polished admin UI for multi-backend object
storage" through "control-plane substrate with federation, M2M
auth, mobile + agent + native-filesystem clients, compliance +
integrity primitives" all the way to **"production-ready, externally
adoptable platform"** with v1.11. Twelve minor versions in one
extended operator session, each milestone-tagged after a senior-run
smoke + screenshot + manual exercise pass.

| Tag | Headline | Key contracts |
|---|---|---|
| **v1.0** | Multi-backend admin + RBAC + scoped creds | ADR-0001 three-tier roles; capability matrix; encrypted admin_token; audit log; metrics persistence |
| **v1.1** | Region tier replaces phantom Connections at user persona | ADR-0002 (UserRegion as primary user noun); Garage admin bridge; per-user S3 signing |
| **v1.2** | Sudo-style admin elevation | ADR-0003 (USER + ADMIN modes, configurable TTL, drop-in-place banner, OIDC step-up); key-first user keychain |
| **v1.3** | Multi-user polish | OIDC group mapping; invite tokens; bulk-import; per-cluster cluster_admin UI; per-region S3 addressing toggle |
| **v1.4** | Scale + perf | Virtualized bucket browser (10K+ rows); paginated key perms; batch object ops; audit pagination + CSV; storage growth analytics; block-scrub UI |
| **v1.5** | Backup story | Scheduled S3 → S3 backups; mirror + snapshot modes; GFS retention; 3-step restore wizard |
| **v1.6** | Federation + multi-backend replication | `FederatedBucket` first-class concept; polling replication engine; manual + opt-in auto-failover |
| **v1.7** | Service accounts + webhooks + event-driven federation | `BMNT...` bearer creds (M2M substrate for v1.8/v2.0); HMAC-signed webhooks; in-process pub/sub flips federation from polling to sub-second |
| **v1.8** | MCP server + Mobile PWA + service-account config UX + rebrand/relicense | `basement-mcp` stdio server (ten tools); `<McpConfigSection>` Use-with-MCP UX; installable PWA + mobile bucket browser; `basement-ui` → `basement`, MIT → AGPLv3 |
| **v1.9** | WebDAV gateway + pluggable gateway architecture | `/webdav/` tree (Finder / Explorer / Nautilus / iOS Files / rclone); `internal/gateway/` `Gateway` + `Backend` + `Registry` interfaces; 4 stub gateways (SMB / NFS / FTP / S3) registered at boot |
| **v1.10** | Compliance + integrity — Versioning + Object Lock + SSE | Three primitives that make federated replicas resilient + recoverable + private; AWS S3 + MinIO full, Garage stubs; capability-gated UI |
| **v1.11 (this release)** | Launch readiness — first-run wizard + 5-min install + observability + deployment guide | First-run onboarding; auto-bootstrap JWT + admin password; `/metrics` Prometheus + Grafana + alerts + slog; SECURITY + CONTRIBUTING + SBOM; production deployment guide; screenshot gallery + README polish |

**v1.x is complete.** The roadmap from ADR-0001 onward has now
shipped end-to-end and basement is genuinely consumable by external
operators without insider knowledge.

**v2.0 = basement IS a backend (S3 gateway).** The next major slot
makes basement itself an S3 endpoint — an SigV4-verifying gateway
that routes inbound requests using the v1.6 federation topology,
authed by v1.7 service-account credentials, observable via v1.7
webhooks, manageable from v1.8 CLI/MCP/PWA + v1.9 WebDAV + (v2.x)
SMB/NFS/FTP gateways, and protected by v1.10 versioning + lock +
SSE. The `Backend` interface that v1.9.0c shipped is already
S3-shaped so the v2.0 implementation is a SigV4 verifier + an S3
request parser on top of the same data plane that powers WebDAV
today. ADR-0006 is the next senior artifact. v2.0 ships when
external demand surfaces — basement v1.x already covers the
substrate the v2.0 gateway will live on.

**v2.x sketch** (post-v2.0):
- v2.1: client-side encryption (E2EE) for untrusted-backend
  federation replicas
- v2.2: search + tags + smart collections
- v2.3: SMB + NFS gateways alongside WebDAV
- v2.4: cost engine + cross-backend lifecycle
- v3.0: plugin SDK + multi-site mesh + IPFS / CDN + marketplace
