## Headline

**Compliance + integrity — Versioning + Object Lock + SSE; v1.x
roadmap complete.**

v1.10 closes the v1.x arc with the ransomware shield + compliance
posture that complements v1.6 federation. Federation replicates data
across backends; versioning + object-lock + server-side encryption
make those replicas resilient, recoverable, and private. A
compromised primary can't poison the chain if every replica honours
WORM; an exfiltrated bucket dump is opaque without the KMS key. Each
piece is opt-in per bucket and capability-gated per driver, following
the v1.4-style "honest capability" pattern: AWS S3 + MinIO ship full
implementations, Garage v1 / v2 advertise unsupported (upstream
limitation) and the UI renders graceful "Not supported by this
backend driver" notices instead of fighting the gap.

This is the last minor before v2.0. With v1.10 shipped, the v1.x
roadmap is feature-complete. The next major (v2.0) makes basement
itself an S3 backend — an SigV4-verifying gateway that routes
inbound requests across the federation topology landed in v1.6,
authed by the service accounts minted in v1.7, observable via the
webhooks subscribed in v1.7, manageable from the CLI + MCP +
mobile PWA + WebDAV surfaces lit up in v1.8 / v1.9, and protected
by the versioning + lock + SSE primitives landed here.

## What changed since v1.9.0

### Bucket versioning — driver, API, UI (`v1.10.0a` + `v1.10.0b`)

- **Driver interface gains seven members.** `VersioningSupport() bool`
  capability gate (the FE renders the toggle only when true, mirrors
  `LifecycleSupport` / `ScrubSupport` posture);
  `GetVersioningStatus` + `EnableVersioning` + `SuspendVersioning`
  bucket-level toggle (enabled / suspended / disabled per S3);
  `ListObjectVersions` + `GetObjectVersion` + `DeleteObjectVersion`
  per-object version history. `ListObjectVersions` returns a fused
  slice of normal versions + delete markers (S3 surfaces them as
  two distinct arrays); the wire opaque marker collapses S3's
  two-marker pagination (`KeyMarker + VersionIdMarker`) into a
  single `{k}|{v}` blob so callers treat it as a token.
- **AWS S3 + MinIO full** via aws-sdk-go-v2's
  `GetBucketVersioning` / `PutBucketVersioning` /
  `ListObjectVersions` / `GetObject(VersionId)` /
  `DeleteObject(VersionId)`. `VersioningSupport()=true` on both.
- **Garage v1 + v2 stubs.** `VersioningSupport()=false` +
  `ErrUnsupported` from every method. Upstream Garage does not
  implement bucket versioning in its S3 surface
  (content-addressed block store semantics conflict with versioned
  overwrites); each driver carries a `TODO` referencing a future
  swap to the aws_s3-style proxy if upstream adds support.
- **User-tier API.** `GET / PUT
  /api/v1/user/regions/{rid}/buckets/{bid}/versioning` + `GET /
  DELETE /versions` + `GET /versions/{vid}` for download
  (presign-style streaming). Wire envelope folds the capability
  flag into the GET payload so the FE branches on one fetch:
  `{supported: bool, status: "disabled"|"enabled"|"suspended"}`.
  PUT enforces server-side that you can't flip from
  `suspended → disabled` (S3 doesn't allow it). Audit per
  mutation via `bucket:versioning:{enable,suspend}` +
  `object_version:delete`.
- **`VersioningSection` card** at the bottom of the bucket detail
  page. Three branches mirror `LifecycleSection`'s posture:
  unsupported / disabled / supported-and-enabled-or-suspended.
  Save stays disabled until the operator changes the select away
  from the current backend state — saves a "clicked Save without
  changing anything" round trip and avoids re-auditing a no-op
  transition.
- **`ObjectVersionsPanel`** mounts inline below the file list when
  the operator clicks "Versions" on any file row. Lists every
  version with version-ID, size, modified timestamp, "current"
  badge, and delete-marker chip. Per-row Download + Delete
  affordances. "Show all versions" toggle on the bucket browser
  header surfaces a per-row prior-version count next to the
  current version — when on, the toggle hydrates the panel with
  every version inline rather than requiring a click.

### Object Lock — driver, API, UI (`v1.10.0c`)

- **Seven new driver methods.** `ObjectLockSupport() bool` capability
  gate; `GetObjectLockConfig` + `PutObjectLockConfig` bucket-level
  lock configuration (enabled flag + optional default retention
  with mode + days); `GetObjectRetention` + `PutObjectRetention`
  per-object-version retention (Compliance / Governance mode +
  `retainUntilDate`); `GetObjectLegalHold` + `PutObjectLegalHold`
  per-object-version legal-hold toggle. AWS S3 + MinIO full;
  Garage stubs.
- **User-tier API.** `GET / PUT
  /api/v1/user/regions/{rid}/buckets/{bid}/object-lock` (bucket
  config); `GET / PUT
  /api/v1/user/regions/{rid}/buckets/{bid}/objects/{key}/versions/{vid}/retention`
  (per-version retention); `GET / PUT
  .../legal-hold` (per-version legal hold). Per-axis capability
  gating: the bucket-config endpoint folds `supported:bool` into
  the GET response so the FE handles the unsupported branch
  without a driver-name check.
- **`ObjectLockSection` card** layered on top of versioning per the
  S3 spec — Object Lock REQUIRES versioning enabled. Three
  branches the card surfaces internally:
  - `supported=false` → static "Not supported by this backend
    driver" notice. Garage lands here.
  - `supported`, versioning not enabled → "Versioning required"
    notice with controls disabled. Operator must enable
    versioning first.
  - `supported` + versioning enabled → full editor. Default
    retention toggle + mode (Governance / Compliance) + days
    input.
- **One-way contract surfaced.** Once Object Lock is enabled on a
  bucket the S3 contract is one-way: it cannot be disabled. The
  UI surfaces this distinctly — once enabled, the disable
  affordance disappears and the operator can only adjust default
  retention. The card carries a "Once enabled, Object Lock cannot
  be disabled (S3 contract)" hint when the bucket is in the
  enabled state.
- **Per-version lock UX in `ObjectVersionsPanel`.** Each version
  row gains a lock status pill (Compliance / Governance until
  YYYY-MM-DD or Legal hold), a Set retention action that opens a
  2-field modal (mode + days) with reduce-detection that surfaces
  the bypass-governance toggle, a Set / Release hold toggle, and a
  Delete affordance gated on the lock state — compliance retention
  blocks delete; governance retention surfaces a Bypass governance
  toggle on the delete dialog. New actions only render when the
  bucket has Object Lock enabled.

### Bucket default encryption — driver, API, UI (`v1.10.0d`)

- **Four new driver methods.** `BucketEncryptionSupport()
  (supportedS3, supportedKms bool)` returning per-axis capability
  bits (some backends support SSE-S3 but not SSE-KMS — the FE
  needs both bits to gate the algorithm radio correctly);
  `GetBucketEncryption` + `PutBucketEncryption` +
  `DeleteBucketEncryption`. AWS S3 full (both axes); MinIO full
  with the KMS axis live only when the MinIO server is wired with
  a KMS backend (the driver surfaces the live state honestly); Garage
  stubs.
- **User-tier API.** `GET / PUT / DELETE
  /api/v1/user/regions/{rid}/buckets/{bid}/encryption`. GET folds
  the per-axis capability bits into the response so the FE handles
  all four branches off one fetch:
  `{supportedS3:bool, supportedKms:bool, enabled:bool,
  algorithm?: "AES256"|"aws:kms", kmsKeyId?: string,
  bucketKey?: bool}`. Audit per mutation via
  `bucket:encryption:{put,delete}`.
- **`EncryptionSection` card** on the bucket detail page. Four
  branches the section handles internally:
  - both axes false → "Not supported by this backend driver"
    notice
  - only `supportedS3` true → enable / disable toggle without the
    algorithm radio (SSE-S3 is the only option)
  - both true → full editor: algorithm radio (SSE-S3 / SSE-KMS) +
    KMS key ID input + S3 Bucket Key optimization checkbox
    (SSE-KMS only — reduces KMS call costs on write-heavy buckets
    by ~one call per 5 min instead of one per object)
  - enabled state: Save updates the config; a separate Disable
    button calls DELETE
- **Capability-honest gating** per the driver-as-middleman doctrine
  — the card reads the `(supportedS3, supportedKms)` bits from
  the GET response and gates UI elements directly off them, no
  driver-name checks anywhere in the FE. Client-side guard
  disables Save when SSE-KMS is selected with an empty key field
  (the API rejects it too, but the guard avoids the wasted
  round-trip).

### Smoke gate + AppShell hotfix (`v1.10.0d.1` + `v1.10.0e`)

- **Smoke gate hotfix surfaced AppShell hydration race.** The
  v1.10.0e smoke gate caught an `/admin/*` redirect regression
  introduced in v1.9.0e.2 — every full-page navigation to
  `/admin/*` bounced to `/files` even on freshly-elevated sessions,
  because the tight-coupling redirect fired on the first render
  before `AuthModeHydrator` could push the cookie-derived mode into
  the provider. Same shape as the v1.7.0a.3/a.4
  `AdminEntryElevationGuard` hotfix:
  - defer the redirect while `useUser()` is loading
  - belt-and-braces — read `user.mode` directly off the `/auth/me`
    payload and short-circuit when admin / elevated (the provider
    mode updates one render later via the hydrator)
- **Five new smoke checks** cover the v1.10 surfaces:
  - `[v1.10a]` `/files/{rid}/b/{bid}` renders all three compliance
    sections (Versioning + Object Lock + Encryption) with the
    expected testids + h2 header text
  - `[v1.10b]` `GET /buckets/{bid}/versioning` returns the
    `{supported, status}` shape
  - `[v1.10c]` `GET /buckets/{bid}/object-lock` returns the
    `{supported, enabled}` shape
  - `[v1.10d]` `GET /buckets/{bid}/encryption` returns the
    `{supportedS3, supportedKms, enabled}` shape
  - `[v1.10e]` Garage backends render "Not supported" notices on
    all three sections (probes the live user-region for a
    `supported:false` versioning bucket, then asserts the three
    cards render their unsupported branches)

## Architecture context — ransomware shield + compliance posture

v1.6 federation replicates data across backends (home Garage +
off-site B2). v1.10's three primitives complete the integrity story
for those replicas:

- **Versioning** keeps prior copies of overwritten / deleted objects.
  An accidental `aws s3 rm --recursive` no longer destroys history;
  a compromised user key can't delete object content, only mark it
  with a delete marker that's reversible.
- **Object Lock** makes specific object versions immutable for a
  configurable window. Compliance mode is unbeatable until expiry
  (regulator-grade retention); Governance mode permits override with
  a bypass capability. Layered on federation: every replica honours
  the lock, so a compromised primary can't poison the chain — the
  attacker has the primary's keys but can't reach back through the
  federation to delete the locked versions on the replicas.
- **Server-side encryption** (SSE-S3 / SSE-KMS) renders an
  exfiltrated bucket dump opaque without the key. SSE-KMS hands
  key control to an external KMS so the basement-side backend
  can't decrypt — even a compromised backend operator can't read
  the contents without separately compromising KMS.

These three together change the threat model: ransomware that
encrypts an entire primary bucket loses access to historical
versions (versioning), can't delete the historical versions
(object-lock), and (if SSE-KMS is on) can't read what it already
encrypted because the source material was already encrypted to a
key it doesn't control. Federation spreads recovery surface across
backends; these three make the federation recoverable, immutable,
and private respectively.

## Configuration

**No new env vars** for the server. Versioning, Object Lock, and
default encryption are all opt-in per bucket — existing deployments
see no behavior change until the operator enables them. Each
bucket's settings live on the backend (S3 contract); basement
surfaces + manages them but doesn't shadow them in its own state.

**SSE-KMS configuration on the backend side.** SSE-KMS requires the
backend's IAM role / equivalent to have `kms:Encrypt` /
`kms:Decrypt` permission on the chosen key. basement surfaces this
as a hint in the encryption editor; the actual permission grant is
on the operator and the backend, not on basement.

## Breaking changes

**None.** The v1.10 cycle is additive throughout:

- All three primitive sets (versioning, object-lock, encryption)
  are net-new driver methods + API endpoints + UI cards. No
  existing wire shape changes.
- The driver interface grows 7 + 7 + 4 = 18 new methods. Out-of-tree
  driver implementations (rare — basement is the only consumer)
  need to add stubs returning `*Support()=false` + `ErrUnsupported`
  for the new methods. The four in-tree drivers all ship full or
  stub implementations as appropriate.
- No data migrations. No `org_capabilities.json` changes.

## Upgrade notes

Pull the new image:

```bash
docker pull ghcr.io/mattjackson/basement:v1.10.0
```

Existing deployments see no behavior change on upgrade. To turn on
the new primitives:

1. Visit `/files/{regionId}/b/{bucketId}` for any bucket on an
   AWS S3 / MinIO region.
2. Versioning section at the bottom — select Enabled, hit Save.
3. Once versioning is on, the Object Lock section unblocks. Enable
   it (one-way per S3 spec) and optionally set a default
   retention.
4. Encryption section is independent — enable + pick SSE-S3 (the
   simpler path) or SSE-KMS (operator-controlled key).

Buckets on Garage user-regions show "Not supported by this backend
driver" notices for all three sections. That's expected — upstream
Garage doesn't implement these S3-API-level primitives. If you need
versioning / object-lock / SSE on a Garage-backed bucket today, the
recommended pattern is to federate it with an AWS S3 / MinIO replica
(v1.6 federation) where the primitives are honoured.

## End-of-v1.x summary — what shipped across the v1.x line

v1.x took basement from "polished but single-purpose admin UI for
multi-backend object storage" to "control-plane substrate with
federation, M2M auth, mobile + agent + native-filesystem clients,
and compliance + integrity primitives." Eleven minor versions in
~one operator session-and-change. Every minor was milestone-tagged
after a senior-run smoke + screenshot + manual exercise pass.

| Tag | Headline | Key contracts |
|---|---|---|
| **v1.0** | Multi-backend admin + RBAC + scoped creds | ADR-0001 three-tier roles; capability matrix; encrypted admin_token; audit log; metrics persistence |
| **v1.1** | Region tier replaces phantom Connections at user persona | ADR-0002 (UserRegion as primary user noun); Garage admin bridge; per-user S3 signing |

| **v1.10 (this release)** | Compliance + integrity — Versioning + Object Lock + SSE | Three primitives that make federated replicas resilient + recoverable + private; AWS S3 + MinIO full, Garage stubs; capability-gated UI |

**v1.x is feature-complete.** The roadmap from ADR-0004's v1.6
through v1.10 has now shipped end-to-end with the cycle dependencies
honoured throughout: v1.6 federation supplies the bucket-to-backend
routing topology, v1.7 service accounts supply the M2M credentials
+ event substrate, v1.8/v1.9 light up the interaction surfaces
(CLI / MCP / Mobile / WebDAV / SMB+NFS+FTP+S3 stubs), and v1.10's
versioning + lock + SSE make federated replicas immutable + private.

**v2.0 = basement IS a backend (S3 gateway).** The next major slot
makes basement itself an S3 endpoint — an SigV4-verifying gateway
that routes inbound requests using the v1.6 federation topology,
authed by v1.7 service-account credentials, observable via v1.7
webhooks, manageable from v1.8 CLI/MCP/PWA + v1.9 WebDAV + (v2.x)
SMB/NFS/FTP gateways, and protected by v1.10 versioning + lock +
SSE. The `Backend` interface that v1.9.0c shipped is already
S3-shaped (`AuthSigV4` slot, `PutObject` / `GetObject` /
`HeadObject` / `DeleteObject` / `CopyObject` / `ListObjects` /
`CreateBucket` / `DeleteBucket`) so the v2.0 implementation is a
SigV4 verifier + an S3 request parser on top of the same data plane
that powers WebDAV today. ADR-0006 (v2.0 gateway design) is the
next senior artifact.
