# S3 API gateway

> **Status:** stub. Registered in v1.9.0c so the `/admin/gateways`
> roster surfaces the planned v2.0 gateway from day one; the
> implementation is not built yet. The full design is in
> [ADR-0006](../adr/0006-v2-s3-gateway.md) (proposed — awaiting
> operator decision).

## What it is

basement currently routes S3 traffic via the user's region keychain
(the user supplies an S3 endpoint + key per region; basement is the
admin pane on top). The v2.0 S3 GATEWAY inverts the relationship:
basement itself terminates an S3 endpoint, verifies SigV4 against
service-account credentials, and proxies the request to the
underlying driver.

This is the substrate the SDK story rides on — once basement speaks
S3 natively, every existing S3 SDK in any language is a first-class
basement client.

A working basement S3 gateway would let:

- `aws s3 ...` / `aws s3api ...` target basement directly with
  service-account credentials, no separate S3 backend in the path.
- Every existing S3 SDK (Python `boto3`, Go `aws-sdk-go-v2`, JS
  `@aws-sdk/client-s3`, Rust `aws-sdk-s3`, etc.) point at basement
  the same way they point at AWS or MinIO.
- Multi-region requests fan out to the basement region behind the
  scenes; the SDK never needs to know there's a control plane in
  front.

## Why it's a stub in v1.9.0c

The v1.9.0c cycle was the gateway abstraction itself. The S3 gateway
needs the heavyweight pieces:

- SigV4 request verification (the protocol's signing scheme; non-
  trivial because Go's stdlib doesn't ship a verifier — only a
  signer for outbound requests).
- A request proxy that translates incoming S3 verbs back into the
  Backend interface's verbs (mostly a 1:1, but multipart upload
  state has to live somewhere).
- An auth path for service-account credentials presented in SigV4's
  AKID + secret shape (basement's service accounts already issue an
  AKID + secret pair; the binding is the easy part).

That bundle is the v2.0 milestone.

## What to use instead — today

For service-account-driven automation that needs an S3-shaped
endpoint, point your SDK at one of your basement regions directly —
the same credentials work, and basement is transparent at the wire
level.

For the operator-facing "every SDK works against basement" story,
v2.0 is the cycle that delivers it.

## Implementation tracking

The native S3 gateway is the v2.0 headline. The full design + scope
+ open questions are in [ADR-0006](../adr/0006-v2-s3-gateway.md);
the ADR is "Proposed — awaiting operator decision" as of v1.11.

When it ships, this doc will be replaced by the full integration
guide (SDK examples, service-account credential setup, region
semantics, troubleshooting). The gateway interface it'll implement
is documented in [adding-a-gateway.md](adding-a-gateway.md).
